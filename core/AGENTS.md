# core/AGENTS.md

Inherit the root `AGENTS.md` instructions for all core package work.

## Executable Guide TDD

- Use `bun run dev:guides` from the repo root as the dev-time TDD loop for executable guides. It regenerates guide scenarios, typechecks the generated path, and re-runs the affected guide scenario tests on changes to guide MDX, core/sdk/plugin source, or the scenario generator.
- Use `bun run dev:guides docs/guides/<path>.mdx` for a focused single-guide loop, and add `--once` for a single non-watching pass.

## Test fixtures

- `core/test/cli/fixtures/*.json` is NOT in biome's ignore list, so committed JSON fixtures get reformatted to pretty-printed JSON by `bun run lint`. A renderer that emits compact JSON (e.g. `JSON.stringify(value)`) therefore cannot be string-compared against its committed fixture — assert `JSON.parse(rendererOutput)` deep-equals `Bun.file(fixture).json()` instead, so biome formatting of the fixture never diverges from the renderer's whitespace.

## Git recipe source

- `lando init --source=git` is acquisition-only: it shallow-clones through `core/src/recipes/git-source.ts`, publishes by commit SHA under `<userDataRoot>/recipe-cache/git/`, and then uses the existing recipe render path. Non-bundled directory recipe rendering is still the shared Alpha boundary, so do not assume git recipes scaffold end-to-end yet.
- `lando init --source=tarball` mirrors git acquisition in `core/src/recipes/tarball-source.ts`: it downloads via an injectable `fetcher` (default = global `fetch`, which accepts `file://` for tests), SHA-256-verifies, extracts via a pure-JS tar(.gz) reader (no host `tar` dependency at runtime), and publishes under `<userDataRoot>/recipe-cache/tarball/<sha256>/`. `--checksum` makes verification hard-fail (`checksum-mismatch`, computed BEFORE any extraction); without it the resolver warns once with the computed digest and, when an interactive `confirmUnverified` seam is supplied (i.e. not `--yes`/`--no-interactive`), prompts once — a decline aborts with `checksum-unverified`. The verify/prompt seam lives in `loadTarballRecipe` (`init.ts`), not the resolver, so the resolver stays pure/injectable.
- Keep remote-source CLI parsing centralized in `core/src/cli/commands/init-source.ts`; both OCLIF init and the compiled dispatcher must accept/reject `--source`/`--url`/`--path`/`--checksum` identically (preserve the exact per-source missing-url wording — `<git-url>` vs `<tarball-url>` — to keep the parity tests green), and the default git cloner must keep interactive prompts disabled.

## Programmatic `recipe.ts`

- A local recipe directory may ship `recipe.ts` *instead of* `recipe.yml` (the two are mutually exclusive in a directory — `resolveLocal` in `core/src/recipes/source.ts` rejects a dir carrying both). The TS form default-exports a static `Recipe` object or an `async (ctx) => Recipe` factory; it loads via `core/src/recipes/ts-loader.ts` (`loadRecipeTs`), which mirrors the `.lando.ts` loader — reuses `sandboxScan` from `core/src/landofile/ts-loader.ts` (remapping `LandofileSandboxError` → `RecipeManifestParseError`), dynamic-imports through Bun's native TS loader (no `tsc`), unwraps the default export, and bounds evaluation with a 30s timeout (`LANDO_RECIPE_TS_TIMEOUT_MS`).
- The loaded value goes through the SAME validation as YAML: `validateRecipeManifestObject` (exported from `core/src/recipes/manifest/service.ts`) runs the Beta-rejection scan + strict `RecipeManifest` decode + semantic checks on the object. `resolveLocal` then enforces `manifest.id === <dir basename>`. The validated manifest rides on the new optional `ResolvedRecipe.manifest` field (with `manifestYaml: ""`); `parseResolvedRecipe` in `init.ts` short-circuits when it is present, so the manifest never gets re-parsed.
- `Recipe`/`RecipeContext`/`RecipeFactory` (types) and `defineRecipe` (identity helper) live in `sdk/src/schema/recipe.ts` (mirroring `defineLandofile`). `defineRecipe` is a value export so it must be in the API_COMPATIBILITY "Additive Alpha schema exports" list; the type aliases are erased and must NOT be listed (the compat test's third assertion does an exact `toEqual` against `Object.keys(sdkSchema)`). No schema-snapshot regen — nothing is added to `JSON_SCHEMA_REGISTRY`.
- A `recipe.ts` that imports `@lando/sdk` only resolves where that package is on the module path (the repo tree, or a project with it installed) — a bare temp dir cannot resolve it and fails with a clear `RecipeManifestParseError`. The portable form for tests/fixtures outside the repo is a plain `export default { … }` with no imports; `defineRecipe` is purely author-time type sugar. Local `recipe.ts` recipes still hit the shared "Recipe file rendering … is not implemented yet" Alpha boundary after a successful load+validate (same as local `recipe.yml`).

## Recipe prompts — dynamic `choicesFrom:`

- A `select`/`multiselect` prompt can source choices from a canonical Lando command via `choicesFrom: { command, args?, parse: "json"|"lines" }`. Execution lives in `core/src/recipes/prompts/choices-command.ts` behind an injectable `ChoicesCommandRunner`. The default runner re-invokes Lando as a child process using `landoInvocationPrefix(execPath, argv)`: compiled (`bun build --compile`) binaries embed the entry under `$bunfs` so `process.execPath` IS `lando` (prefix `[execPath]`); source mode prefixes `argv[1]` (the CLI entry). Do NOT set `BUN_BE_BUN` here — choicesFrom wants normal lando dispatch, not bun-mode (unlike `bun-self-runner.ts`).
- The dynamic-choices flow is implemented in `collectPrompts` (`runtime.ts`), not the manifest layer: it expands `choicesFrom` into `choices` at resolve time. A supplied `--answer` for a dynamic prompt is taken verbatim and never spawns the command. Command failure (exit ≠ 0 / spawn error / unparseable / empty) falls back to a manual free-text prompt when interactive, or uses the recipe default / fails with `RecipeChoicesError` when non-interactive. Because directory/remote recipes bail at the "rendering not implemented" boundary before `collectPrompts`, a full `lando init` never runs the dynamic-choices path yet — cover runtime behavior with an injected fake `choicesRunner` in unit tests.

## Renderer task-tree tail (interactive Lando renderer)

- The interactive (TTY) Lando renderer's per-task detail tail lives in `core/src/cli/renderer/task-tree-tail.ts`: `TaskDetailRing` (fixed-capacity, oldest-out) plus `LandoTreePainter`, a pure state machine driven by `task.*` events that returns the CSI byte chunk to write. It uses a **whole-frame redraw** model — it tracks how many rows the previous frame occupied, rewinds the cursor (`ESC[<n>A`), erases downward (`ESC[0J`), and repaints the entire active tree — rather than per-panel cursor accounting, so concurrent stacked sibling panels and collapsing frames work without bespoke choreography. Non-task-tree events route through `painter.passthrough(line)` so plain lines scroll above the live frame without corrupting the cursor.
- `makeLandoRendererLive(io)` branches on `RendererIO.isTTY`: `true` engages the painter; `undefined`/`false` falls back to `makePlainRendererLive` (the `[taskId] line` line-per-event contract). `createBufferedRendererIO()` leaves `isTTY` unset on purpose so existing buffered renderer tests stay on the plain path. Assert painter CSI bytes on the pure `LandoTreePainter` directly; assert renderer selection via a `{ ...createBufferedRendererIO(), isTTY: true }` double.
- `task.tree.start` seeds a first-paint **skeleton**: each declared `children` id becomes a `"pending"` `TaskState` (label = id) pushed to `#order` and rendered as a plain `  ◌ <id>` placeholder before its `task.start` arrives (which transitions it to `"running"` in place, preserving order via the existing `#tasks.has` guard). Pending lines MUST stay plain text — the painter's first paint carries no control bytes (the `renderer-task-tree-tail` "first paint emits no cursor-up" test asserts the chunk contains no `ESC[` at all), so do not dim or add CSI to the skeleton. `focusableTaskIds()` excludes `"pending"` so keyboard focus only lands on started tasks; `snapshot().activeTaskIds` and the `(n/m running)` denominator still count `"running"` only.
- The `--tail` / `--no-tail` toggle stays deferred in `core/src/cli/renderer-deferred.ts` (the ring buffer is fixed-on at depth 4 for Beta); do not wire a CLI flag for it.
- Expand/collapse keybindings live in `core/src/cli/renderer/keybindings.ts`: `parseKey` decodes raw stdin chunks (a lone `ESC` must stay distinct from the `ESC[A`/`ESC[B` CSI arrows — match the whole chunk), and `TaskTreeInputController` wraps a `LandoTreePainter` to track focus + expanded state, returning `{ events, changed, redraw }`. Expanding renders the full per-task stream (`fullStream` ring, bounded by `terminalRows`) and hides the tree; collapsing restores the 4-line `ring`. Drive the state machine in tests with key tokens or `handleInput(rawBytes)` against a real painter — no TTY needed.
- `task.detail.expand` / `task.detail.collapse` are deliberately absent from `format.ts`'s `isRenderableEvent` set: the TTY input layer publishes them on `EventService` for transcript/recording subscribers, but every renderer (plain/json/lando) must emit nothing for them, which is also what prevents a publish→consume echo loop. Do not add them to any renderable set.
- TTY key input is wired inside the renderer Live Layer (`makeTaskTreeInputLive` in `runtime.ts`), not at the CLI command boundary: it captures `EventService` + an `Effect.runtime()` so the stdin callback can `Runtime.runFork(events.publish(...))`, and registers the `io.subscribeInput` unsubscribe as a scope finalizer. `createStdioRendererIO` implements `subscribeInput` via `stdin` raw mode (no-op when `stdin.isTTY` is falsy); `createBufferedRendererIO({ isTTY: true, terminalRows })` exposes `injectKey` for fake-TTY tests.

## CLI dispatch parity (source OCLIF ↔ compiled `$bunfs`)

- `@oclif/core`'s `execute()` cannot dispatch inside a `bun build --compile` binary: `Config.load()` → `findRoot()` cannot find `package.json` next to a relocated single-file binary, and even with a static manifest, command dispatch still does a runtime `import()` of a computed path the bundler never embedded. Proven by the probe `core/test/cli/parity/oclif-static-probe.ts`. The dual-dispatch fork (`runCompiledCli`) is therefore structural, not provisional-pending-effort.
- A faithful compiled-`$bunfs` reproduction MUST run the binary from a dir OUTSIDE the repo tree — a binary sitting inside the repo lets `findRoot` walk up to the repo `package.json` and accidentally resolve on-disk commands.
- Source↔compiled output parity holds at two granularities (see `core/test/cli/dispatch-unification-spike.test.ts` + `parity/normalize.ts`): the `--renderer=json` event envelope is **byte-identical** across both paths (modulo `timestamp`/temp paths); plain/`lando` stderr is NOT byte-identical (source-mode OCLIF prefixes `Error:` + wraps at terminal width; the compiled path prints the raw block + `logsDir`/`cacheDir`). Assert byte-identical on JSON, identical tagged-error fields (`code`/`commandId`/`remediation`/`specSection`) on plain.
- Host-safe `meta:setup` parity: spawn with env `PATH=/no-such-path` + temp `LANDO_USER_CONF_ROOT`/`LANDO_USER_DATA_ROOT` + `--provider=podman` so setup fails fast at capability detection (`ProviderUnavailableError`) with no network fetch and no host mutation. The default `lando` provider instead attempts a network bundle download — avoid it in tests.
