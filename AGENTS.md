# AGENTS.md

This file holds non-obvious repo context. Keep it current. When you find a repo quirk worth noting, add the smallest useful note.

## Context

- Lando Core v4 is a Bun workspace for `@lando/core`, `@lando/sdk`, and bundled reference plugins under `plugins/*`.

## Commands

- Use Bun, not Node/npm/yarn/pnpm: `bun install`, `bun run typecheck`, `bun run lint`, `bun test`, `bun run build`.
- Run a focused test by path, e.g. `bun test core/test/unit/bootstrap.test.ts`.
- Root `typecheck` is `tsc -b`; it may create `dist/` and `.tsbuildinfo`. `bun run clean` removes generated package outputs.
- For a single workspace package, follow the repo's Bun filter style, e.g. `bun run --filter='@lando/core' typecheck`.
- After adding a new `plugins/*` workspace package, run `bun install` so root `node_modules` gets the workspace symlink before testing imports from the repo root.

## Gotchas

- **CLI fast path:** `core/src/cli/index.ts` must not statically import OCLIF, Effect, or transitives — ESM hoists imports before `import.meta.main`. Use dynamic `import()` from a wrapper. Full CLI: `core/bin/lando.ts`.
- **Compile entry:** `bun build --compile` must target `core/bin/lando.ts`, not `core/src/cli/index.ts` (the latter silently exits 0 for `--help` and all commands).
- **Compiled binary:** Until full OCLIF routing, each CLI command needs a matching handler in `core/src/cli/run.ts` (`$bunfs`), including matching error `remediation` and `NotImplementedError` behavior. Do not use `import.meta.url` for package metadata or install dir — use `core/src/version.ts` and `process.execPath` (`shellenv`).
- **OCLIF tests/manifest:** Fixture tests need `ignoreManifest: true` on `Config.load`. Generate manifest via `bun run codegen`, not `bunx oclif manifest` (Bun breaks on workspace TS symlinks).
- **Fresh CLI vs provider cache:** Stop/info/destroy in a new process cannot rely on in-memory applied plans; pass `AppPlan` via `AppSelector.plan` / `ServiceSelector.plan`.
- **FileSystem:** `writeAtomic` is copy-based, not crash-atomic rename. `remove` deletes files only, not directories.
- **CI codegen:** Bun version comes from `.bun-version`, not `package.json#engines.bun` (a semver range broke lockfile alignment).
- **Generated TS code formatting:** Codegen scripts that write into `core/src/**` must end with `biome check --write <output>` (see `scripts/build-bundled-recipes.ts`), not just `biome format --write`. Biome's import-line wrapping rule **and** its organize-imports rule fire on the lint gate and the test harness `core/test/build/biome.test.ts`; `format` alone misses organize-imports and turns into a lint failure as soon as a bundled set grows past a handful of entries. The drift gate (`bun run codegen` + `git diff --exit-code`) only catches drift if the generator output is already biome-clean.
- **Recipe source resolution:** `core/src/recipes/source.ts#resolveRecipeRef` returns an Effect that either succeeds with a `ResolvedRecipe` (`{ id, source, manifestYaml, root }`) for bare-id bundled recipes (`BUNDLED_RECIPES`) and local `./path`/`/abs/path` directories, or fails with `NotImplementedError(commandId: "recipe.source.resolve", specSection: "§8.8.4")` for `github:`/`git+`/`git@`/`git://`/`npm:`/`registry:` schemes. Built-in recipe ids are matched by `/^[a-z0-9][a-z0-9-]*$/`; anything else (uppercase, tarball/http URLs, unknown prefixes, empty string) falls through to "unknown" scheme rejection — still tagged `NotImplementedError` with the same Beta remediation. For local directory refs the resolver enforces §8.8.3's hard requirement that `recipe.yml`'s top-level `id:` match the directory basename and rejects with `RecipeManifestValidationError` if not. Local IO is wrapped with `Effect.tryPromise` so EACCES/EIO failures stay tagged (`RecipeManifestNotFoundError`) instead of escaping as Effect defects. The compiled `$bunfs` `init` handler and the source OCLIF `apps:init` command both unwrap the Effect via `Effect.runPromiseExit` + `Cause.failureOption` so callers see the tagged error rather than `FiberFailure`.
- **Landofile TS form:** `.lando.ts` is loaded by `core/src/landofile/ts-loader.ts` next to the YAML path; discovery walks both forms per directory and rejects "both present" with a tagged `LandofileParseError`. The loader does a STATIC sandbox scan via `Bun.Transpiler.scan({ loader: "ts" })` plus a defensive `require()` regex that covers bare `require("x")`, parenthesized `(require)("x")`, and template-literal `` require(`x`) `` forms across single/double/back-tick string args (ESM-mode TS hides `require` from `Bun.Transpiler.scan`). Forbidden node builtins (`fs`, `fs/promises`, `child_process`, `http`/`https`/`http2`, `net`, `tls`, `dgram`, `cluster`, `worker_threads`, `vm`, `dns`, `dns/promises`, `repl`, `inspector`), URL-scheme imports (anything that isn't `node:`), and relative imports that resolve outside the app root surface as `LandofileSandboxError` with a §7.1.1 remediation. The whole `import(...) + default-export resolution` is wrapped in `Effect.timeoutFail(LandofileTimeoutError)`; the default 5000ms timeout is overridable via the `LANDO_LANDOFILE_TS_TIMEOUT_MS` environment variable. The default-export resolver supports value, function, `Promise<...>`, and `Effect.Effect<...>` return shapes (Effects are run via `Effect.runPromise` inside the same timeout). TS-form output is piped through the same `rejectBetaTopLevelKeys` / `rejectBetaToolingFeatures` gates as the YAML path, so `{ includes: [...] }` or Beta tooling fields returned by a TS Landofile surface as the same `NotImplementedError(commandId: "landofile.parse", specSection: "§7.7" | …)` with the Beta remediation. Extending `LandofileService.discover`'s error union transitively widens every CLI command error alias (`StartAppError`/`StopAppError`/etc.); update them in the same change or `tsc` will fail under `exactOptionalPropertyTypes: true`. **Trust boundary:** the static scan is preflight only, not a real sandbox — `Bun.spawn`/`Bun.write`/`fetch`/`process.env`/`eval`/`new Function` and top-level side effects (`console.log`, top-level `await`) all execute inside the host process during `import(...)`. Treat `.lando.ts` as trusted author-code.
- **Recipe prompts:** `core/src/recipes/prompts/` exposes `collectPrompts` plus a `PromptIO` seam (`createStdioPromptIO` for prod, `createBufferedPromptIO({ inputs })` for tests). Per-type prompt tests should use the buffered IO to script answers and assert `io.stdout()`/`io.stderr()` — no `Bun.spawn` round-trip needed. CLI integration tests of `lando init` that DO drive subprocess prompts MUST pass `stdin: "pipe"` and write `\n`-separated answers; tests that DON'T script stdin MUST set `stdin: "ignore"` AND pass `--no-interactive`, otherwise `Bun.spawn`'s default `"inherit"` reads from the parent test runner's stdin and hangs. OCLIF wraps long error messages at ~80 columns and indents continuation lines with `\n    `; substring assertions over CLI stderr should normalize whitespace first (`text.replace(/\s+/g, " ")`). `--full` on `lando init` is a confirmation gate only; use `--yes` for defaults-only and `--no-interactive` to fail fast on missing answers.
- **Init recipe selection:** When `--recipe` is omitted in interactive mode (not `--no-interactive`, not `--yes`), `initApp` drives a synthetic `select`-type prompt over the `BUNDLED_RECIPES` catalog before parsing the manifest. `--no-interactive` / `--yes` without `--recipe` fall back to `NODE_POSTGRES_RECIPE_ID`. Picker metadata (title / description) is extracted lazily from each `manifestYaml` string via regex in `core/src/recipes/catalog.ts` so the compiled `$bunfs` binary needs no runtime FS access for the recipe list. CLI subprocess tests that script stdin MUST answer the recipe-selection prompt first (blank line accepts the default), then the recipe's own prompts. In-process `initApp` calls from tests that don't supply a `PromptIO` and don't set `nonInteractive: true` will hang on `process.stdin` waiting for the recipe pick; tests that exercise post-prompt failure paths (existing-directory guard, etc.) MUST pass `nonInteractive: true`.
- **EventService no-replay subscribers:** Use `EventService.subscribeQueue` (a scoped Effect that resolves to a `Queue.Dequeue<LandoEvent>` once `PubSub.subscribe` has handshaken) for any consumer that must not miss a `publish` from a sibling fiber issued in the same tick. The older `subscribe(name)` is a `Stream` that subscribes lazily on first pull, so publishes that beat the consumer's first take are dropped. Renderers (`core/src/cli/renderer/runtime.ts#makeRendererLive`) use `subscribeQueue` inside `Layer.scopedDiscard`, fork a `Queue.take` loop, and add a finalizer that drains `Queue.takeAll` before interrupting the consumer fiber — without the drain, events published in the same tick as scope close get swallowed. Extending the `EventService` interface (new fields/methods) breaks every in-tree mock; sweep `Layer.succeed(EventService, { … })` literals in `core/test/cli/*.scenario.test.ts` and `sdk/test/services/tags.test.ts` in the same change or `tsc -b` fails.
- **Renderer modules:** `core/src/cli/renderer/` is the home for renderer plumbing (`io.ts` for the `RendererIO` seam, `format.ts` for pure `renderPlainLine` / `renderJsonLine`, `runtime.ts` for the `make{Plain,Json,Lando}RendererLive` Layers). Plain renderer writes to stdout, JSON renderer writes NDJSON to stderr, `lando` aliases plain until rich-TTY behavior lands. Tests inject `createBufferedRendererIO()` and assert on `stdout()` / `stderr()` / `stdoutLines()` directly — no `Bun.spawn` round trip needed. Biome `lint/suspicious/noControlCharactersInRegex` rejects `\u001b` literals in regex; build escape chars via `String.fromCharCode(27)` and use `String.prototype.includes` for ANSI-presence checks instead. Biome also rejects empty template literals — wrap the escape in a `${var}` interpolation or use string concatenation.
- **Message events (`message.info`/`warn`/`error`):** Schemas live in `sdk/src/events/index.ts` as `MessageInfoEvent` / `MessageWarnEvent` / `MessageErrorEvent` with `body: string` (already-redacted by publisher) and, for `message.error`, an optional `remediation: string`. The plain formatter prefixes `ℹ` / `⚠` / `✗` and renders error remediation on a continuation line with `  ↳ <remediation>` (two-space indent, distinct from `task.fail`'s `[taskId]   ↳` form because messages are not task-scoped). Adding ANY new `LandoEvent` discriminant requires three lock-step edits or the event silently no-ops at the renderer: (1) the TaggedStruct + type alias in `sdk/src/events/index.ts`; (2) adding the member to the `LandoEvent = Schema.Union(...)` list at the bottom of the same file (this is the one `tsc -b` will catch); (3) extending `isTaskTreeEvent` / `isMessageEvent` in `core/src/cli/renderer/format.ts` AND the `formatPlainEvent` switch — without that, `renderPlainLine` / `renderJsonLine` return `null` and the renderer drops the event without warning. Any new top-level event field that should sort before `timestamp` in NDJSON output also needs to be inserted into `orderedKeys` in `format.ts#stableStringify`. Contract: `message.info` / `message.warn` / `message.error` are pure presentation events — they MUST NOT mutate `process.exitCode`. Tests that touch `process.exitCode` MUST save the prior value AND set a non-`undefined` sentinel before the assertion: other tests in `core/test/cli/*.scenario.test.ts` leave `process.exitCode = 0` (or non-zero) globally, so asserting `.toBeUndefined()` after the program runs is flaky.
- **Renderer flag:** `--renderer=lando|json|plain` is validated by `core/src/cli/renderer-selection.ts#resolveRendererMode` at THREE entry points so both CLI paths reject identically across the `--help` / `--version` short-circuit too: (a) `runCli` in `core/src/cli/run.ts` validates at the TOP, BEFORE `execute()` hands off to OCLIF — OCLIF's `--help` / `--version` short-circuit otherwise swallows errors thrown from the OCLIF init hook AND from `LandoCommandBase.runEffect` (the command class never instantiates), which means `--renderer=tui --help` would silently print help with exit 0 without this gate. The source-mode gate skips when argv head is a `bun` / `meta:bun` / `x` / `meta:x` passthrough; (b) `LandoCommandBase.runEffect` mutates `this.argv` in place BEFORE the early command guard / `this.parse(this.ctor)` so OCLIF's per-command flag parser never sees the unknown flag — accepted position is AFTER the command id in source mode; (c) `runCompiledCli` in `core/src/cli/run.ts` resolves at the top of the function BEFORE the `--help` / `--version` / per-command short-circuits, with the same `bun` / `meta:bun` / `x` / `meta:x` passthrough. `meta:bun` / `meta:x` Command subclasses override `run()` and never call `runEffect`, so they are automatically renderer-passthrough in source mode without an explicit exemption. `extractRendererFlag` honors the POSIX `--` argument terminator and stops consuming `--renderer=*` tokens after it (so `app:exec -- bash -c 'echo --renderer=plain'` forwards the child arg verbatim). Selection precedence is flag > `LANDO_RENDERER` env > `configValue` arg > default `"lando"`. The `configValue` arg is a helper seam for the global `renderer:` config layer; end-to-end wiring of `GlobalConfig.renderer` into source/compiled callers is deferred for now. The helper-level precedence is exercised by unit tests, callers currently pass only `argv` + `env`. Invalid values raise tagged `RendererSelectionError` (with `value` + `source: "flag" | "env" | "config"` + remediation) BEFORE command execution; the source error union goes through `Error(commandErrorMessage(...))` while the compiled and `runCli`-source paths print + `process.exitCode = 1`. Both `commandErrorMessage` copies (one per file) carry a matching `_tag === "RendererSelectionError"` block that does `details.unshift(tag)` to lead with the tag name like other tagged errors. CLI subprocess tests that re-spawn `bun run build` must prepend `dirname(process.execPath)` to `PATH` or the child shell fails with exit 127 on host distros lacking `/usr/local/bin/bun`.
