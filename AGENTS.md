# AGENTS.md

This file holds non-obvious repo context. Keep it current. When you find a repo quirk worth noting, add the smallest useful note.

## Context

- Lando Core v4 is a Bun workspace for `@lando/core`, `@lando/sdk`, and bundled reference plugins under `plugins/*`.

## Commands

- Use Bun, not Node/npm/yarn/pnpm: `bun install`, `bun run typecheck`, `bun run lint`, `bun test`, `bun run build`.
- Run a focused test by path, e.g. `bun test core/test/unit/bootstrap.test.ts`.
- Root `typecheck` is `tsc -b`; it may create `dist/` and `.tsbuildinfo`. `bun run clean` removes generated package outputs.
- For a single workspace package, follow the repo's Bun filter style, e.g. `bun run --filter='@lando/core' typecheck`.
- After adding a new `plugins/*` workspace package, run `bun install` so root `node_modules` gets the workspace symlink before testing imports from the repo root.

## Gotchas

- **CLI fast path:** `core/src/cli/index.ts` must not statically import OCLIF, Effect, or transitives — ESM hoists imports before `import.meta.main`. Use dynamic `import()` from a wrapper. Full CLI: `core/bin/lando.ts`.
- **Compile entry:** `bun build --compile` must target `core/bin/lando.ts`, not `core/src/cli/index.ts` (the latter silently exits 0 for `--help` and all commands).
- **Compiled binary:** Until full OCLIF routing, each CLI command needs a matching handler in `core/src/cli/run.ts` (`$bunfs`), including matching error `remediation` and `NotImplementedError` behavior. Do not use `import.meta.url` for package metadata or install dir — use `core/src/version.ts` and `process.execPath` (`shellenv`).
- **OCLIF tests/manifest:** Fixture tests need `ignoreManifest: true` on `Config.load`. Generate manifest via `bun run codegen`, not `bunx oclif manifest` (Bun breaks on workspace TS symlinks).
- **Fresh CLI vs provider cache:** Stop/info/destroy in a new process cannot rely on in-memory applied plans; pass `AppPlan` via `AppSelector.plan` / `ServiceSelector.plan`.
- **FileSystem vs CacheService writeAtomic:** `FileSystem.writeAtomic` is copy-based, not crash-atomic rename. `CacheService.writeAtomic` (`core/src/cache/atomic.ts#writeFileAtomicViaRename`, added by US-042) IS crash-atomic via `node:fs/promises.rename` with a per-call `${path}.tmp-<randomUUID>` temp file. The CacheService variant is the one binary caches under `<userCacheRoot>/` go through (`app-plan`, plugin/app command indexes, etc.); FileSystem is still copy-based. `FileSystem.remove` deletes files only, not directories.
- **App-plan cache cross-project safety:** `appPlanCachePath(cacheRoot, appName, appRoot)` (`core/src/cache/paths.ts`) namespaces by `appName-<sha256(appRoot).slice(0,12)>` and `deriveAppPlanCacheKey` includes `appRoot` in the hash so two projects with the same `name:` in different directories never read or overwrite each other's plan (US-042 review fix). `AppPlannerLive` in `makeAppRuntimeLive` MUST receive `CacheServiceLive` via `Layer.provide(Layer.mergeAll(PluginRegistryLive, CacheServiceLive))` — `Effect.serviceOption(CacheService)` resolves at layer-build time against the sub-layer's provided context, not against `Layer.mergeAll` siblings, so providing CacheService only at the sibling level silently disables the cache in production.
- **Binary cache encoding fixtures:** Header/schema changes for binary caches must update both exported header constants in `core/src/cache/*` and the committed fixtures under `core/test/cache/fixtures/binary-cache/*.bin`; `core/test/cache/binary-encoding.test.ts` compares current encoder output byte-for-byte against those fixtures.
- **CI codegen:** Bun version comes from `.bun-version`, not `package.json#engines.bun` (a semver range broke lockfile alignment).
- **SDK public schema exports:** Any new export from `@lando/sdk/schema` must also be listed under "Additive Alpha schema exports" in `sdk/API_COMPATIBILITY.md`; `sdk/test/library/sdk-backward-compatibility.test.ts` fails if the additive surface is undocumented.
- **CI drift gates:** Generated CI jobs live in `scripts/build-ci-workflow.ts`; codegen/snapshot gates should run focused `bun run codegen:*` commands followed by `git diff --exit-code -- <generated files>`. Generators that emit JSON fixtures (for example `scripts/build-schema-snapshot.ts`) must format the output via `biome check --write <output>` before returning, or `bun run lint` rejects the fixture.
- **npm dev package workflow:** Generated release jobs live in `scripts/build-release-workflow.ts`; `scripts/prepare-npm-dev-packages.ts` temporarily rewrites CI checkout package metadata so `@lando/sdk` publishes as the same-version `dev` dependency before `@lando/core`, then `@lando/core` publishes with `--tag dev` while asserting `latest` is unchanged. Do not commit alpha versions or non-private package flags to workspace `package.json` files.
- **Generated TS code formatting:** Codegen scripts that write into `core/src/**` must end with `biome check --write <output>` (see `scripts/build-bundled-recipes.ts`), not just `biome format --write`. Biome's import-line wrapping rule **and** its organize-imports rule fire on the lint gate and the test harness `core/test/build/biome.test.ts`; `format` alone misses organize-imports and turns into a lint failure as soon as a bundled set grows past a handful of entries. The drift gate (`bun run codegen` + `git diff --exit-code`) only catches drift if the generator output is already biome-clean.
- **Guide scenario MDX parser:** `scripts/build-guide-scenarios.ts` walks MDX with `unified@^11.0.5`, `remark-parse@^11.0.0`, `remark-mdx@^3.1.1`, and `remark-frontmatter@^5.0.0` (plus `@mdx-js/mdx@^3.1.1` recorded for the MDX toolchain). The walker consumes `mdxJsxFlowElement` nodes and `yaml` frontmatter nodes with source positions; do not swap parser stacks without updating fixtures/tests because AST node shapes are part of the generator contract.
- **CI codegen tests:** `core/test/scripts/codegen-ci.test.ts` shells out to `bun run codegen`, so local focused runs can leave unrelated generated fixtures dirty (for example schema snapshots). Check `git status` after running it and restore unrelated generated drift before committing.
- **Recipe source resolution:** `core/src/recipes/source.ts#resolveRecipeRef` returns an Effect that either succeeds with a `ResolvedRecipe` (`{ id, source, manifestYaml, root }`) for bare-id bundled recipes (`BUNDLED_RECIPES`) and local `./path`/`/abs/path` directories, or fails with `NotImplementedError(commandId: "recipe.source.resolve", specSection: "§8.8.4")` for `github:`/`git+`/`git@`/`git://`/`npm:`/`registry:` schemes. Built-in recipe ids are matched by `/^[a-z0-9][a-z0-9-]*$/`; anything else (uppercase, tarball/http URLs, unknown prefixes, empty string) falls through to "unknown" scheme rejection — still tagged `NotImplementedError` with the same Beta remediation. For local directory refs the resolver enforces §8.8.3's hard requirement that `recipe.yml`'s top-level `id:` match the directory basename and rejects with `RecipeManifestValidationError` if not. Local IO is wrapped with `Effect.tryPromise` so EACCES/EIO failures stay tagged (`RecipeManifestNotFoundError`) instead of escaping as Effect defects. The compiled `$bunfs` `init` handler and the source OCLIF `apps:init` command both unwrap the Effect via `Effect.runPromiseExit` + `Cause.failureOption` so callers see the tagged error rather than `FiberFailure`.
- **Landofile TS form:** `.lando.ts` is loaded by `core/src/landofile/ts-loader.ts` next to the YAML path; discovery walks both forms per directory and rejects "both present" with a tagged `LandofileParseError`. The loader does a STATIC sandbox scan via `Bun.Transpiler.scan({ loader: "ts" })` plus a defensive `require()` regex that covers bare `require("x")`, parenthesized `(require)("x")`, and template-literal `` require(`x`) `` forms across single/double/back-tick string args (ESM-mode TS hides `require` from `Bun.Transpiler.scan`). Forbidden node builtins (`fs`, `fs/promises`, `child_process`, `http`/`https`/`http2`, `net`, `tls`, `dgram`, `cluster`, `worker_threads`, `vm`, `dns`, `dns/promises`, `repl`, `inspector`), URL-scheme imports (anything that isn't `node:`), and relative imports that resolve outside the app root surface as `LandofileSandboxError` with a §7.1.1 remediation. The whole `import(...) + default-export resolution` is wrapped in `Effect.timeoutFail(LandofileTimeoutError)`; the default 5000ms timeout is overridable via the `LANDO_LANDOFILE_TS_TIMEOUT_MS` environment variable. The default-export resolver supports value, function, `Promise<...>`, and `Effect.Effect<...>` return shapes (Effects are run via `Effect.runPromise` inside the same timeout). TS-form output is piped through the same `rejectBetaTopLevelKeys` / `rejectBetaToolingFeatures` gates as the YAML path, so `{ includes: [...] }` or Beta tooling fields returned by a TS Landofile surface as the same `NotImplementedError(commandId: "landofile.parse", specSection: "§7.7" | …)` with the Beta remediation. Extending `LandofileService.discover`'s error union transitively widens every CLI command error alias (`StartAppError`/`StopAppError`/etc.); update them in the same change or `tsc` will fail under `exactOptionalPropertyTypes: true`. **Trust boundary:** the static scan is preflight only, not a real sandbox — `Bun.spawn`/`Bun.write`/`fetch`/`process.env`/`eval`/`new Function` and top-level side effects (`console.log`, top-level `await`) all execute inside the host process during `import(...)`. Treat `.lando.ts` as trusted author-code.
- **Generated guide scenario runner:** Authored `<Run command="lando ...">` commands are normalized by `ScenarioContext.runCli` (leading `lando` stripped) for the in-process runner. After generated `init` and `UseFixture` calls, the ScenarioContext working directory advances to the scaffolded app / copied fixture so later `lando start` and `lando destroy` commands run in the expected directory; generated files under `test/scenarios/generated/guides/**` remain gitignored and must not be committed.
- **Recipe prompts:** `core/src/recipes/prompts/` exposes `collectPrompts` plus a `PromptIO` seam (`createStdioPromptIO` for prod, `createBufferedPromptIO({ inputs })` for tests). Per-type prompt tests should use the buffered IO to script answers and assert `io.stdout()`/`io.stderr()` — no `Bun.spawn` round-trip needed. CLI integration tests of `lando init` that DO drive subprocess prompts MUST pass `stdin: "pipe"` and write `\n`-separated answers; tests that DON'T script stdin MUST set `stdin: "ignore"` AND pass `--no-interactive`, otherwise `Bun.spawn`'s default `"inherit"` reads from the parent test runner's stdin and hangs. OCLIF wraps long error messages at ~80 columns and indents continuation lines with `\n    `; substring assertions over CLI stderr should normalize whitespace first (`text.replace(/\s+/g, " ")`). `--full` on `lando init` is a confirmation gate only; use `--yes` for defaults-only and `--no-interactive` to fail fast on missing answers.
- **Init recipe selection:** When `--recipe` is omitted in interactive mode (not `--no-interactive`, not `--yes`), `initApp` drives a synthetic `select`-type prompt over the `BUNDLED_RECIPES` catalog before parsing the manifest. `--no-interactive` / `--yes` without `--recipe` fall back to `NODE_POSTGRES_RECIPE_ID`. Picker metadata (title / description) is extracted lazily from each `manifestYaml` string via regex in `core/src/recipes/catalog.ts` so the compiled `$bunfs` binary needs no runtime FS access for the recipe list. CLI subprocess tests that script stdin MUST answer the recipe-selection prompt first (blank line accepts the default), then the recipe's own prompts. In-process `initApp` calls from tests that don't supply a `PromptIO` and don't set `nonInteractive: true` will hang on `process.stdin` waiting for the recipe pick; tests that exercise post-prompt failure paths (existing-directory guard, etc.) MUST pass `nonInteractive: true`.
- **EventService no-replay subscribers:** Use `EventService.subscribeQueue` (a scoped Effect that resolves to a `Queue.Dequeue<LandoEvent>` once `PubSub.subscribe` has handshaken) for any consumer that must not miss a `publish` from a sibling fiber issued in the same tick. The older `subscribe(name)` is a `Stream` that subscribes lazily on first pull, so publishes that beat the consumer's first take are dropped. Renderers (`core/src/cli/renderer/runtime.ts#makeRendererLive`) use `subscribeQueue` inside `Layer.scopedDiscard`, fork a `Queue.take` loop, and add a finalizer that drains `Queue.takeAll` before interrupting the consumer fiber — without the drain, events published in the same tick as scope close get swallowed. Extending the `EventService` interface (new fields/methods) breaks every in-tree mock; sweep `Layer.succeed(EventService, { … })` literals in `core/test/cli/*.scenario.test.ts` and `sdk/test/services/tags.test.ts` in the same change or `tsc -b` fails.
- Extending any SDK service tag method surface (for example `FileSystem`) also requires updating all Live/runtime/test doubles, `sdk/test/services/tags.test.ts`, and the frozen fixture `sdk/test/fixtures/sdk-mvp-surface.json` or the SDK backward-compatibility gate fails.
- `makeLandoRuntime({ config: ... })` currently validates the option shape but `ConfigServiceLive` still resolves config roots through defaults/env; tests needing isolated roots should set and restore `LANDO_USER_CONF_ROOT` / `LANDO_USER_DATA_ROOT` or override `ConfigService` through `plugins.layers`.
- **Renderer modules:** `core/src/cli/renderer/` is the home for renderer plumbing (`io.ts` for the `RendererIO` seam, `format.ts` for pure `renderPlainLine` / `renderJsonLine`, `runtime.ts` for the `make{Plain,Json,Lando}RendererLive` Layers). Plain renderer writes to stdout, JSON renderer writes NDJSON to stderr, `lando` aliases plain until rich-TTY behavior lands. Tests inject `createBufferedRendererIO()` and assert on `stdout()` / `stderr()` / `stdoutLines()` directly — no `Bun.spawn` round trip needed. Biome `lint/suspicious/noControlCharactersInRegex` rejects `\u001b` literals in regex; build escape chars via `String.fromCharCode(27)` and use `String.prototype.includes` for ANSI-presence checks instead. Biome also rejects empty template literals — wrap the escape in a `${var}` interpolation or use string concatenation.
- **Pre-renderer module (`core/src/cli/oclif/pre-renderer.ts`):** Tiny sync helper that writes the §8.9.1 first-paint banner directly to stdout. To preserve the §2.1 cold first-byte budget and the existing `core/test/cli/fast-path-canary-preload.ts` "no Effect on the fast path" canary, the source MUST NOT import Effect, `@oclif/core`, anything under `@lando/sdk` (the SDK transitively pulls `effect`), anything under `core/src/cli/renderer/`, or any plugin code — node builtins are the only permitted dependency. `core/test/cli/paint-banner.test.ts` enforces this with a static source scan that walks `from "..."` / `import("...")` / `require("...")` / bare `import "..."` grammars; widen the forbidden-prefix list if a new fast-path-poisoning dependency appears (e.g. a future bundled-plugins surface). The companion `paint.banner` `LandoEvent` lives in `sdk/src/events/index.ts` and is consumed by `core/src/cli/renderer/format.ts`: plain/lando renderers return `null` for it (banner is already on stdout from the pre-renderer) and the json renderer emits exactly one NDJSON line on stderr; `banner` is registered in `orderedKeys` so it sorts deterministically before `timestamp`.
- **Message events (`message.info`/`warn`/`error`):** Schemas live in `sdk/src/events/index.ts` as `MessageInfoEvent` / `MessageWarnEvent` / `MessageErrorEvent` with `body: string` (already-redacted by publisher) and, for `message.error`, an optional `remediation: string`. The plain formatter prefixes `ℹ` / `⚠` / `✗` and renders error remediation on a continuation line with `  ↳ <remediation>` (two-space indent, distinct from `task.fail`'s `[taskId]   ↳` form because messages are not task-scoped). Adding ANY new `LandoEvent` discriminant requires three lock-step edits or the event silently no-ops at the renderer: (1) the TaggedStruct + type alias in `sdk/src/events/index.ts`; (2) adding the member to the `LandoEvent = Schema.Union(...)` list at the bottom of the same file (this is the one `tsc -b` will catch); (3) extending `isTaskTreeEvent` / `isMessageEvent` in `core/src/cli/renderer/format.ts` AND the `formatPlainEvent` switch — without that, `renderPlainLine` / `renderJsonLine` return `null` and the renderer drops the event without warning. Any new top-level event field that should sort before `timestamp` in NDJSON output also needs to be inserted into `orderedKeys` in `format.ts#stableStringify`. Contract: `message.info` / `message.warn` / `message.error` are pure presentation events — they MUST NOT mutate `process.exitCode`. Tests that touch `process.exitCode` MUST save the prior value AND set a non-`undefined` sentinel before the assertion: other tests in `core/test/cli/*.scenario.test.ts` leave `process.exitCode = 0` (or non-zero) globally, so asserting `.toBeUndefined()` after the program runs is flaky.
- **Deferred-command remediation:** `core/src/cli/deferred-commands.ts` is the SINGLE SOURCE OF TRUTH for the `NotImplementedError` payload (`message` + `commandId` + `specSection` + `remediation`) that every non-MVP canonical command id surfaces. Both the source OCLIF guard (`LandoCommandBase.runEffect` in `core/src/cli/oclif/command-base.ts`) and the compiled `$bunfs` fallback (the `findCommand` tail in `core/src/cli/run.ts#runCompiledCli`) re-export the same `notImplementedErrorForCommand` so a single edit updates both paths. Adding a new non-MVP command id requires three lock-step edits or the test gate (`core/test/cli/deferred-commands.test.ts`) fails: (1) register the OCLIF stub under `core/src/cli/oclif/commands/<namespace>/<topic>/<verb>.ts` — use the `makeDeferredStubCommand` helper in `core/src/cli/oclif/commands/deferred-stub.ts` instead of hand-rolling another 17-line stub; (2) add the id to `core/src/cli/oclif/compiled-commands.ts`; (3) add a `DEFERRED_COMMAND_PLANS` entry plus, if it appears in the AC fixture, an entry in `core/test/cli/fixtures/deferred-commands.json`. Remediation text MUST name the target roadmap phase verbatim (`Phase 3 Beta` / `Phase 4 RC`) and reference `spec/ROADMAP.md`; the deferred-commands test explicitly rejects stack frames and absolute source-file paths in remediation. Source-mode stderr arrives with OCLIF chrome (`Error:` prefix + 4-space indent + 80-col wrap) while compiled-mode stderr is the raw `console.error(commandErrorMessage(...))` output; assertions over both paths must normalize through the `extractErrorBlock` + whitespace-collapse helper in the deferred-commands test before comparing.
- **Bug-report failure formatter:** `core/src/cli/bug-report.ts#formatBugReport` is the SINGLE SOURCE OF TRUTH for the diagnostic block every CLI failure surfaces. It returns multi-line text for `plain`/`lando` modes and a single NDJSON `{ _tag: "message.error", code, commandId, body, remediation?, appId?, providerId?, logsDir, cacheDir, ...extra, timestamp }` line for `json` mode. Both source OCLIF (`LandoCommandBase.runEffect` in `core/src/cli/oclif/command-base.ts`) and the compiled `$bunfs` dispatcher (`runCompiledCli` in `core/src/cli/run.ts`) call it. The compiled path threads `commandId` via a module-scoped `activeCommandId` set once at the top of `runCompiledCli` by `setActiveCommandId(resolveCanonicalCommandId(argv[0]))` against the `CANONICAL_COMMAND_ID_BY_TOKEN` map; per-runner `console.error(commandErrorMessage(failure.value))` sites stay one-line because `commandErrorMessage`'s `commandId` parameter defaults to that variable. Adding a new top-level dispatch arm requires extending `CANONICAL_COMMAND_ID_BY_TOKEN` with BOTH the canonical id key and any short alias (e.g. `start` → `app:start`) or the failure path will report `commandId: <alias-or-unknown>` instead of the canonical id. **JSON-mode chrome bypass:** in source mode, the JSON failure line MUST be written directly via `process.stderr.write(\`${text}\\n\`)` + `process.exitCode = 1` + `return` from `runEffect` — throwing wraps the JSON in OCLIF's `Error:` 80-col chrome and breaks NDJSON consumers. This applies to BOTH the deferred-stub `isCanonicalLandoCommandId && !isMvpCommandId` branch and the runtime-failure branch. Extending the tagged-error-specific diagnostic lines (the `key: value` block under the body) goes through `extractExtraTagFields` in `bug-report.ts`; both plain and JSON renderers pick up new entries automatically (but `issues` and any multi-line value is deliberately skipped from the JSON shape to keep it one-line). Redaction lives in `core/src/cli/redact.ts` as a deliberate copy of `plugins/provider-lando/src/redact.ts` — bit-identical regex pair; if policy ever diverges, unify them via a new `@lando/sdk/redact` submodule rather than importing across the core ↔ plugin boundary.
- **Provider-side task events:** `RuntimeProviderRegistryLive` resolves `EventService` from context via `Effect.serviceOption` and threads the resulting `EventPublisher` into `makeLandoRuntimeProvider({ eventService })` whenever EventService is present. This is what makes `setupProviderLando` (and `bringUp`) emit task tree events during real CLI runs via the registry path — the publisher seam in the provider was already there, but until the registry was taught to inject it from the runtime layer, `lando setup` / `lando start` published into the void. `makeProviderRuntimeLive` (in `core/src/runtime/layer.ts`) includes `EventServiceLive` so the `provider` bootstrap level (which `meta:setup` uses) has EventService available. The registry's Live Layer takes EventService as an optional dep — tests that bypass Live via `Layer.succeed(RuntimeProviderRegistry, fakeRegistry)` are unaffected; tests that exercise `setupSpec.run` against a real `makeRuntimeProvider({ eventService: sink.publish })` see all task tree events on the sink (`core/test/cli/setup.scenario.test.ts`). The remaining wiring gap — actually subscribing a renderer Layer to the published events so the user sees them on stdout/stderr — is still owned by a follow-up renderer-integration story; the events flow through `EventService` correctly today, but no consumer in the source OCLIF (`LandoCommandBase.runEffect`) or compiled `$bunfs` (`runCompiledCli`) attaches `makePlainRendererLive` / `makeJsonRendererLive` / `makeLandoRendererLive` over the runtime yet.
- **Deferred renderer surfaces:** `core/src/cli/renderer-deferred.ts` is the SINGLE SOURCE OF TRUTH for renderer flags / mode values that touch §8.9 features deferred past Phase 2 Alpha per `spec/alpha/prd-alpha-05-renderer-errors-and-diagnostics.md` US-040. Two tables: `DEFERRED_RENDERER_MODES` (mode-value → Phase 3 Beta/Phase 4 RC plan; currently only `verbose`) and `DEFERRED_RENDERER_FLAGS` (CLI flag name → Phase 4 RC plan; currently `--expand` / `--no-expand` / `--collapse` / `--no-collapse` / `--tail` / `--no-tail`). `core/src/cli/renderer-selection.ts#validate` consults `isDeferredRendererMode` BEFORE `isRendererMode` so deferred modes throw tagged `NotImplementedError(commandId: "cli:renderer-selection", specSection: "spec/08-cli-and-tooling.md")` with phase remediation instead of the generic `RendererSelectionError`; `extractRendererFlag` calls `findDeferredRendererFlag` BEFORE parsing `--renderer` so deferred flags surface the same way regardless of argv ordering, instead of an OCLIF "unknown flag" message. The thrown `NotImplementedError` is caught by the same four catch blocks that handle `RendererSelectionError` (`LandoCommandBase.runEffect`, `runCli` + `runCompiledCli` in `core/src/cli/run.ts`, `apps/init.ts`) — each block widens its `instanceof` check to `RendererSelectionError || NotImplementedError`; if you add a new renderer-selection entry point you MUST broaden its catch too or the error escapes as an uncaught defect. `formatBugReport` already formats the tagged payload via `extractExtraTagFields`'s generic `specSection` branch, so no per-tag formatter change is needed. When adding a new deferred renderer surface, update three things in lock-step or the test gate (`core/test/cli/renderer-deferred.test.ts`) fails: (1) the entry in `DEFERRED_RENDERER_MODES` / `DEFERRED_RENDERER_FLAGS`; (2) the matching row in the US-040 "Deferred renderer surfaces" table in `spec/alpha/prd-alpha-05-renderer-errors-and-diagnostics.md`; (3) per-source assertions in the test suite. Remediation text MUST name the phase verbatim (`Phase 3 Beta` / `Phase 4 RC`) and reference both `spec/ROADMAP.md` and `spec/08-cli-and-tooling.md`; source/compiled parity is byte-identical because both paths reuse `commandErrorMessage` → `formatBugReport`. Existing tests for unknown renderer values (`--renderer=tui` in `core/test/cli/renderer-flag.scenario.test.ts`) deliberately use a value NOT in the deferred set so they continue to validate the `RendererSelectionError` fallthrough path.
- **Renderer flag:** `--renderer=lando|json|plain` is validated by `core/src/cli/renderer-selection.ts#resolveRendererMode` at THREE entry points so both CLI paths reject identically across the `--help` / `--version` short-circuit too: (a) `runCli` in `core/src/cli/run.ts` validates at the TOP, BEFORE `execute()` hands off to OCLIF — OCLIF's `--help` / `--version` short-circuit otherwise swallows errors thrown from the OCLIF init hook AND from `LandoCommandBase.runEffect` (the command class never instantiates), which means `--renderer=tui --help` would silently print help with exit 0 without this gate. The source-mode gate skips when argv head is a `bun` / `meta:bun` / `x` / `meta:x` passthrough; (b) `LandoCommandBase.runEffect` mutates `this.argv` in place BEFORE the early command guard / `this.parse(this.ctor)` so OCLIF's per-command flag parser never sees the unknown flag — accepted position is AFTER the command id in source mode; (c) `runCompiledCli` in `core/src/cli/run.ts` resolves at the top of the function BEFORE the `--help` / `--version` / per-command short-circuits, with the same `bun` / `meta:bun` / `x` / `meta:x` passthrough. `meta:bun` / `meta:x` Command subclasses override `run()` and never call `runEffect`, so they are automatically renderer-passthrough in source mode without an explicit exemption. `extractRendererFlag` honors the POSIX `--` argument terminator and stops consuming `--renderer=*` tokens after it (so `app:exec -- bash -c 'echo --renderer=plain'` forwards the child arg verbatim). Selection precedence is flag > `LANDO_RENDERER` env > `configValue` arg > default `"lando"`. The `configValue` arg is a helper seam for the global `renderer:` config layer; end-to-end wiring of `GlobalConfig.renderer` into source/compiled callers is deferred for now. The helper-level precedence is exercised by unit tests, callers currently pass only `argv` + `env`. Invalid values raise tagged `RendererSelectionError` (with `value` + `source: "flag" | "env" | "config"` + remediation) BEFORE command execution; the source error union goes through `Error(commandErrorMessage(...))` while the compiled and `runCli`-source paths print + `process.exitCode = 1`. Both `commandErrorMessage` copies (one per file) carry a matching `_tag === "RendererSelectionError"` block that does `details.unshift(tag)` to lead with the tag name like other tagged errors. CLI subprocess tests that re-spawn `bun run build` must prepend `dirname(process.execPath)` to `PATH` or the child shell fails with exit 127 on host distros lacking `/usr/local/bin/bun`.
