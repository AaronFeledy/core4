# AGENTS.md

Keep repo-specific quirks here. **This file is authoritative** for agents and contributors working in this repo — follow it unless a task explicitly overrides it. Update or remove a note when implementation changes make it stale. Document how the code behaves today and the correct engineering action for editing it — not roadmap, release scoping, or which future phase/story a capability lands in.

## Context

- Lando Core v4 is a Bun workspace for `@lando/core`, `@lando/sdk`, and bundled reference plugins under `plugins/*`.

## Commands

- Use Bun, not Node/npm/yarn/pnpm: `bun install`, `bun run typecheck`, `bun run lint`, `bun test`, `bun run build`.
- Treat `bun run typecheck` and `bun test` as the combined gate — root `tsc -b` does not typecheck `sdk/test/`.
- Run a focused test by path, e.g. `bun test core/test/unit/bootstrap.test.ts`.
- `bun run dev:guides` is the dev-time TDD driver for executable guides: it regenerates scenarios, typechecks the generated path, and re-runs only the affected guide's scenario tests on every change to `docs/guides/**/*.mdx`, `core/src/**`, `sdk/src/**`, `plugins/*/src/**`, or the scenario generator. Use `bun run dev:guides docs/guides/<path>.mdx` to constrain the loop to one guide, and `--once` for a single non-watching pass. It exits cleanly on Ctrl-C.
- Root `typecheck` is `tsc -b`; it may create `dist/` and `.tsbuildinfo`. `bun run clean` removes generated package outputs.
- For a single workspace package, follow the repo's Bun filter style, e.g. `bun run --filter='@lando/core' typecheck`.
- After adding a new `plugins/*` workspace package, run `bun install` so root `node_modules` gets the workspace symlink before testing imports from the repo root.

## Gotchas

- **RemoteSource/Dataset contract freeze:** Remote data sync is contract-only in Beta 1 and feature work in the 4.1 feature wave. Keep the split as `Dataset` x `RemoteSource` (N+M): `RemoteSource` owns egress over `HttpClient`/tool provisioning, `Dataset` owns local landing over `DataMover`, and the core pull/push orchestration composes them. Remote sync moves datasets (DB, files, config) and never syncs application code; "hosting" is category language only, not the contract name.
- **Hot path:** `core/src/cli/index.ts` and `core/src/cli/oclif/pre-renderer.ts` sit on the cold-start / first-byte path. No static imports of Effect, OCLIF, `@lando/sdk`, renderer code, or plugins — pre-renderer is bun builtins only; the index wrapper uses dynamic `import()` for everything else. Tests in `core/test/cli/fast-path-canary-preload.ts` and `paint-banner.test.ts` enforce this.
- **Compiled binary entry:** `bun build --compile` targets `core/bin/lando.ts`, not `index.ts`. Compiled code must not use `import.meta.url` for package metadata, install paths, or compiled-mode detection — bytecode builds can preserve the source `.ts` URL; use `core/src/version.ts` and `process.execPath`/the executable name instead. The compiled entry must also avoid top-level `await` because `bun build --compile --bytecode` rejects it.
- **`windows-x64` is the CI/release platform id; `win32-x64` is the runtime host key — they are different:** the per-platform CI/distribution ids live in `CI_PLATFORMS` in `scripts/ci-platforms.ts` (drives `scripts/build-ci-workflow.ts` jobs like `build-<id>`, `provider-integration-<id>`, the `static-checks-platform` matrix, artifact `lando-<id>`, the release compile stage, and the nightly distribution rehearsal compile list) and the nightly file-sync `JOBS` in `scripts/build-nightly-workflow.ts`; per spec (§02/§13/§15/ROADMAP) the windows entry MUST be `windows-x64` on runner `windows-2022`, compiling to `dist/lando-windows-x64.exe` in CI and `dist/lando-windows-x64.exe`/`dist/bundle/lando-windows-x64.exe` in release bundles. Do NOT confuse this with the *runtime* host key `win32-x64` (= `${process.platform}-${process.arch}` with `process.platform === "win32"`) used by `plugins/file-sync-mutagen/src/download.ts`, the `@lando/provider-lando` runtime bundle (`lando-runtime-win32-x64.zip`), and the frozen SDK `HostPlatform = "win32"` literal (schema-snapshot) — those stay `win32` forever. When you add/rename a CI platform id, update in lockstep: the shared platform module, the two generators + regenerated `.github/workflows/{ci,nightly}.yml`, `docs/ci-runbook.md` (build list, 5-platform matrix sentence, and the `## Branch protection` required-checks list that `core/test/build/branch-protection.test.ts` cross-checks against both the runbook and the generated job ids), and the build/codegen contract tests (`core/test/build/ci-workflow.test.ts`, `core/test/build/ci-runbook.test.ts`, `core/test/scripts/codegen-ci.test.ts`). The compiled binary's windows smoke runs `scripts/smoke-windows-binary.ts` (exit-code + fatal-UTF-8 stdout checks); the workflows force `defaults.run.shell: bash` so bash-isms work on the windows runner via Git Bash.
- **Dual CLI dispatch (permanent):** Source mode uses OCLIF (`core/src/cli/oclif/` via `execute()`). The compiled `$bunfs` binary bypasses OCLIF and routes through a hand-rolled dispatcher (`runCompiledCli` in `core/src/cli/run.ts`). This is **permanent**: the §14.2 "Compiled-binary CLI dispatch unification" decision closed as option (b) — `@oclif/core`'s `execute()` cannot dispatch inside a `bun build --compile` binary (`Config.load` → `findRoot` and the `module-loader` runtime `import()` both break inside `$bunfs`; see `spec/14-appendices.md` Appendix D.1). The parity rules + accepted divergences in `spec/08-cli-and-tooling.md` §8.4.1 are now normative. Keep error handling, flags, and routing in sync across both paths; the §13.1 dispatch parity test layer (`core/test/cli/parity/`) enforces it.
- **Single source of truth (normative):** Shared CLI behavior belongs in one module both paths import (e.g. `deferred-commands.ts`, `bug-report.ts`, `renderer-selection.ts`) — don't fork logic into OCLIF commands and `run.ts` separately. This is normative under spec §8.4.1's parity rules. When you add a canonical command id to `MVP_COMMAND_IDS`, add the matching `argv[0] === "<id>"` branch in `runCompiledCli` in the same change, or the §13.1 dispatch parity layer fails (both paths must dispatch implemented ids and defer the rest identically).
- **Compiled CLI test coverage (normative):** Most scenario tests exercise source mode (`core/bin/lando.ts`), not `core/dist/lando`. Spec §8.4.1 mandates the §13.1 layer-coverage rules treat both as the same surface even when the per-test target differs. The compiled-binary dispatch parity test layer (`core/test/cli/parity/dispatch-parity.test.ts`) is the cross-path contract: its structural half runs everywhere with no spawn, its behavioral half drives `core/dist/lando` on linux-x64.
- **Renderer boundary is enforced by the §13.4 lint gate:** Command output flows through the `Renderer` service. Both dispatch paths import one shared seam — `core/src/cli/renderer-boundary.ts` — which selects the mode→Live-Layer (`makeRendererServiceLiveForMode`), runs the command effect (`runWithRendererHandling`), and routes output: command *results* go to `Renderer.output.stdout`, failure diagnostics to `Renderer.output.stderr` (preserved stdout/stderr split), while `message.{info,warn,error}` keeps its per-mode destination (plain/lando/verbose→stdout, json→stderr). The `Renderer` tag has a raw `output: { stdout, stderr }` channel for already-formatted result/diagnostic text (additive, zero compat churn). The `renderer:` value is a real `GlobalConfig` field and precedence is `flag > env (LANDO_RENDERER) > config > default`, resolved by `resolveCliRendererMode` (config is read only when no flag/env, never on the level-`none` fast path). Per-command writes such as scratch foreground "started", exec stderr passthrough, and `meta:x` banner consume the renderer via `Effect.serviceOption` and NO-OP when absent so library callers still get typed results. The §13.4 gate (`scripts/check-renderer-boundary.ts`, wired into CI static checks via `bun run check:renderer-boundary`) fails on direct `console.*` or `process.std*.write` calls under `core/src/**` and `plugins/**`; keep the two §2.4 first-byte carve-outs (`core/bin/lando.ts`, `core/src/cli/oclif/pre-renderer.ts`) explicit.
- **OpenTUI interactive prompts are a dynamic-import-only driver behind the renderer plugin:** the recipe prompt runtime (`core/src/recipes/prompts/runtime.ts` `collectPrompts`) stays line-based by default and gains an optional `interactiveDriver?: PromptDriver` (`core/src/recipes/prompts/driver.ts`). When a driver is present AND `io.isTTY`, the interactive raw-read step delegates to `driver.readRaw(request)` (which returns a RAW string the runtime still coerces/validates — schemas unchanged); with no driver / non-TTY the existing path runs byte-for-byte. The OpenTUI implementation lives in `@lando/renderer-lando` (`src/opentui/prompt-driver.ts`) and is reached ONLY via dynamic import: `index.ts` exports a zero-heavy `loadInteractivePromptDriver()` → `await import("./opentui/prompt-driver.ts")`, and that file imports OpenTUI via a CONSTRUCTED specifier (`"@opentui/" + "core"`) with LOCAL structural types — NEVER a literal `from "@opentui/core"` or `import("@opentui/core")`. This is mandatory because core STATICALLY imports `@lando/renderer-lando` (`bundled.ts`, `bundled-renderers.ts`), so any static OpenTUI import would drag a native Zig/FFI lib onto the cold-start graph and into `bun build --compile` (where its native assets are absent from `$bunfs`). The cold-start canary `core/test/cli/opentui-coldstart-canary.test.ts` enforces the no-static-import rule. The CLI resolver `core/src/cli/prompts/interactive-driver.ts` gates the driver (TTY + not CI + not `--yes`/`--no-interactive`, escape hatch `LANDO_NO_OPENTUI_PROMPTS=1`) and adapts it: a plugin cancellation is signaled by a thrown `Error` whose `.name === "PromptCancelledError"` (it cannot import core's class) and is translated to core's `PromptCancelledError`; any other throw → `undefined`/fallback so a missing/broken renderer or the compiled binary degrades to the line-based reader. The driver declines `secret`/`multiselect` (throws non-cancel → line fallback) since `InputRenderable` has no mask. `secret` masking therefore stays on the proven silent line reader. Ad-hoc confirm/select surfaces (init tarball-checksum confirm, plugin trust, gated `meta:setup` provider select on `source === "default"`) share the driver via `core/src/cli/prompts/interactive.ts` `tryDriverConfirm`/`tryDriverSelect`, which keep the existing line output byte-for-byte when no driver is active. `@opentui/core` is a `@lando/renderer-lando` dependency only (per-platform native binaries are `optionalDependencies`, safe across platforms); never add it to `@lando/core`. The driver starts a live paint loop through an injectable `startRenderer` (default `renderer.start()`); a test driving `@opentui/core/testing`'s `createTestRenderer` MUST inject a no-op `startRenderer` and drive frames itself via `renderOnce()`/`mockInput` — calling `start()` on a test renderer spins an FPS loop that never settles and hangs the test indefinitely.
- **`globalServices:` collision detection is consumption-time, not bootstrap-time:** Cross-plugin `globalServices.id` collisions surface the tagged `GlobalServiceCollisionError` from `resolveGlobalServiceContributions` in `core/src/services/global-services.ts`, but only when contributions are *consumed* (`materializeGlobalServices` → `meta:global:install`), not at runtime bootstrap / plugin load as spec §18 (line 136) and §10 (line 263) require ("fail at plugin load"). Today only `@lando/proxy-traefik` (the `traefik` global service) declares `globalServices:`, so a cross-plugin collision cannot occur in shipped code. Do NOT force bootstrap-time validation by widening the frozen `PluginRegistry.list` error union (`PluginManifestError` only) — add a non-hot-path contribution-validation seam that fails with `GlobalServiceCollisionError` instead.
- **Bundled `globalServices:` resolve via the static map, not the `module:` path:** a bundled plugin that contributes `globalServices:` (e.g. `@lando/proxy-traefik`) must also export a `globalServices` map (`id` → `Effect<ServiceConfig>`) — the generated `core/src/plugins/bundled.ts` captures it and `bundledFirstGlobalServiceLoader` (`core/src/services/bundled-global-service-loader.ts`, used by `meta:global:install`) reads that map. The manifest `module:` field is documentation + the dynamic-`import()` fallback for *non-bundled* plugins only; raw `import(module)` cannot resolve a relative plugin path from `global-services.ts` nor work in a `bun build --compile` binary. A bundled plugin whose static map omits a declared id fails loudly with `GlobalAppError` (no silent dynamic fallback). To bundle a plugin that contributes `globalServices`/`proxies`, declare them under `BundledPluginEntry.contributes` in `core/build.config.ts` (the generator's `layerExportFor` maps `proxies` → the plugin's exported `proxy` layer) and regenerate `bundled.ts` + the schema snapshot.
- **Bundled `templateEngines:` follow the same static-map seam, plus a `template` Layer slot:** a bundled template-engine plugin (`@lando/template-handlebars`, `@lando/template-mustache`) exports `templateEngine = Layer.empty` (the layer slot — `layerExportFor` maps a `templateEngines` contribution to the `templateEngine` export) AND a static `templateEngines` map (`id` → `TemplateEngine` from `@lando/sdk/template`). The generated `bundled.ts` captures it via `templateEnginesFrom`; core aggregates all bundled `templateEngines` maps into `bundledTemplateEngineRegistry` (`core/src/landofile/template-render.ts`). The engine *contract* lives on the additive `@lando/sdk/template` subpath (type-only, like `@lando/sdk/expressions`) — its `TemplateCompileError`/`TemplateRenderError`/`TemplateEngineUnresolvedError` deliberately do NOT ride the frozen `@lando/sdk/errors` barrel; the parse seam maps them into `LandofileParseError`. Adding any bundled plugin (template or otherwise) must bump THREE count/list fixtures in lockstep: `EXPECTED_BUNDLED_PLUGIN_NAMES` in `core/test/services/plugin-registry.test.ts`, the `toHaveLength(N)` + `EXPECTED_BUNDLED_PLUGINS` list in `core/test/plugins/bundled.test.ts`, and the bundled-plugin manifest list in `sdk/test/fixtures/schema-snapshot.json` (regenerate via `bun run codegen:schema-snapshot`).
- **Landofile whole-file template rendering happens pre-parse via a column-0 `template:` directive:** a Landofile opts into an engine with a bare first-content-line `template: handlebars|mustache|none` directive (leading blank lines + `#` comments are skipped; a `# template:` comment is NOT a directive). `renderLandofileTemplate` (`core/src/landofile/template-render.ts`) detects it, blanks the directive line in the source (preserving line count so template syntax/render errors carry TEMPLATE-SOURCE line numbers and the `template` key never reaches strict `LandofileShape` decode), then compiles+renders BEFORE the `{{`-rejecting `scanContentForBetaExpressions` scanner. It is wired into BOTH `loadYamlLandofile` (`service.ts`, after `readFileContent`) and `lintLandofile` (`lint.ts`, before parse) so source-mode and `app:config:lint`/`doctor --app` stay in parity. `.lando.ts` Landofiles are never templated. Default (no directive / `none`) returns the raw text unchanged.
- **Scratch tier roots at `userCacheRoot`, not `userDataRoot`:** the `scratch` bootstrap level already existed in `BootstrapLevel` (rank 7, between `global` and `app` per spec §3.2) but used to alias the provider tier; it now has its own generated scratch bootstrap layer under `core/src/runtime/generated/layers/scratch.ts`. `ScratchAppService` (additive SDK tag) resolves the scratch base from `<userCacheRoot>/scratch/` via `resolveUserCacheRoot()` (`core/src/cache/paths.ts`, honors `LANDO_USER_CACHE_ROOT`/`XDG_CACHE_HOME`) — NOT `<userDataRoot>/global/` like `GlobalAppService`.
- **Scratch recipe mode plans the rendered file, not `discover`:** `ScratchAppService.acquire({ source: { kind: "recipe", ref } })` renders a bundled recipe into `<userCacheRoot>/scratch/<id>/root/` by reusing `initApp({ destination: <root>, runPostInit: false })` (the `destination`/`runPostInit` options were added to `InitAppOptions` for exactly this — `apps:init` keeps `<cwd>/<appName>` defaults). It then reads `<root>/.lando.yml`, decodes `LandofileShape` directly (`LandofileService.discover` EXCLUDES the scratch dir per spec §21.3, so discovery cannot resolve it — mirror `loadGlobalPlan`'s parse-the-dist-file approach), overrides `name: scratchId`, and re-plans under `withProcessCwd(<root>, …)` because `AppPlannerLive.plan` stamps `root = process.cwd()`. Recipe-resolution failures (`RecipeManifestNotFoundError`/`RecipeManifestValidationError`/`NotImplementedError`) map to `ScratchSourceUnresolvedError`; render/IO/parse failures map to `ScratchAppError`. Mirrors fork's materialize→plan→`provider.apply` shape (no scope finalizer / registry entry in this path). CLI `--answer`/`--option` (synonyms; `--option` wins on key collision via `[...answer, ...option]` merge), `--yes`, and `--no-interactive` thread through `ScratchAcquireInput` (additive, non-frozen) and MUST stay parsed identically in OCLIF `apps/scratch/start.ts` and `run.ts` `parseScratchStartArgv`.
- **Scratch fork forks by re-planning, and the scratch tier carries the planner:** `ScratchAppService.acquire({ kind: "fork" })` discovers the cwd Landofile and re-derives the plan under a fresh `scratch-<base>-<6hex>` identity by RE-PLANNING `{ ...landofile, name: scratchId }` — never by cloning the resolved `AppPlan`. Reason: `serviceType.toServicePlan({ appName, ... })` bakes the app name into each `ServicePlan` (env `LANDO_PROJECT`/`LANDO_APP_NAME`, derived volume names, DNS), so a clone would leak the source identity; re-planning makes every slug-keyed resource name the scratch id and leaves the source untouched. To support this, `core/src/runtime/generated/layers/scratch.ts` now provides `LandofileService` + `AppPlanner` (so the tier DOES do cwd Landofile discovery, unlike the global tier) and feeds `FileSystem`/`LandofileService`/`AppPlanner`/`RuntimeProviderRegistry` into the `ScratchAppServiceLive` sub-layer via one shared `providerBase` ref (Effect memoizes it). The deps live in the Live Layer closure, so `acquire`'s signature stays `Scope.Scope` + `ScratchSourceUnresolvedError | ScratchAppError` (no SDK widening, no `run.ts` compiled-cast change). Default fork plans (`--isolate=none`) keep `root = process.cwd()` (shared source mount); `--isolate=full` content-copies the source app root into `<userCacheRoot>/scratch/<id>/root/` (via `copyAppRoot` — Linux `cp -a --reflink=auto` with a `node:fs/promises` `cp` byte-copy fallback) and re-plans under `withProcessCwd(scratchPaths.root, …)` so the fork roots at the copy. The `isolate?: IsolateMode` (`"none" | "full"`) field on `ScratchAcquireInput` is additive (undefined ⇒ `none`); all three providers declare `copyOnWriteAppRoot: false`, so the copy is performed in core via `copyAppRoot`; `copyOnWriteAppRoot` is the seam for a provider to perform the copy instead.
- **Scratch registry + GC union:** scratch instances are recorded in `<userCacheRoot>/scratch/registry.bin` and writes are guarded by a token lock at `<userCacheRoot>/scratch/registry.lock` (release deletes only a matching token). Scratch provider labels come from the `plan.extensions["@lando/core/scratch"]` marker, not an id-prefix heuristic; providers emit `dev.lando.scratch` + `dev.lando.scratch-id` only when that marker id equals `plan.id`. `ScratchResourceScanner` is a core-private seam with a fail-closed default layer; the default layer performs no live docker/podman label queries, so GC tests inject a fake scanner. `apps:scratch:gc` cross-references `registry ∪ <userCacheRoot>/scratch/<id>/ dirs ∪ provider-label ids` and `--prune` reaps through the shared scratch cleanup path.
- **Scratch `--mount-cwd` / `--share-global-storage` are a post-plan transform, not new isolate modes:** `applyScratchStartFlags` in `core/src/scratch-app/service.ts` mutates the FINAL scratch `AppPlan` (in both `acquireFork`/`acquireRecipe`, right before `startScratchPlan`, with `hostCwd = process.cwd()` captured at the top of each gen BEFORE any `withProcessCwd` planning). `--mount-cwd[=<container-path>]`: with no value (or a value equal to the primary service's `appMount.target`) it REBINDS the primary `ServicePlan.appMount.source` to `$PWD` + forces `realization: "passthrough"` + drops that service's `app-mount` `fileSync` entry; with a different target it appends a `type:"bind"` passthrough mount. The default rebind is only observable in RECIPE mode (fork `--isolate=none` already roots `plan.root` at cwd, so source==cwd already). `--share-global-storage` requires `capabilities.sharedCrossAppNetwork` (else fail `ScratchAppError`), ensures `plan.networking.sharedNetworkMembership` (via `landoNetworkingPlan`), and stamps `extensions["@lando/core/scratch"].shareGlobalStorage = true` — so `markScratchPlan` now MERGES the scratch extension (`{ ...existing, id }`) instead of overwriting. `--mount-cwd` + `--isolate=full` fails fast with the additive `ScratchIsolationConflictError` (checked at the top of `acquire`). Did NOT extend `IsolateMode` (`none`/`full` only) nor relax the planner's `rejectGlobalScope`. This path implements network membership + the marker, not §21.8 storage-scope rewriting or live volume sharing.
- **Optional-value CLI flags need argv normalization for OCLIF:** oclif 4.x `Flags.string` REJECTS a bare flag ("expects a value"), so a `--flag[=value]` flag (e.g. `--mount-cwd`) must be declared `Flags.string` AND the command's `run()` must rewrite a bare `--flag` → `--flag=` (empty sentinel) in `this.argv` BEFORE `this.runEffect` (which calls `this.parse(this.ctor)`). `normalizeScratchStartArgv` in `core/src/cli/commands/scratch.ts` is the shared helper; the compiled `run.ts` `parseScratchStartArgv` hand-parses `--flag`/`--flag=value` directly (no oclif), so keep both in lockstep. Empty string from the bare form maps to "use the default" (`mountCwdFromValue` → `{}`).
- **Scratch scope-bound teardown lives in `acquire`, foreground blocking lives in `scratchStart`:** for a foreground (non-`detached`) acquire, `startScratchPlan` registers an `Effect.addFinalizer` against the AMBIENT `Scope.Scope` (the `Effect.scoped` the caller opens) AFTER the inner `Effect.scoped(provider.apply(...))` returns — placement matters, since registering inside the apply sub-scope would fire immediately. The finalizer is ONE best-effort step (`provider.destroy({app,plan},{volumes:true,removeState:true})` then `cleanupScratchInstance(instanceRoot)`, each `Effect.catchAllCause`→`logWarning` so it is total) and a `tapError` runs the same teardown on apply failure so a failed start does not leak the materialized dir. SECURITY: teardown removes `scratchPaths.instanceRoot` (always under `<userCacheRoot>/scratch/<id>/`), NEVER `plan.root` — an `--isolate=none` fork plans against the source cwd, so removing `plan.root` would delete the user's own app. `detached: true` registers no finalizer (the scratch survives). The CLI keeps the scope open in `scratchStart` (the single shared helper both dispatch paths import): foreground wraps acquire + `console.log("started: …")` + `waitForAbortSignal(signal)` in one `Effect.scoped`, so the scope closes (and the finalizer destroys) when the signal aborts. Use `Effect.async` (resolve on `'abort'`), NOT `Effect.never` + fiber interrupt — the OCLIF harness (`command-base.ts`) only `.abort()`s an `AbortController` and passes its `signal` into the command input; it does NOT interrupt the running fiber. Both paths must install the signal: OCLIF threads `input.signal` via `signalFromInput`, and `run.ts` `runAppsScratchStart` installs its own `process.once("SIGINT"/"SIGTERM")` → `AbortController` and removes the handlers in `finally`. Foreground prints "started" inside the effect and marks the result `rendered: true` so `renderScratchStartResult` returns `undefined` (no duplicate line from the post-run renderer).
- **Durable on-disk state goes through `StateStore` (§12.7), not hand-rolled atomic writes:** the canonical durable-persistence seam is the `StateStore` core service (`@lando/core/StateStore`, eager at level `minimal`) and its `StateBucket` handles — one file per bucket, with atomic write-temp-then-rename, schema decode, version header, corruption quarantine (`<path>.corrupt-<ts>`), and an optional `advisory` `O_CREAT|O_EXCL` token lockfile. Do NOT reinvent `writeFileAtomicViaRename` + lockfile + version-envelope + quarantine inline. The reference consumers are the scratch registry (`core/src/scratch-app/registry.ts` → `advisory`/`quarantine` bucket) and the include lockfile (`core/src/landofile/includes.ts` → `none`-lock bucket with a custom YAML codec that preserves the committed `.lando.lock.yml` format byte-for-byte). `StateStore` is host/test-overridable but is NOT a §4.2 plugin contribution surface (state-integrity invariant, like `EmbeddedAssetService`/`RedactionService`); plugins persist state only through the pre-namespaced `LandoPluginContext.stateStore` factory. Root resolution flows through the Paths primitive (`@lando/core/paths`), never re-derived `$HOME`/XDG/`%APPDATA%`. `CacheService` stays the separate in-memory/TTL hot-path memo; hot-path binary caches (`plan.bin`, `cwd-app-map.bin`) keep their own readers under the §12.5 budgets.
- **Lando-owned artifact downloads route through the core `Downloader` service, not bespoke `fetch`:** verified byte-acquisition (provider-lando runtime bundle, recipe/include `tarball` archives, `lando update` manifest/binary/checksum/signature) goes through `@lando/core/Downloader` (`core/src/downloader/service.ts`), which issues every byte through the core-private `HttpClient` chokepoint (`core/src/http-client/service.ts`). `Downloader` is **https-only** (`validateSource` rejects `http://`; `file://` only with `allowFileSource: true`) and returns no bytes in memory mode (`DownloadResult` has `path`, not `bytes`) — call sites that need bytes download to a file destination and `readFile` it. Network trust (proxy/CA) is NOT carried on `DownloadRequest`; it rides the core-private ambient `NetworkTrust` tag (`core/src/http-client/network-trust.ts`, the single home for `fetchInitForNetwork`/`shouldBypassProxy`) which `HttpClientBasicLive.stream` reads via `Effect.serviceOption` at call time and applies as Bun `fetch` `proxy`/`tls.ca`. A caller with resolved trust (e.g. `lando setup`'s preflight) wraps its download effect with `Effect.provideService(NetworkTrust, networkTrustFromResolved(network))`; `HttpClientBasicLive` construction stays inert (no CA/config reads at layer build). `RuntimeProviderRegistryLive` resolves `Downloader` and injects a `Downloader`-backed `artifactDownload` into the lando provider (the provider package never imports core internals — it exposes the `ArtifactDownload` seam). Two deliberate carve-outs remain on their own `fetch`/runner paths: `@lando/file-sync-mutagen`'s host-CLI/agent downloader (`plugins/file-sync-mutagen/src/download.ts`, still carries its own `fetchInitForNetwork` copy and tar/zip extractor) and plugin install/update (a `BunSelfRunner` package-manager operation). git clone, npm-registry packument fetch, and cosign/GPG verification are NOT downloads and stay on their existing seams. `DownloaderLive` publishes redacted `pre-download`/`download-progress`/`post-download` events through a `{redactText, publish}` seam in its Live closure (EventService via `Effect.serviceOption`, R stays `Scope`) — so EVERY construction site that wants events must `Layer.provide(EventServiceLive)` to `DownloaderLive` (the bootstrap generator + `recipes/tarball-source.ts` + `cli/commands/update.ts` do); a site that omits it silently no-ops events. Events carry scheme+host `urlOrigin` only and route `callerId`/`failureDetail` (a controlled tag-derived summary, never a raw URL/cause) through a per-call redactor seeded from `request.redactionTokens`. - **Guide-lint fixture rules key off frontmatter `id`, not the MDX path:** `scripts/lint-guides.ts` resolves a guide's fixtures at `docs/guides/<id>/fixtures/<name>` even when the MDX is nested (e.g. `docs/guides/services/mongodb.mdx` → id `mongodb` → `docs/guides/mongodb/fixtures/`). `lintGuideContent(sourcePath, content, options?)` stays pure — fixture rules read an injected `options.fixtures` inventory so they unit-test without disk; only `lintGuides(root)` walks the filesystem (and detects symlinks at the fixture root or any descendant). Add guide-content rules as pure `lint*` helpers; add disk/fixture rules behind the injected inventory. Component-prop checks reuse the SDK `decode*PropsEither` helpers so lint mirrors codegen with no new SDK exports.
- **`scripts/lint-guides.ts` is the §19.10 merge gate and now enforces more than per-file content rules:** content rules (raw shell fences inside `<Guide>`, `<Run>` display-vs-execute binding via `decodeRunPropsEither`, the `decode*PropsEither` component-prop sweep) are pure `lint*` helpers in `lintGuideContent`; the public-transcript discipline + generated source-map header checks live in `lintGuideTranscripts(sourcePath, content)` (exported), which `lintGuides` runs per file ONLY when `lintGuideContent` returned zero diagnostics. `lintGuideTranscripts` parses via `parseGuideScenarioAst`, builds each renderable `buildPublicTranscript` and asserts every frame's `sourceLine` is in an AST-derived allowlist (visible non-hidden step + non-`Variable`/`UseFixture` component lines + tab/scenario lines; `Cleanup` IS public), then renders each scenario via the now-exported `renderScenarioTest` and asserts the `// @generated` / `// @source:` / `// @scenario:` headers. Two gotchas: (1) lint's `expressionToValue` must stay byte-for-byte in sync with the generator's backtick template-literal handling (reject `${…}` interpolation) or library `<Run code={`…`}>` props crash lint; (2) `diataxis:` is NOT a `GuideFrontmatter` field — `lintGuideContent` strips it before decode but `parseGuideScenarioAst` rejects it, and zero real guides declare it, so lint test fixtures that exercise `parseGuideScenarioAst`/`lintGuideTranscripts` must omit `diataxis:`.
- **A `recipes/<id>/README.mdx` is consumed by two generators:** `scripts/build-guide-scenarios.ts` discovers it and emits a *gitignored* executable scenario test (it runs in `bun test`), and `scripts/build-recipe-readmes.ts` (§19.13 strip/flatten) emits the *committed* `recipes/<id>/.scaffold/<axis-value...>.md` (singleton → `default.md`). So a recipe README must be a valid `provider: test` executable guide — not just strip-able prose — and its `frontmatter.id` is the guide id (drives the generated test path) and must be unique across all `docs/guides/**` ids (no cross-file uniqueness lint exists; a collision silently overwrites generated tests). The `.scaffold/*.md` output is committed and guarded only by the codegen `git diff --exit-code` drift gate (biome does not lint Markdown), so keep `build-recipe-readmes` deterministic.
- **Guide coverage matrix (`docs/guides/INDEX.md`) is the union of PRD `## Guide Coverage` tables and is gated by `bun run check:guide-coverage`:** every user-facing Beta PRD declares a `## Guide Coverage` table (`scripts/check-guide-coverage.ts` parses it; PRDs that ship no guides declare `**None — internal/infra PRD.**`). The CI gate (`guide-scenarios-linux-x64`, between `lint:guides` and the scenario run) fails when a PRD-declared guide path has no `INDEX.md` row, when a non-`Planned` INDEX row points at a guide missing from disk, or when a `Status` is neither `Shipped` nor `Planned`. The gate also enforces the per-PRD section convention: PRDs are classified by their filename number — user-facing (`01–08`, `10`, `11`) MUST carry the `## Guide Coverage` section (else `coverage.missing-section`), internal/infra (`09`, `13`) MUST carry it too but declare `**None — …**`, and the index/meta PRDs (`00`, `12`) are exempt. So a new beta PRD that ships CLI surface must add the section or the gate fails. Consequence: adding/removing a guide row in a PRD's `## Guide Coverage` requires the matching `INDEX.md` edit, and when a guide file lands its INDEX row flips `Planned` → `Shipped` in the same PR. `Planned` rows are deliberately exempt from the on-disk check so the gate stays green for declared-but-unimplemented guides mid-flight. The script mirrors `lint-guides.ts`: pure parsers (`parseGuideCoverageSection` reports `{present, none, paths}`; `classifyPrd` maps a filename to its tier) + a `checkGuideCoverageOnDisk(root, opts)` disk entry (override `specDir`/`indexPath` for tests).
- **Guide-drift gate (`bun run check:guide-drift`, `scripts/check-guide-drift.ts`) keys off a SECOND block inside each user-facing PRD's `## Guide Coverage` section:** a `**CLI / source surface paths covered (drift gate input):**` bullet list of glob-like surface paths (e.g. `core/src/recipes/**`, `plugins/recipe-*/src/**`, exact files). `parseGuideCoverageSurfacePaths` reads only those bullets (it reuses `extractGuideCoverageSection` from `check-guide-coverage.ts` and stops at the first blank line after the bullets, so the guide table above it is never picked up). On a PR the gate fails (`drift.guide-not-touched`) when a changed file matches a PRD's surface globs (matched via `Bun.Glob`) but none of that PRD's owned guides changed; escape hatch is a PR body line `Guide-Coverage-Skip: <reason ≥ 24 chars>` (`drift.skip-reason-too-short` when shorter). It runs in CI ONLY on `pull_request` (after `check:guide-coverage`), reading the PR's changed files + body from `GUIDE_DRIFT_*` env (`GUIDE_DRIFT_CHANGED_FILES` override, else `GUIDE_DRIFT_BASE_SHA`+`GUIDE_DRIFT_HEAD_SHA`, else `GUIDE_DRIFT_BASE_REF` merge-base diff; `GUIDE_DRIFT_PR_BODY`) — so the `guide-scenarios-linux-x64` checkout uses `fetch-depth: 0`. No env ⇒ no-op pass (push/local). When you add a user-facing PRD that owns CLI surface, add BOTH the guide table AND the surface-paths bullet block, or the drift gate has nothing to enforce.
- **Generated guide scenario tests are not in any committed tsconfig:** the tests under `test/scenarios/generated/guides/**` are gitignored and `tsc -b` does not typecheck them (the root project references only `core`/`sdk`/`plugins`, and `core/test/tsconfig.json` is a standalone project that isn't built by `tsc -b`). To typecheck a generated guide path on its own, write an ephemeral tsconfig (e.g. to the OS tmpdir) that `extends` `tsconfig.base.json`, sets `typeRoots` to `<repoRoot>/node_modules/@types` (so `types: ["bun"]` resolves from outside the repo), maps the `@lando/core`/`@lando/sdk` `paths` to the workspace sources, and `include`s the generated dir — this is what `scripts/dev-guides.ts` does for its per-guide `tsc --noEmit` step.
- **Two guide transcripts, different lifecycles:** the INTERNAL transcript (`@lando/sdk` `Transcript`) is written at TEST-RUN time by `core/src/testing/scenario-context.ts` `persistTranscript` to `dist/transcripts/guides/<guideId>/<scenarioId>.json` with full execution data (stdout/stderr/exit). The PUBLIC transcript (`@lando/sdk` `PublicTranscript`, additive schema) is a CODEGEN artifact emitted at BUILD time by `scripts/build-guide-scenarios.ts` `emitPublicTranscripts`/`buildPublicTranscript` purely from the MDX AST, to `dist/transcripts/public/guides/<guideId>/<scenarioId>[.<variant>].json` (both gitignored). Public frames are reader-visible only: `render={false}` scenarios emit NO public transcript, and `<Hidden>` steps, `<UseFixture>`, and `<Variable>` components are excluded; `<Cleanup>` IS a public frame but the internal finalizer detail is not. Build-time emission is deliberate (the same MDX generates the gated asserting test, so the documented `resultSummary` stays coupled) — do not move it to runtime. Adding any `@lando/sdk` schema export still needs the four-place lockstep: `sdk/src/schema/json-schema.ts` registry + switch, `SDK_SCHEMA_NAMES` in `scripts/build-schema-snapshot.ts`, `sdk/API_COMPATIBILITY.md`, then `bun run codegen:schema-snapshot`.
- **Guide e2e smoke scenarios are generated but live-gated:** `defaultLayer: e2e` or `<Scenario layer="e2e">` emits a normal generated guide test using `ScenarioContextFactory.e2e`; the test self-skips unless `LANDO_GUIDE_E2E=1`, `LANDO_SCENARIO_E2E_BINARY`, and `LANDO_TEST_PODMAN_SOCKET` are present. The per-PR CI e2e subset selects generated test names containing `@smoke` and `[e2e]`, so keep smoke tags authored as `"@smoke"` and keep e2e guide scenarios cleanup-backed for provider teardown.
- **Per-PR guide scenarios run as a five-platform matrix, gated by an OS-family allow-list, with MDX-sourced CI annotations:** `scripts/build-ci-workflow.ts` emits one `guide-scenarios-<id>` job per `CI_PLATFORMS` entry (linux-x64, linux-arm64, darwin-x64, darwin-arm64, windows-x64); each regenerates guides + runs the scenario layer, but the e2e `@smoke` second pass (binary download + Podman) stays linux-x64-only (Beta non-goal) — only `guide-scenarios-linux-x64` keeps `needs: [static-checks, build-linux-x64]`, the others `needs: [static-checks]`. A WSL host satisfies a `linux` allow-list (it runs the same Linux build), so don't reintroduce blanket WSL skips or local `bun test` silently drops guide-scenario coverage. Adding/removing a platform job is a lockstep edit across the generator, regenerated `.github/workflows/ci.yml`, `docs/ci-runbook.md` (build/e2e prose + order-sensitive `## Branch protection` list), `core/test/build/{ci-workflow,branch-protection}.test.ts`, and `core/test/scripts/codegen-ci.test.ts` (the `bun-version-file`/`setup-bun` occurrence COUNT — each guide job adds one).
- **Public-transcript docs consumption + missing-artifact gate:** the docs-site consumption layer is `@lando/core/docs/render` (`core/src/docs/render/index.ts`) — it decodes the emitted `PublicTranscript` artifacts with the schema and renders deterministic, source-linked (`sourceFile#L<line>`) HTML scenario blocks. Keep it pure and dependency-free of `scripts/` so `@lando/core` stays importable (it re-implements the tiny variant→suffix mapping rather than importing the build script). - **`@lando/provider-lando` runtime-bundle download verifies a redirectable manifest — verification is never disabled:** `lando setup` resolves the bundle from a per-platform `{ url, sha256, filename, sizeBytes }` manifest in `plugins/provider-lando/runtime-bundle.ts`. The committed `plugins/provider-lando/runtime-bundle-versions.json` is a PLACEHOLDER (404 URLs + `0000…000N` SHAs) until §15 release-artifact generation replaces it, so real default-provider `lando setup` cannot complete end-to-end yet — that is a §15 gap, not a provider bug. To test/verify against a bundle built from the current commit, run `scripts/build-runtime-bundle.ts --local --staging <dir> [--platform <host-key>] --out <manifest>` (emits a `file://` manifest with a locally-computed SHA-256) and point `LANDO_RUNTIME_BUNDLE_MANIFEST` at it; `file://` URLs are allowed ONLY through the override path (env manifest or the paired flags), the bundled production manifest stays `https://`-pinned. `makeDefaultRuntimeBundleDownloader` reads `LANDO_RUNTIME_BUNDLE_MANIFEST` itself, so the env override works on the default (no-flag) path. The override REDIRECTS verification, it never disables it: bytes are always rejected unless `sha256(bytes)` matches the active entry, and there is no skip-verification flag. `--runtime-bundle-url` now REQUIRES a paired `--runtime-bundle-sha256` (a URL-only override fails with the paired error — the old half-override kept the pinned SHA and could never verify different bytes; `ProviderSetupOptions.runtimeBundleSha256` is the SDK seam). Adding a flag to the provider's `setup.flags` contribution (`plugins/provider-lando/src/index.ts` `manifest`) or the OCLIF `SetupCommand.flags` is a THREE-file lockstep: edit both, then `bun run scripts/build-oclif-manifest.ts` (regenerates `core/src/cli/oclif/compiled-manifest.ts`, consumed by the compiled `run.ts` dispatch) and `bun run codegen:schema-snapshot` (the §13.2 snapshot `sdk/test/fixtures/schema-snapshot.json` captures provider `setup.flags`). Host platform keys are the runtime `win32-x64` form (`${process.platform}-${process.arch}`), not the `windows-x64` CI id.
- **Telemetry transport lives in the minimal bootstrap layer, gated by `telemetryEnabled`:** the real `Telemetry` Live Layer is `makeTelemetryLayer(enabled)` in `core/src/telemetry/service.ts`, emitted into the generated `minimal.ts` by `scripts/build-bootstrap-layers.ts` (edit the generator + regenerate, never the generated file). When `enabled` is false it provides the `makeLibraryTelemetry` no-op stub (no buffer/worker/sink calls); when true it builds a scoped layer with a bounded dropping queue, a forked worker that drains into sinks, and an `Effect.interruptible`-wrapped finalizer flush. `Telemetry.record` is fire-and-forget: it offers to the queue and returns immediately, and all sink failures/timeouts are swallowed so transport errors never change the exit code. Sinks are injected through the core-internal `TelemetrySinks` `Context.Tag` (read via `Effect.serviceOption`, so absent ⇒ no sinks) — there is intentionally NO `PluginContribution.telemetrySinks` schema field yet, so a plugin-sink wiring story maps a manifest contribution into this tag without touching the frozen SDK surface. The `Telemetry` tag interface (`sdk/src/services/cli.ts`) is frozen; only the Layer implementation changed. Before enqueue, `record` runs `redactTelemetryData` (`core/src/telemetry/redaction.ts`): it allowlists a known event to its inventory fields (dropping everything else, which is what enforces "update telemetry constrained to version/target/channel/platform/outcome"), constrains enum values to the inventory allow set, and scrubs free-string values of paths/hosts/URLs/credentials/emails/UUIDs/tokens. Redaction therefore runs before buffering and before any sink, so a disabled/failing/plugin sink never sees a raw payload — add new event scrubbing here, not at call sites. `TELEMETRY_EVENTS` is `as const`, so iterate `spec.fields` as `ReadonlyArray<TelemetryFieldSpec>` before reading `field.allowedValues` or `tsc -b` fails. The redaction/retention policy (`docs/telemetry/retention.md`) is linked from the event doc, the inventory module comment, and the `lando config telemetry status|off` output (`policy` field).
- **Every recorded telemetry event must be in the inventory (gated):** the canonical machine-readable inventory is `core/src/telemetry/inventory.ts` (`TELEMETRY_EVENTS` + the `TELEMETRY_EVENT_FIELD_NAMES` tuples that `events.ts` re-exports as `TELEMETRY_EVENT_INVENTORY` — do NOT re-inline that literal). The published human doc is `docs/telemetry/events.md`. The `check:telemetry-inventory` gate (`scripts/check-telemetry-inventory.ts`, wired into CI `static-checks` and `bun run check:telemetry-inventory`) AST-scans `core/src`/`sdk/src`/`plugins` for `.record("<string-literal>", …)` calls on a `telemetry`-named receiver and fails when the event name is absent from `TELEMETRY_EVENTS`; the `inventory-doc.test.ts` consistency test fails when `events.md` drifts from the inventory. So adding a telemetry event is a three-place lockstep: `inventory.ts` + `docs/telemetry/events.md` + the actual `.record(...)` call, in one change. The transport's `sink.record(record.event, …)` dispatch (non-literal first arg) is intentionally not scanned, and no always-on runtime health event is recorded today.
- **Platform release signing is table-driven in Stage 9:** `scripts/release.ts` owns one platform-signing table for native binary signing. macOS signs with Developer ID (`LANDO_RELEASE_SIGNING_IDENTITY`) and notarizes in Stage 10 through `LANDO_RELEASE_APPLE_KEYCHAIN_PROFILE` only; do not add Apple ID/password argv forms. Do NOT satisfy Windows binary signing with `LANDO_RELEASE_COSIGN_KEY`/`COSIGN_PRIVATE_KEY`; those belong to manifest/signature work, not the Windows binary path. Any unavoidable secret argv flag must be listed in `SECRET_ARGV_FLAGS` so `redactReleaseCommand` scrubs failure messages; today only Windows `signtool /p` is allowed.
