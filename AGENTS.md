# AGENTS.md

Keep repo-specific quirks here. **This file is authoritative** for agents and contributors working in this repo — follow it unless a task explicitly overrides it. Update or remove a note when implementation changes make it stale. Items marked **(interim)** describe temporary behavior or known gaps; revise them when the underlying quirk is fixed (do not leave stale interim notes).

## Context

- Lando Core v4 is a Bun workspace for `@lando/core`, `@lando/sdk`, and bundled reference plugins under `plugins/*`.

## Commands

- Use Bun, not Node/npm/yarn/pnpm: `bun install`, `bun run typecheck`, `bun run lint`, `bun test`, `bun run build`.
- Treat `bun run typecheck` and `bun test` as the combined gate — root `tsc -b` does not typecheck `sdk/test/`.
- Run a focused test by path, e.g. `bun test core/test/unit/bootstrap.test.ts`.
- `bun run dev:guides` is the dev-time TDD driver for executable guides: it regenerates scenarios, typechecks the generated path, and re-runs only the affected guide's scenario tests on every change to `docs/guides/**/*.mdx`, `core/src/**`, `sdk/src/**`, `plugins/*/src/**`, or the scenario generator. Use `bun run dev:guides docs/guides/<path>.mdx` to constrain the loop to one guide, and `--once` for a single non-watching pass. It exits cleanly on Ctrl-C.
- Root `typecheck` is `tsc -b`; it may create `dist/` and `.tsbuildinfo`. `bun run clean` removes generated package outputs.
- For a single workspace package, follow the repo's Bun filter style, e.g. `bun run --filter='@lando/core' typecheck`.
- After adding a new `plugins/*` workspace package, run `bun install` so root `node_modules` gets the workspace symlink before testing imports from the repo root.

## Gotchas

- **Hot path:** `core/src/cli/index.ts` and `core/src/cli/oclif/pre-renderer.ts` sit on the cold-start / first-byte path. No static imports of Effect, OCLIF, `@lando/sdk`, renderer code, or plugins — pre-renderer is bun builtins only; the index wrapper uses dynamic `import()` for everything else. Tests in `core/test/cli/fast-path-canary-preload.ts` and `paint-banner.test.ts` enforce this.
- **Compiled binary entry:** `bun build --compile` targets `core/bin/lando.ts`, not `index.ts`. Compiled code must not use `import.meta.url` for package metadata or install paths — use `core/src/version.ts` and `process.execPath` instead.
- **Dual CLI dispatch (interim):** Source mode uses OCLIF (`core/src/cli/oclif/` via `execute()`). The compiled `$bunfs` binary bypasses OCLIF and routes through a hand-rolled dispatcher (`runCompiledCli` in `core/src/cli/run.ts`). Tracked as `spec/08-cli-and-tooling.md` §8.4.1 (parity rules + accepted divergences) and as a GA-blocking open decision in `spec/01-mission-and-tenets.md` §14.2 ("Compiled-binary CLI dispatch unification"); spike scheduled in `spec/ROADMAP.md` Phase 3 Beta. Remove or rewrite this bullet when the spike closes the §14.2 row. Until then, keep error handling, flags, and routing in sync across both paths.
- **Single source of truth (interim):** While dual dispatch exists, shared CLI behavior belongs in one module both paths import (e.g. `deferred-commands.ts`, `bug-report.ts`, `renderer-selection.ts`) — don't fork logic into OCLIF commands and `run.ts` separately. This is normative under spec §8.4.1's parity rules; drop the constraint if dispatch converges to a single router and §8.4.1 is folded into a historical note.
- **Compiled CLI test coverage (interim):** Most scenario tests exercise source mode (`core/bin/lando.ts`), not `core/dist/lando`. Spec §8.4.1 mandates the §13.1 layer-coverage rules treat both as the same surface even when the per-test target differs; update this note if compiled-binary parity becomes the default test harness.
- **Renderer not wired at CLI command boundary (interim):** Commands in `core/src/cli/run.ts` and per-command `render` helpers write to `process.stdout`/`process.stderr` directly via `console.log`/`console.error`. The `--renderer=lando|json|plain` flag parses through `core/src/cli/renderer-selection.ts` but its mode does not reach a `Renderer` Live Layer. The §2.4 "two carve-outs only" rule and the §13.4 lint gate are therefore not yet enforceable. Tracked as `spec/01-mission-and-tenets.md` §14.2 ("Renderer wiring at the CLI command boundary"); resolution scheduled in `spec/ROADMAP.md` Phase 3 Beta. Remove this bullet when that §14.2 row closes.
- **AOT bootstrap-layer codegen not yet shipped (interim):** Spec §2.4 forbids runtime `Layer.merge` / `Layer.provide` chains outside the §17.2 codegen output, but the codegen has not landed and `core/src/runtime/layer.ts` builds the runtime via runtime `Layer.mergeAll` chains. Scheduled in `spec/ROADMAP.md` Phase 3 Beta ("AOT bootstrap-layer codegen (§17.2)"). Remove this bullet when the Beta deliverable lands.
- **`globalServices:` collision detection is consumption-time, not bootstrap-time (interim):** Cross-plugin `globalServices.id` collisions surface the tagged `GlobalServiceCollisionError` from `resolveGlobalServiceContributions` in `core/src/services/global-services.ts`, but only when contributions are *consumed* (`materializeGlobalServices` → `meta:global:install`), not at runtime bootstrap / plugin load as spec §18 (line 136) and §10 (line 263) require ("fail at plugin load"). This is acceptable while discovery is bundled-only and only one bundled plugin (`@lando/proxy-traefik`, contributing the `traefik` global service) declares `globalServices:`, so a cross-plugin collision is still impossible to trigger in shipped code. Do NOT force bootstrap-time validation by widening the frozen `PluginRegistry.list` error union (`PluginManifestError` only) — add a non-hot-path contribution-validation seam that fails with `GlobalServiceCollisionError` when dynamic/non-bundled plugin loading lands (US-116). Remove this bullet when that seam exists.
- **Bundled `globalServices:` resolve via the static map, not the `module:` path:** a bundled plugin that contributes `globalServices:` (e.g. `@lando/proxy-traefik`) must also export a `globalServices` map (`id` → `Effect<ServiceConfig>`) — the generated `core/src/plugins/bundled.ts` captures it and `bundledFirstGlobalServiceLoader` (`core/src/services/bundled-global-service-loader.ts`, used by `meta:global:install`) reads that map. The manifest `module:` field is documentation + the dynamic-`import()` fallback for *non-bundled* plugins only; raw `import(module)` cannot resolve a relative plugin path from `global-services.ts` nor work in a `bun build --compile` binary. A bundled plugin whose static map omits a declared id fails loudly with `GlobalAppError` (no silent dynamic fallback). To bundle a plugin that contributes `globalServices`/`proxies`, declare them under `BundledPluginEntry.contributes` in `core/build.config.ts` (the generator's `layerExportFor` maps `proxies` → the plugin's exported `proxy` layer) and regenerate `bundled.ts` + the schema snapshot.
- **Scratch tier roots at `userCacheRoot`, not `userDataRoot`:** the `scratch` bootstrap level already existed in `BootstrapLevel` (rank 7, between `global` and `app` per spec §3.2) but used to alias the provider tier; it now has its own `makeScratchRuntimeLive` in `core/src/runtime/layer.ts`. `ScratchAppService` (additive SDK tag) resolves the scratch base from `<userCacheRoot>/scratch/` via `resolveUserCacheRoot()` (`core/src/cache/paths.ts`, honors `LANDO_USER_CACHE_ROOT`/`XDG_CACHE_HOME`) — NOT `<userDataRoot>/global/` like `GlobalAppService`.
- **Scratch recipe mode plans the rendered file, not `discover`:** `ScratchAppService.acquire({ source: { kind: "recipe", ref } })` renders a bundled recipe into `<userCacheRoot>/scratch/<id>/root/` by reusing `initApp({ destination: <root>, runPostInit: false })` (the `destination`/`runPostInit` options were added to `InitAppOptions` for exactly this — `apps:init` keeps `<cwd>/<appName>` defaults). It then reads `<root>/.lando.yml`, decodes `LandofileShape` directly (`LandofileService.discover` EXCLUDES the scratch dir per spec §21.3, so discovery cannot resolve it — mirror `loadGlobalPlan`'s parse-the-dist-file approach), overrides `name: scratchId`, and re-plans under `withProcessCwd(<root>, …)` because `AppPlannerLive.plan` stamps `root = process.cwd()`. Recipe-resolution failures (`RecipeManifestNotFoundError`/`RecipeManifestValidationError`/`NotImplementedError`) map to `ScratchSourceUnresolvedError`; render/IO/parse failures map to `ScratchAppError`. Mirrors fork's materialize→plan→`provider.apply` shape (no scope finalizer / registry yet). CLI `--answer`/`--option` (synonyms; `--option` wins on key collision via `[...answer, ...option]` merge), `--yes`, and `--no-interactive` thread through `ScratchAcquireInput` (additive, non-frozen) and MUST stay parsed identically in OCLIF `apps/scratch/start.ts` and `run.ts` `parseScratchStartArgv`.
- **Scratch fork forks by re-planning, and the scratch tier carries the planner:** `ScratchAppService.acquire({ kind: "fork" })` discovers the cwd Landofile and re-derives the plan under a fresh `scratch-<base>-<6hex>` identity by RE-PLANNING `{ ...landofile, name: scratchId }` — never by cloning the resolved `AppPlan`. Reason: `serviceType.toServicePlan({ appName, ... })` bakes the app name into each `ServicePlan` (env `LANDO_PROJECT`/`LANDO_APP_NAME`, derived volume names, DNS), so a clone would leak the source identity; re-planning makes every slug-keyed resource name the scratch id and leaves the source untouched. To support this, `makeScratchRuntimeLive` now provides `LandofileService` + `AppPlanner` (so the tier DOES do cwd Landofile discovery, unlike the global tier) and feeds `FileSystem`/`LandofileService`/`AppPlanner`/`RuntimeProviderRegistry` into the `ScratchAppServiceLive` sub-layer via one shared `providerBase` ref (Effect memoizes it). The deps live in the Live Layer closure, so `acquire`'s signature stays `Scope.Scope` + `ScratchSourceUnresolvedError | ScratchAppError` (no SDK widening, no `run.ts` compiled-cast change). Fork plans intentionally keep `root = process.cwd()` (shared source mount); the later `--isolate=full` story is the seam that swaps the app root to a content copy under `<userCacheRoot>/scratch/<id>/root/`.
- **Effect layer wiring:** `Effect.serviceOption(X)` resolves against what the current layer provides at build time — not against sibling layers in `Layer.mergeAll`. Dependencies must be `Layer.provide`'d on the sub-layer that needs them, or the service silently stays unavailable.
- **User-app resolution seam:** CLI commands MUST resolve the current project's Landofile via `loadUserLandofile(landofileService)` from `core/src/cli/app-resolution.ts`, never `landofileService.discover` directly. The helper rejects the reserved `global` app id with `AppIdReservedError` before any plan is built; a canary test (`core/test/cli/reserved-app-id-canary.test.ts`) fails if any `core/src/cli/commands/**` file calls `.discover`. The reserved-id check lives here (not on `LandofileService.discover`) because that tag's error union is frozen by the SDK backward-compat fixture — widen a non-frozen seam instead of mutating a frozen `Context.Tag` signature.
- **TaggedError auto-`message` in bug reports:** an Effect `TaggedError` with no `message` field still exposes a non-empty `.message` (a JSON dump of its fields). `core/src/cli/bug-report.ts` `extractMessage` therefore special-cases such tags by `_tag` *before* reading `record.message`; add a derive-case there when introducing a message-less error (e.g. `AppIdReservedError`) or the CLI renders raw JSON.
- **Global-app lifecycle commands plan from the dist file, not `discover`:** the `meta:global:*` lifecycle commands (`start`/`stop`/`status`/`destroy`/`config`/`uninstall`) resolve the global app via `loadGlobalPlan()` in `core/src/cli/commands/meta/global-plan.ts` — it reads `<userDataRoot>/global/.lando.dist.yml` directly (via `GlobalAppService.paths` + `FileSystem`), parses+validates to `LandofileShape`, then plans. `LandofileService.discover` is cwd-bound and loads only `.lando.yml`, so it cannot resolve the global stack. Because `AppPlannerLive.plan` is frozen + hard-bound to `process.cwd()`, `loadGlobalPlan` wraps `planner.plan(...)` in a `process.chdir(globalRoot)`/restore scope (`withProcessCwd`) so the plan roots at the global dir. These commands run on the new `bootstrap: "global"` tier = `makeGlobalRuntimeLive` (provider tier + `AppPlanner`, with `PluginRegistry`/`CacheService`/`ConfigService` provided to the `AppPlannerLive` sub-layer); `provider`/`scratch` stay on `makeProviderRuntimeLive`. The lifecycle commands intentionally do NOT publish lifecycle events (the `Global` event scope + `ensureRunning` are a later story); `meta:global:status` deliberately degrades each service to `"unknown"` when the provider is unavailable instead of failing, unlike per-app `info`.
- **Deferred-command parity probe hard-codes ids:** when promoting a deferred command to real (move its id from `DEFERRED_COMMAND_PLANS` in `core/src/cli/deferred-commands.ts` + the fixture `core/test/cli/fixtures/deferred-commands.json` into `MVP_COMMAND_IDS` in `core/src/cli/oclif/command-base.ts`, and replace its OCLIF stub), also re-point the hard-coded `parityProbes` entry in `core/test/cli/deferred-commands.test.ts` to a still-deferred id, or the compiled-`$bunfs` parity test throws on an undefined fixture lookup.
- **User-app `start` auto-starts the global app when `plan.requires.globalServices` is non-empty:** the planner (`core/src/services/planner.ts`) populates the optional `AppPlan.requires.globalServices` (currently `["traefik"]` for any routed app, since the v4 proxy is realized by the global `traefik` service) — this is the interim stand-in for the spec's `AppFeature.requires.globalServices` aggregation (the full AppFeature subsystem is unbuilt) and is the forward-compatible seam later AppFeature aggregation feeds. `startApp` calls `ensureGlobalServicesRunning` (`core/src/cli/commands/meta/ensure-global-services.ts`, cold-path only: `globalInstall` + `loadGlobalPlan` + publish `pre-global-start{triggeredBy:"ensure-running"}` → apply selected global services → `post-global-start`, all BEFORE the user app's `pre-app-start`) and wraps any failure in the chained `GlobalAutoStartError` (`cause` = e.g. `GlobalServiceMissingError`). `stop` never touches the global app (auto-stop is explicit-only). This widened `startApp`'s required-service union to include `GlobalAppService | FileSystem | PluginRegistry`, so **composing commands (`rebuild`/`restart`) and every test layer that drives `startApp`/`stopApp` must now provide those services even for apps with no global requirements** (the deps are structural in the `R` type; at runtime they're only touched when `requires.globalServices` is non-empty). Orchestration lives in the CLI helper, NOT on `GlobalAppServiceLive` (which sits below `AppPlanner` in the layer graph — putting `ensureRunning` there is a layer inversion).
- **SDK public surface:** Anything exported from `@lando/sdk` is compatibility-locked. When changing schema, services, events, or errors, follow `sdk/AGENTS.md` (update `sdk/API_COMPATIBILITY.md`, fixtures, and test doubles as needed).
- **Generated output:** Codegen scripts must finish with `biome check --write` on emitted files (not `format` alone). After generator changes, run the relevant `bun run codegen:*` and confirm `git diff --exit-code` on the generated paths.
- **Guide-lint fixture rules key off frontmatter `id`, not the MDX path:** `scripts/lint-guides.ts` resolves a guide's fixtures at `docs/guides/<id>/fixtures/<name>` even when the MDX is nested (e.g. `docs/guides/services/mongodb.mdx` → id `mongodb` → `docs/guides/mongodb/fixtures/`). `lintGuideContent(sourcePath, content, options?)` stays pure — fixture rules read an injected `options.fixtures` inventory so they unit-test without disk; only `lintGuides(root)` walks the filesystem (and detects symlinks at the fixture root or any descendant). Add guide-content rules as pure `lint*` helpers; add disk/fixture rules behind the injected inventory. Component-prop checks reuse the SDK `decode*PropsEither` helpers so lint mirrors codegen with no new SDK exports.
- **A `recipes/<id>/README.mdx` is consumed by two generators:** `scripts/build-guide-scenarios.ts` discovers it and emits a *gitignored* executable scenario test (it runs in `bun test`), and `scripts/build-recipe-readmes.ts` (§19.13 strip/flatten) emits the *committed* `recipes/<id>/.scaffold/<axis-value...>.md` (singleton → `default.md`). So a recipe README must be a valid `provider: test` executable guide — not just strip-able prose — and its `frontmatter.id` is the guide id (drives the generated test path) and must be unique across all `docs/guides/**` ids (no cross-file uniqueness lint exists; a collision silently overwrites generated tests). The `.scaffold/*.md` output is committed and guarded only by the codegen `git diff --exit-code` drift gate (biome does not lint Markdown), so keep `build-recipe-readmes` deterministic.
- **Guide coverage matrix (`docs/guides/INDEX.md`) is the union of PRD `## Guide Coverage` tables and is gated by `bun run check:guide-coverage`:** every user-facing Beta PRD declares a `## Guide Coverage` table (`scripts/check-guide-coverage.ts` parses it; PRDs that ship no guides declare `**None — internal/infra PRD.**`). The CI gate (`guide-scenarios-linux-x64`, between `lint:guides` and the scenario run) fails when a PRD-declared guide path has no `INDEX.md` row, when a non-`Planned` INDEX row points at a guide missing from disk, or when a `Status` is neither `Shipped` nor `Planned`. Consequence: adding/removing a guide row in a PRD's `## Guide Coverage` requires the matching `INDEX.md` edit, and when a guide file lands its INDEX row flips `Planned` → `Shipped` in the same PR. `Planned` rows are deliberately exempt from the on-disk check so the gate stays green for declared-but-unimplemented guides mid-flight. The script mirrors `lint-guides.ts`: pure parsers + a `checkGuideCoverageOnDisk(root, opts)` disk entry (override `specDir`/`indexPath` for tests).
- **Generated guide scenario tests are not in any committed tsconfig:** the tests under `test/scenarios/generated/guides/**` are gitignored and `tsc -b` does not typecheck them (the root project references only `core`/`sdk`/`plugins`, and `core/test/tsconfig.json` is a standalone project that isn't built by `tsc -b`). To typecheck a generated guide path on its own, write an ephemeral tsconfig (e.g. to the OS tmpdir) that `extends` `tsconfig.base.json`, sets `typeRoots` to `<repoRoot>/node_modules/@types` (so `types: ["bun"]` resolves from outside the repo), maps the `@lando/core`/`@lando/sdk` `paths` to the workspace sources, and `include`s the generated dir — this is what `scripts/dev-guides.ts` does for its per-guide `tsc --noEmit` step.
