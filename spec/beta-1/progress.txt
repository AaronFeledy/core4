## Codebase Patterns
- Shared SDK contract suites that prove seam usage should expose typed structural observation records (request URLs, DownloadRequest shapes, DataEndpoint tags, operation names, finalizer/probe metadata), not opaque counters; keep core-only implementation types inside `@lando/core/testing` and let `@lando/sdk/test` assert portable record shapes.
- A boundary lint gate (`scripts/check-<x>-boundary.ts`) follows one shape: exported pure `check<X>({ root? }) => Promise<{ ok, offenders }>`, recursive `readdir` over `SCANNED_ROOTS = ["core/src","plugins"]` collecting `.ts` non-`.test.ts`, a `CARVE_OUTS` Set (prefer EMPTY ŌĆö migrate offenders, do not carve), and an `import.meta.main` block that writes pass to stdout / failure list to stderr + `process.exitCode = 1`. Test it with a `mkdtemp` fake `root` proving exact offenders AND the legit-pass cases. Wire it by adding `"check:<x>-boundary"` to `package.json` AND a step in `scripts/build-ci-workflow.ts` `renderStaticChecks()` (after the other boundary gates), then `bun run codegen` and assert `git diff --exit-code .github/workflows/ci.yml` is only the new step; update `core/test/build/ci-workflow.test.ts` (step assertion), `docs/ci-runbook.md` + `core/test/build/ci-runbook.test.ts` (the runbook enumerates the gate list), and confirm `core/test/scripts/codegen-ci.test.ts` setup-bun count is UNCHANGED (a step in the existing job adds no new job). Authored gate/test files are NOT auto-biome-formatted ŌĆö run `bun run biome check --write <files>` before lint or the `biome check .` lint gate fails on import-order/formatting.
- The redaction-boundary gate (`scripts/check-redaction-boundary.ts`) flags via TypeScript AST: (1) `ts.isStringLiteral`/`ts.isNoSubstitutionTemplateLiteral` whose `.text` EXACTLY equals `[redacted]` or `[REDACTED]` (whole-string, not substring, not comments), and (2) `RegularExpressionLiteral` nodes whose source is an ad-hoc secret pattern via a CONSERVATIVE multi-signal heuristic (bearer/authorization; URL-userinfo `@`+`[^@`/`:[^`; signed-query `[?&]ŌĆ”(token|api_key|ŌĆ”)=`; or >=2 of password/secret/token/credential/bearer/apikey/api_key keywords) ŌĆö a single `token` mention in an ordinary URL regex is NOT flagged. The one redaction implementation is `@lando/sdk/secrets` `createRedactor(profile)`; the gate forced consolidation of the last hand-rolled redactors. Writing the gate surfaced a FOURTH ad-hoc redactor (`plugins/provider-podman/src/index.ts` inline `/^([^:]+:\/\/)[^@/]+@/u` masking to `<redacted>`) that exploration missed ŌĆö the gate is the source of truth, not a grep. Migrate, never carve out: each provider now has a `src/redact.ts` mirroring `provider-lando/src/redact.ts` (`createRedactor("secrets")` ŌåÆ `redactString`/`redactDetails`). NOTE the canonical `secrets` `urlUserinfo` class masks the WHOLE `user:pass` userinfo to `[redacted]@` (not just the password), so a migrated redactor changes `user:[REDACTED]@` ŌåÆ `[redacted]@`; update assertions to the real primitive output. "Audited services cannot weaken redaction" is proved at each service's REAL redaction point, not by fabricating a seam: ShellRunner/BunSelfRunner pull from `RedactionService.forProfile("secrets")`; Downloader composes `createRedactor("secrets", { values })` in its event seam (switched from value-only `createSecretRedactor` to gain pattern coverage); HostProxy (disabled) emits no secret-bearing field and its diagnostics route through `core/src/cli/redact.ts`; FileSync's SDK errors are redacted at the renderer/`redactDetails` boundary, not in the engine.
- The recipe YAML subset parser (`parseRecipeYaml`) does NOT support block scalars (`key: |` / `key: >`); recipe manifests and test fixtures must use inline string values (e.g. `default: seed line`) or strict decode fails. `collectPrompts` is plain-async (no Effect context), so any child-process need follows the `choicesRunner` precedent: an injectable runner with a `Bun.spawn`-based default plus a `spawner` seam for unit tests (see `editor-command.ts`/`choices-command.ts`). The `editor` prompt type intercepts at the top of `runInteractivePrompt` and returns `undefined` to fall through to text-semantics when no `$VISUAL`/`$EDITOR` is configured; recipe prompt-type deferral is gated by the (now empty) `REJECTED_PROMPT_TYPES` set in `manifest/service.ts`, mirroring `REJECTED_BUN_VERBS`.
- Effect-managed "handle" objects (an App handle, etc.) that own start-state resources should: capture the ambient `Scope.Scope` via `yield* Effect.scope`, fork a per-handle `handleScope = Scope.fork(parent, ExecutionStrategy.sequential)`, and track at most one live child scope in a `Ref<Scope.CloseableScope | undefined>` (effect-v3 type is `Scope.CloseableScope`, NOT `Scope.Closeable`). Serialize lifecycle mutations with `Effect.makeSemaphore(1)`, run every ref/scope swap under `Effect.uninterruptible`, and clean up a freshly-forked scope on failed/interrupted work via `Effect.onError`. Manually closing a child scope before its parent closes is a safe no-op (Effect scopes finalize exactly once), which yields "close exactly once" + "tear down on parent close" for free. A scope-acquired resource (`FileSyncEngine.createSession`) should own its own terminate finalizer; route teardown by providing the managed scope, never by adding a second explicit finalizer at the call site (double-terminate).
- Lando-owned verified downloads route through the core `Downloader` (`@lando/core/Downloader`), which issues bytes through the core-private `HttpClient` chokepoint and is HTTPS-ONLY with no memory-mode bytes (download to a file dest + `readFile`). Request-scoped proxy/CA trust rides the core-private ambient `NetworkTrust` tag (`core/src/http-client/network-trust.ts`), applied by `HttpClientBasicLive.stream` via `Effect.serviceOption` at call time (inert construction); a caller wraps its download effect with `Effect.provideService(NetworkTrust, ...)`. `fetchInitForNetwork`/`shouldBypassProxy` live ONLY there + the file-sync-mutagen exception; never re-add a bespoke per-site copy.
- `DownloaderLive` publishes `pre-download`/`download-progress`/`post-download` (schemas pre-exist in `sdk/src/events/download.ts`, already in the snapshot) via a `{redactText, publish}` seam baked into its Live closure ŌĆö `EventService` resolved with `Effect.serviceOption` (R stays `Scope`), so EVERY construction site that wants events must `Layer.provide(EventServiceLive)` (the bootstrap generator + recipe-tarball + self-update do). Progress = `Stream.tap`+`Ref` over the response body (keep `verified-stream` pure); `post-download` = `Effect.onExit` (interruption still brackets). Events carry scheme+host `urlOrigin` ONLY; `callerId`/`failureDetail` go through a per-call redactor from `request.redactionTokens`, and `failureDetail` is a controlled tag-derived summary, never a raw URL/cause. The `Downloader` contract suite is `runDownloaderContract(harness)` in `@lando/sdk/test`; the in-memory `TestDownloader` (`@lando/core/testing`) reuses the real `makeDownloaderService` over an in-memory `HttpClient` (the network seam), exactly like `TestManagedFileStore`. Egress fence (every byte via `HttpClient`, one stream per miss) is a byte-provenance+call-count invariant in the optional `harness.egress` section; proxy/CA/`NO_PROXY` assertions stay in a CORE-only canary (never put `NetworkTrust`/`fetchInitForNetwork` in `@lando/sdk/test`).
- `spec/beta-1/prd.json` has INCONSISTENT unicode escaping across entries (some literal `┬¦`/`ŌĆö`, some `\uXXXX`). NEVER round-trip it through `json.dump`/`JSON.stringify` to flip a story's `passes`/`notes` ŌĆö any `ensure_ascii` setting rewrites ~half the file and creates massive spurious churn. Edit the target story's `passes`/`notes` lines surgically with the editor (anchor on the unique `"priority": N,` line) and verify the diff is ~2 lines.
- Resolving a PRD priority/dependency inversion (a higher-priority story needs a tag/service a LATER story publishes): introduce a CORE-PRIVATE provisional `Context.Tag` carrying only the slice you need, keep it out of `@lando/sdk` and `core/src/services/index.ts`, give it the same string id the SDK tag will use, and let the later story promote it + swap the live layer. Wire it provided-only into the consumer's bootstrap layer (not as a standalone minimal service) and do NOT add it to `MinimalRuntimeServices`. Example: core-private `HttpClient` for `DownloaderLive` (US-286) ahead of the HttpClient SDK story.
- A `Layer.effect(Tag, gen yields Dep)` service whose published method signature has `R = Scope.Scope` (no `Dep`) closes over `Dep` at CONSTRUCTION; the method body then only needs `Scope`. Provide `Dep` via `.pipe(Layer.provide(DepLive))` at wiring so the public service R stays clean.
- Helper params that consume `Schema.optional(...)` request fields (which produce `T | undefined`) must type those params `?: T | undefined`, not `?: T` ŌĆö under `exactOptionalPropertyTypes` you cannot pass a `T | undefined` value into a `?: T` slot.
- `Bun.spawn` does NOT propagate JS-mutated `process.env` to the child (verified: setting `process.env.X` then spawning leaves `$X` empty in the child). A subprocess test that needs env isolation (e.g. pointing `LANDO_USER_DATA_ROOT` at a temp dir so `lando init`'s managed-file ledger does not pollute the real home) MUST pass it explicitly via `Bun.spawn({ env: { ...process.env, X: value } })`. In-process helpers can still mutate `process.env` directly.
- Widening a public `Schema.Literal` enum (e.g. adding `FileFormat` members) is additive/backward-compatible but breaks every exhaustive `switch` over it that has no `default` (the codec `encode`/`decode` switches) at typecheck time ŌĆö add the new cases in the same atomic change. Watch for DUPLICATED literals: `core/src/managed-file/service.ts` `LedgerEntrySchema.format` was a hand-copied `FileFormat` literal; reuse the imported SDK `FileFormat` schema value so the set stays in one place. Note it in `sdk/API_COMPATIBILITY.md` "Compatibility notes" (enum widening is not a new field), then `bun run codegen:schema-snapshot`; appending members at the END keeps the first value stable so public-schema contract fixtures do not churn.
- Publishing a new public registered SDK schema is a FIVE-place lockstep: `rawPublicSchemaRegistry` + `PUBLIC_SCHEMA_DESCRIPTIONS` + per-top-level-field `PUBLIC_FIELD_DESCRIPTION_EXEMPTIONS` in `sdk/src/schema/json-schema.ts`, plus a `PUBLIC_SCHEMA_CONTRACT_FIXTURES` entry in `sdk/test/schema/public-schema-contracts.ts` at the SAME ordinal position (its coverage test does an order-sensitive `toEqual(JSON_SCHEMA_NAMES)`). Every new RUNTIME export from `@lando/sdk/{schema,errors}` and every new `Context.Tag` in `services/index.ts` must also be listed in `sdk/API_COMPATIBILITY.md` (the backward-compat test does an exact `toEqual` on the additive lists; type-only interfaces are exempt). Then `bun run codegen:schema-snapshot` and `git add` the new `dist/schemas/*.json` + `docs/reference/schemas/*.mdx` (the drift gate asserts artifacts are git-tracked). A new SDK service tag follows the scratch pattern: real `Context.Tag` class in a per-service file + `export *` + an `export declare class` mirror in `services/index.ts`; pick a service-prefixed name for its options interface since plain `ApplyOptions` is already the provider's.
- To use a native/heavy library (e.g. `@opentui/core`) from a bundled plugin that core STATICALLY imports, hide it behind a constructed-specifier dynamic import (`"@opentui/" + "core"`) inside a sibling module the plugin entry only `await import()`s; the bundler cannot trace it, so `bun build --compile` stays green and the cold-start graph is untouched. Lock the rule with a grep canary and a fail-open loader so the compiled binary (no native assets in `$bunfs`) degrades to the non-native path.
- Default TTY renderer visual tests should drive `LandoTreePainter` directly for deterministic 80/100/120-column frames, then use tmux + visual-qa `tui-check` for the real terminal surface; keep plain/json/non-TTY regression assertions beside any TTY polish.
- Renderer frame capture has two surfaces: `LandoTreePainter.snapshot().frameLines` is the UNSTYLED logical frame; `painter.consume(event)` RETURNS the styled repaint bytes (last return over the full event list = final styled frame), fully synchronous/deterministic. The painter's redraw clear prefix is `cursorUp(n) + \r + eraseDown`; the bare `\r` is not an ANSI escape and survives `stripAnsi`, silently adding an invisible +1 column ŌĆö strip carriage returns when extracting a frame from `consume()`. The task-tree painter is NOT wide-char aware (measure by code units, pad `<= cols`); `formatSummary` IS (`console-layout.displayWidth`, pads `=== cols`). Assert widths on the ANSI-stripped raw frame, never a tokenized string. The committed visual-QA golden corpus + readable 4-class diff lives at `core/test/cli/renderer/visual-qa-harness.ts`; regenerate with `LANDO_UPDATE_VISUAL_QA=1`.
- The default `lando` renderer is plugin-owned: `@lando/renderer-lando` exports a `rendererFactories` map (id `lando`) + the `renderer = Layer.empty` slot; core injects reusable rendering primitives via `coreLandoRendererPrimitives` (typed by the type-only `@lando/sdk/renderer` subpath) and materializes the factory in `core/src/cli/renderer/bundled-renderers.ts` (`bundledRendererRegistry` + the `landoRenderer` convenience contribution). `renderer-boundary.ts` resolves `lando` from that registry; `plain`/`json`/`verbose` stay core. Painter/keybindings/formatters stay in core (shared with the verbose TTY path) ŌĆö only the lando assembly lives in the plugin. Tests needing the lando layer import `landoRenderer` and use `landoRenderer.makeEventConsumer(io)` / `landoRenderer.makeService(io)` (the core `makeLandoRenderer*` builders were removed). A new bundled renderer plugin follows the provider-registry pattern (core directly imports the workspace plugin package), not the codegen bundled-plugin table.
- Curl-pipe installer post-install setup is env-driven for tests/non-interactive use: `LANDO_INSTALL_RUN_SETUP=1` opts into `lando setup --yes`; `LANDO_INSTALL_SKIP_SETUP=1` or `LANDO_INSTALL_NONINTERACTIVE=1` skips without prompting. Both installers should print the canonical `lando shellenv` command plus snippet after verified install.
- ┬¦17.9 criteria 1-9 on linux-x64: `core/test/build/linux-acceptance-criteria-1-9.test.ts` is the runnable harness ŌĆö it spawns nested `bun test -t "<filter>"` spot checks against `release.test.ts`, `update-manifest.test.ts`, `ci-workflow.test.ts`, and (on linux-x64) `compile.test.ts`; parent tests need ~180s timeouts.
- ┬¦17.9 criteria 10-14 on linux-x64: `core/test/build/linux-acceptance-criteria-10-14.test.ts` nests `bun test ./<path> -t "<filter>"` spot checks; escape `+` and `(` in `-t` regex filters; use `./` path prefix for bun test file args.
- Linux release signing is manifest-layer, not Stage 9: Stage 11 writes binary-target SHA256SUMS/SHA512SUMS and GPG-signs/verifies them; Stage 12 keyless-cosign signs/verifies SHA256SUMS with complete GitHub OIDC request env; library-only releases keep manifest files but must not require linux binaries.
- External installed plugin contribution modules must be resolved against the plugin package root and normalized to `file://` specifiers during registry discovery; later loaders import the manifest value directly, so validating without rewriting leaves relative modules resolved from core instead of the plugin package.
- Windows release signing in Stage 9 is Authenticode first, then keyless cosign (`.sig` + `.crt`) using GitHub OIDC env; keyed cosign env belongs to manifest/signature stages, not the Windows binary signing gate.
- Release compile targets and generated CI platform ids share `CI_PLATFORMS` from `scripts/ci-platforms.ts`; do not duplicate the platform id/target list in release or workflow code.
- Release rehearsal keeps the canonical 13-stage table intact: use `--through-stage` for ordered prefixes and credential gates for `LOCAL_REHEARSAL=1` warning-skips, not stage removal or reordering.
- The release deprecation gate is woven into `runRelease` AFTER the `1-codegen` stage and OUTSIDE its try/catch (so its failure is not re-wrapped as a `1-codegen` error) ŌĆö it is NOT a `RELEASE_STAGES` entry and never routes through `runner.spawn/shell`, because `release.test.ts` asserts `RELEASE_STAGES` ids and the observed-stage order equal the exact 13-id list. It throws `ReleaseStageError("deprecation-gate", ŌĆ”)` on failure and runs in CI and local rehearsal identically (not credential-gated). The gate is an injectable `deprecationGate` option (default = `checkDeprecationReleaseGate({env})`); existing orchestration tests inject a passing stub so they stay fast and decoupled from real repo deprecation state.
- Compiled-`$bunfs` dispatch parity tests should be authored by first empirically diffing `bun core/bin/lando.ts <args>` vs `core/dist/lando <args>`, then asserting CROSS-PATH equality (`normalizeOutput`/`normalizeJsonEnvelope` from `core/test/cli/parity/normalize.ts`) rather than hardcoded strings. Two recurring gotchas: compiled `--help` prints only the description line + `USAGE` + `ALIASES` (no `ARGUMENTS`/`FLAGS` sections), so help parity must assert only the shared fragments; and compiled argv rejection is NOT automatic ŌĆö each hand-rolled `runMetaPlugin*`/command handler must replicate OCLIF's parse errors (`rejectInvalidInvocation`, missing-required-arg, pre-`--` unknown-flag) explicitly or the compiled path silently over-accepts what source rejects.
- Plugin-root containment is provable end-to-end with `core/test/cli/_util/fs-tree.ts` (`listTree`/`treeCreatedSince`/`pathsOutsidePrefixes`): spawn the command with isolated `LANDO_USER_*` roots, then assert created paths under `LANDO_USER_DATA_ROOT` stay within `plugins/` (or the data root is untouched). In-process variants set/restore `LANDO_USER_DATA_ROOT` with `Reflect.deleteProperty` (biome forbids the `delete` operator).
- Generated e2e guide scenarios stay host-safe by default: author them with `defaultLayer: e2e` or `<Scenario layer="e2e">`, tag the selected per-PR subset with `"@smoke"`, include a cleanup step, and let the generated test self-skip unless `LANDO_GUIDE_E2E=1`, `LANDO_SCENARIO_E2E_BINARY`, and `LANDO_TEST_PODMAN_SOCKET` are present.
- Schema snapshot/codegen drift gates must diff every generated schema output together: `sdk/test/fixtures/schema-snapshot.json`, `dist/schemas`, and `docs/reference/schemas`; otherwise generated docs can drift while CI stays green.
- Public SDK JSON Schema artifacts are generated from `JSON_SCHEMA_NAMES` via `scripts/build-schema-snapshot.ts`; keep the registry, snapshot scope, and `dist/schemas/<kebab-name>.json` artifact loop in lockstep.
- Deprecation runtime usage should emit through `EventService` from `DeprecationService.use` after recording the use and before any severity error; telemetry consumes that same event from a scoped subscriber, not a second deprecation path.
- Deprecation registry population belongs in the plugins bootstrap layer via `DeprecationPluginRegistryLive`; keep the minimal `DeprecationServiceLive` empty and O(1), and call `register` only from registration/bootstrap walks, never from use hot paths.
- Registry plugin postinstall trust keys on the requested package identity (`plugin:add` spec after version stripping), never package-controlled manifest names; authoring-root trust is only for local/link flows and must not apply to managed registry cache paths.
- Plugin install trust is a two-phase path: package extraction/manifest validation is inert and must not run lifecycle scripts; only a trusted result may run a package postinstall through `bunSelfInstall` / BunSelfRunner so pre/post bun-self-exec events and the recursion guard are preserved.
- Schema deprecation metadata should use the shared `SchemaDeprecationAnnotationId` annotation and AST unwrapping helpers so JSON Schema, generated reference content, and optional/union-wrapped fields stay in sync.
- Adding an optional field to a public JSON-schema-registered SDK struct (e.g. `PluginManifest`): the ┬¦13.2 annotation gate (`assertPublicSchemaAnnotations` in `sdk/src/schema/json-schema.ts`) requires either a detectable field description OR the `<Schema>.<field>` path in `PUBLIC_FIELD_DESCRIPTION_EXEMPTIONS`. A description on a `Schema.optional(...)`-wrapped inner schema is NOT detected ŌĆö add the exemption entry, then `bun run codegen:schema-snapshot`. `core/src/plugins/bundled.ts` carries plugin manifests by import, so manifest-field additions need no `codegen:bundled-plugins` diff.
- Library-mode executable guides: select with `<Run runtime="library">` carrying explicit `code` (executed inside the generated `Effect.gen` body with `context`/`Effect`/`expect`/`LandoCore`/`LandoTesting` in scope) and `displayCode` (reader-visible host TS). A scenario must be all-library or all-cli/shell; mixed runs are rejected at generation time. Per-frame runtime metadata is intentionally NOT on the PublicTranscript schema ŌĆö runtime is derived per scenario.
- Public-transcript docs consumption: render the generated `dist/transcripts/public/guides/<guideId>/<scenarioId>[.<variant>].json` artifacts through `@lando/core/docs/render` (`core/src/docs/render`), not by hand-parsing JSON ŌĆö it decodes with the `PublicTranscript` schema and produces deterministic, source-linked HTML. The emit side (`scripts/build-guide-scenarios.ts`) and the missing-artifact gate (`scripts/check-public-transcripts.ts`, run as `bun run check:public-transcripts` in the `guide-scenarios-linux-x64` CI job) MUST share the exported `publicTranscriptRelativePath` helper so artifact paths never drift; a guide counts as "requires a transcript" when it is a `Shipped` INDEX.md row with a renderable (`render !== false`) scenario.
- Effect scope finalizers run uninterruptibly, which masks `Effect.timeout` (the timer fires but cannot interrupt the timed-out effect). A bounded best-effort flush inside `Effect.addFinalizer` must wrap its body in `Effect.interruptible(...)` so the timeout can actually cancel a hanging effect; otherwise shutdown can hang forever on a stuck I/O effect.
- Telemetry redaction is enforced in the transport's `record` (`core/src/telemetry/service.ts`) via `redactTelemetryData` BEFORE `Queue.offer`, so no disabled/failing/plugin sink ever sees a raw payload. The layer both allowlists known events to their inventory fields (constraining update telemetry) and scrubs free-string values; reuse `scrubTelemetryValue`/`redactTelemetryData` from `core/src/telemetry/redaction.ts` for any new event rather than scrubbing ad hoc at the call site.
- Release supply-chain stages should generate artifact-specific SBOM manifest links first, then enforce that every manifest artifact has an SBOM; manifest artifact paths/SBOM links must stay POSIX-style `dist/...` regardless of host OS.
- Binary cosign verification commands should be generated from one release-stage helper and reused for both CI `cosign verify-blob` execution and `dist/release-notes.md`, so published instructions cannot drift from the command CI proved.
- POSIX self-update launch safety uses two probes: first the temp artifact before replacement, then the installed executable after replacement; the second probe is the one that can restore `.bak` and attach rollback-failure details to `UpdateLaunchProbeError` without changing the error tag.
- Windows self-update must never rename over the running `.exe` in-process; stage the verified binary beside the target, launch the delayed replacement helper with explicit backup/fallback paths, and let the current process exit.
- Installer trust roots split by platform: `install.sh` derives/verifies `SHA256SUMS.asc` beside the checksum manifest with its embedded GPG key, while `install.ps1` verifies the manifest's cosign `.sig`/`.crt` using an embedded identity/issuer trust-root object; the update manifest's `checksums.signature` stays `.sig` for self-update and Windows.
- Command RESULT rendering is mode-aware via a `RenderContext` ({mode, columns, isTTY}) built once in `runWithRendererHandling` (`core/src/cli/renderer-boundary.ts`) and passed as the 2nd arg to every `render` callback; `LandoCommandSpec.render` takes `(result, input?, ctx?)` and the compiled `runCompiledCommand`/`runScratchEffect` render params take `(value, ctx)`. To add decorated default-renderer output to a command, build a typed `SummaryDocument` (`core/src/cli/renderer/summary.ts`) and gate it with `isDecoratedContext(ctx)` (true only when `mode==="lando" && isTTY`); the non-decorated branch must reuse the existing render function byte-for-byte so `plain`/`json`/non-TTY output and dual-dispatch parity stay unchanged. Commands with their own `--format=json` keep the json branch ahead of the decoration gate.
- Decorated grouped summaries reuse CJK/wide-char-aware primitives in `core/src/cli/renderer/console-layout.ts` (`displayWidth`, `truncateToWidth`, `wrapToWidth`, `boxTop/boxBottom/boxSeparator/boxBody`, `toneChip`, `paintTone`) ŌĆö NOT the task-tree painter's internal `visibleLength`/`wrapFrameLines` (those are byte-pinned by first-paint tests and are not wide-char aware). Chips are PLAIN text; color is applied to the whole line via `paintTone` so truncation never clips an escape sequence and status is never color-only.
- Generalizing a frozen recipe-scoped SDK type into a published vocabulary (e.g. `RecipePrompt` ŌåÆ `PromptSpec`) without a runtime rewrite: (1) define the generalized schema/sub-schemas in a new `sdk/src/schema/<x>.ts`, then make the recipe schema re-export them as value+type aliases (`export const RecipePromptType = PromptType`) while RE-DECLARING the recipe struct itself to PRESERVE exact field order (reuse the generalized sub-schemas but keep the original key order so the ┬¦13.2 snapshot only shows additive enum widening, not a key reorder). (2) For "preserved as aliases" ERRORS where existing tests/consumers depend on the legacy `_tag` string: do NOT value-alias (would change `_tag`). Define the generalized errors as NEW `TaggedError`s with generalized tags, export the field-schema CONSTANTS, and have the legacy recipe errors spread those same constants (`{ ...interactionRequiredFields }`) while keeping their own `_tag`. Runtime that throws the legacy errors stays byte-identical; the real value-alias/migration is a later story. NEVER add `@deprecated` TSDoc to the legacy errors ŌĆö the `check:deprecations` gate then demands `markDeprecated()` runtime metadata (out of scope).
- Widening a `Schema.Literal` enum that a recipe field uses (e.g. adding `editor` to `PromptType`) is behavior-safe when the unsupported member is rejected by a PRE-DECODE scan (`scanPromptBeta` in `core/src/recipes/manifest/service.ts`), independent of the schema ŌĆö but the widened type still breaks any exhaustive `switch` over that field with no `default` (`coerceAnswer` in `core/src/recipes/prompts/runtime.ts`). Add the new `case` (fall through to the closest existing handler) for type-exhaustiveness; it is unreachable until the pre-decode rejection is removed in a later story. Append the new literal LAST so the public-schema snapshot/contract-fixture order stays stable.
- Registering a new public JSON schema is exactly: append to `rawPublicSchemaRegistry` + add `PUBLIC_SCHEMA_DESCRIPTIONS` entry + add every top-level field to `PUBLIC_FIELD_DESCRIPTION_EXEMPTIONS` (sorted Set) in `sdk/src/schema/json-schema.ts`, plus a `PUBLIC_SCHEMA_CONTRACT_FIXTURES` entry at the SAME ordinal position in `sdk/test/schema/public-schema-contracts.ts` (its coverage test does order-sensitive `toEqual(JSON_SCHEMA_NAMES)`), then `bun run codegen:schema-snapshot` and `git add` the new `dist/schemas/*.json` + `docs/reference/schemas/*.mdx`. There is NO separate `SDK_SCHEMA_NAMES` list ŌĆö it is the derived `JSON_SCHEMA_NAMES`. The ┬¦13.4 backward-compat test (`sdk-backward-compatibility.test.ts`) does sorted SET-equality of `Object.keys(@lando/sdk/schema) ŌłÆ frozen` vs the `## Additive Alpha schema exports` backtick names and of services/index.ts `Context.Tag` classes vs `## Additive Alpha service tags`; errors are smoke-checked only. So EVERY new runtime schema VALUE export and every new service tag must be added to `API_COMPATIBILITY.md`; a type-only interface (e.g. `PromptBatchOptions`) is not gated.
