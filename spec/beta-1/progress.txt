## Codebase Patterns
- Pinning the committed runtime-bundle manifest (`plugins/provider-lando/runtime-bundle-versions.json`) uses the AUTHORITATIVE GitHub release asset digests, never recomputed bytes: `gh release view runtime-v<version> --repo <owner/repo> --json assets --jq '.assets[] | "\(.name)|\(.size)|\(.digest)"'`. The manifest pins exactly the four PUBLISHED host targets (linux-x64, linux-arm64, darwin-arm64, win32-x64 — never darwin-x64), each `https://github.com/<owner/repo>/releases/download/runtime-v<version>/<filename>` with `sha256` (strip the `sha256:` prefix) and real `sizeBytes`. The production `RuntimeBundleEntrySchema` (https-only, 64-hex sha, int size) decodes the manifest at import via `RUNTIME_BUNDLE_MANIFEST`, but it does NOT reject all-zero placeholder shas or `sizeBytes: 0` — a "no placeholders" assertion must be an explicit test. The nightly `published-manifest-setup-linux-x64` job proves the pin end-to-end (compile binary, run `lando setup` with NO `LANDO_RUNTIME_BUNDLE_MANIFEST`, verify managed socket + runtime bin); it reuses `landoRootlessPrereqSteps`/`landoRuntimeSocketPath`/`landoManagedPodmanTeardownCommands` exported from `scripts/build-ci-workflow.ts` (exporting is output-neutral — ci.yml stays byte-identical) and bumps the nightly `bun-version-file` count in `core/test/scripts/codegen-ci.test.ts`.
- Runtime-bundle install and setup must agree on executable layout. Published runtime archives currently contain a top-level `bin/` directory; `installRuntimeBundle` strips that prefix so `runtimeBinDir/podman` and `runtimeBinDir/fuse-overlayfs` match `managedRuntimePodmanArgv0`/`buildManagedRuntimeServiceArgs`. Linux managed setup must use the installed managed Podman when `runtimeBinDir` is present; falling back to system `podman` hides bundle defects and fails the Podman 6 floor on sandboxes with old system Podman.

- Runtime bundles publish as immutable assets on this repo's own `runtime-v<version>` GitHub Releases: `scripts/build-runtime-bundle.ts` release mode derives `https://github.com/${GITHUB_REPOSITORY ?? "lando-community/core4"}/releases/download/runtime-v<version>/<filename>` when no `--base-url` override is given (`releaseBundleUrl`/`releaseBundleBaseUrl`/`resolveRuntimeBundleRepository`). Pinned upstream bundle contents live in `plugins/provider-lando/runtime-bundle-sources.json` (per-host-key components with `url`+`sha256`+`archive`(none|gz|tar.gz|zip)+`member`+`installName`+`mode`); `scripts/assemble-runtime-bundle.ts` downloads, sha256-verifies (fail-closed), extracts, and packs a byte-reproducible archive (`tar --sort=name --mtime=@0 --owner=0 --group=0 --numeric-owner | gzip -n`). To fill/verify pinned checksums use the authoritative GitHub asset digest — `gh release view <tag> --repo <owner/repo> --json assets --jq '.assets[].digest'` — never recompute or trust prose. GOTCHA: netavark/aardvark-dns v2.0.0 publish NO linux-arm64 binary (x86_64 `.gz` only) and passt ships only `.deb`, so those helpers are not yet pinnable on every platform; the publish target list intentionally omits `darwin-x64` (Podman 6 drops Intel Mac) while the shared `RUNTIME_BUNDLE_TARGETS` still carries it.
- To surface bundled-plugin manifest data (e.g. `setup.flags`) inside a CLI command, generate an `import type`-only literal-data module (see `core/src/cli/oclif/generated/setup-plugin-flags.ts` + `scripts/build-setup-plugin-flags.ts`). NEVER import `core/src/plugins/bundled.ts` from a command module: `run.ts` → `compiled-commands.ts` imports every command, so pulling `bundled.ts` there drags all bundled plugin Layers into the compiled CLI graph — a cold-start regression.
- New codegen scripts wire in three places: `package.json` (`codegen:<id>`), `scripts/codegen.ts` (catalog, ordered before any consumer generator), and a matching drift note. A flag/metadata generator that feeds `SetupCommand.flags` must run before `oclif-manifest` since that generator serializes the command's static flags.
- The `lando uninstall` command (`core/src/cli/commands/uninstall.ts`) is a lightweight `Effect.promise` with injectable seams (`remove`, `teardownRuntimeService`, `teardownProviderMachines`); provider-specific teardown that would otherwise import a plugin lives in a core `core/src/runtime/managed-*.ts` module (no plugin import) and is injected as a seam default. Test seams are threaded through `uninstallOptionsFromInput` in `meta/uninstall.ts` via `_`-prefixed input keys.
- Uninstall executor only runs steps where `destructive && status === "owned"` and then calls `remove(step.target)`. A step whose `target` is a logical NAME rather than a filesystem path (e.g. a provider machine) MUST be intercepted with its own branch + `continue` BEFORE `remove(step.target)`, or the name is `rm -rf`'d as a relative path.
- A CLI command's own `--format` flag is cosmetic (help/validation only): `LandoCommandBase` resolves the real result format centrally via `resolveResultFormat({argv})` (accepting `text|json|table|yaml|ndjson`), STRIPS `--format` from argv before `this.parse`, then OVERWRITES `flags.format` with the resolved value. JSON is emitted centrally through `spec.resultSchema` → `encodeCommandResult` in `runWithRendererHandling`, so a per-command `render()` must be human-only and may branch on the result shape (never build JSON). The compiled `run.ts` path mirrors this: `runCompiledCommand` reads `spec.resultSchema` from the active command id, so both paths JSON-encode identically once the OCLIF spec schema is a tagged union.
- To prove a `--no-network`/offline code path never touches the network (the "network-boundary test seam"), inject the network `deps` (e.g. `gitCloner`/`npmRegistryClient`) as a spy that records calls (or throws), then assert it was called zero times; the offline branch must resolve from the content-addressed cache (`<userCacheRoot>/includes/...` keyed by the lockfile's `resolved` ref) without ever invoking those seams.
- A variadic OCLIF positional (`[<source>...]`) uses `static strict = false`; read the positionals by filtering non-`-` tokens from `input.argv` in the command spec's `run`, and mirror that exact filter in the compiled `run.ts` argv parser so both dispatch paths stay in parity. Adding the positional/flag requires regenerating `compiled-manifest.ts`.
- Changing an OCLIF command's `static flags` requires regenerating `core/src/cli/oclif/compiled-manifest.ts` via `bun run codegen:oclif-manifest` (it serializes each command's static flag metadata); verify with `git diff` and run `core/test/cli/oclif-manifest.test.ts` + `core/test/cli/parity/dispatch-parity.test.ts`. `core/oclif.manifest.json` is gitignored; only `compiled-manifest.ts` is tracked.
- Wiring a service `Live` into a generated bootstrap layer is TWO edits: the generator (`scripts/build-bootstrap-layers.ts` + `bun run codegen:bootstrap-layers`) AND the hand-maintained runtime-service type union in `core/src/runtime/layer.ts` (`AppRuntimeServices` etc.). Miss the union and a command whose `R` now needs the service still typechecks in OCLIF but fails compiled `run.ts` dispatch (`runCompiledCommand` requires `R` ⊆ that union). Interactive inherited-stdio host shells launch through `ShellRunner.interactive` (`Bun.spawn` stdio "inherit"), never `child_process.spawn` in a command body. A new `BUILT_IN_COMMAND_DEPRECATIONS` notice must set `since` to a released/pending version within 12 months (use `4.2.0`) or the `check:deprecations` gate (part of `bun run lint`) demands a future major/minor `removeIn`.
- Graduating a `makeDeferredStubCommand` (`meta:*` stub) to a real command is SIX coordinated edits, all enforced by the §13.1 dual-dispatch parity test: (1) add the canonical id to `MVP_COMMAND_IDS` in `core/src/cli/oclif/command-base.ts`; (2) remove it from `DEFERRED_COMMAND_PLANS` in `core/src/cli/deferred-commands.ts`; (3) replace the stub file `core/src/cli/oclif/commands/meta/.../<cmd>.ts` with a real OCLIF wrapper (the `compiled-commands.ts` import is by path so it needs no edit); (4) add a `runMeta...` helper + an `argv[0] === "..."` branch in `core/src/cli/run.ts`; (5) `bun run codegen:oclif-manifest` and `bun run codegen:schema-snapshot`; (6) update the deferral test fixtures — `core/test/cli/fixtures/deferred-commands.json` AND the hard-coded probes in `core/test/cli/deferred-commands.test.ts` (`parityProbes`) and `core/test/cli/not-implemented.test.ts` — or those suites fail because the graduated id no longer emits `NotImplementedError`.
- To reuse an app-scoped command's core logic (`app:info`/`app:logs`) for a global-app command WITHOUT dragging `LandofileService` into the `global` bootstrap layer (which is `ProviderRuntimeServices | AppPlanner` — no `LandofileService`), extract a plan-based helper whose `R` is only `RuntimeProviderRegistry`: `infoForPlan(plan)` / `logsForPlan(plan, opts)` / `followLogsForPlan(plan, opts)`. The app wrapper keeps the `target?.plan ?? loadUserLandofile(...)` resolution (so it retains `LandofileService`); the global command loads its plan via `loadGlobalPlan()` and calls the plan-based helper directly. `meta:global:restart` reuses the existing `globalStop()` + `globalStart()` lifecycle (spec §18), NOT app `restartApp` — global start/stop publish `Pre/PostGlobalStart|Stop` events and use provider `apply`/`destroy`, which app restart does not.

- `CORE_VERSION` (`core/src/version.ts`) is a build-time stamped value: `bun build --define=__LANDO_CORE_VERSION__='"x.y.z"'` substitutes the `declare const __LANDO_CORE_VERSION__` token in compiled/bundled artifacts; source mode reads `corePackage.version`. CRITICAL: Bun's define flag is `--define=KEY=VALUE` (equals) — the esbuild `--define:KEY=VALUE` (colon) form is SILENTLY IGNORED. The value must be JSON-encoded (`JSON.stringify(version)`) so the RHS parses as a string literal. `scripts/release.ts` `compileCommand` stamps `releaseVersion(env)` (`LANDO_RELEASE_VERSION`) through this define — the single source of truth. The CLI `version` command and the compiled `run.ts` help banner both read `CORE_VERSION`; never re-hardcode `"0.0.0"`. A `typeof`-guarded free global (`typeof __LANDO_CORE_VERSION__ === "string" && ...`) is safe in source mode because `typeof` on an undeclared identifier yields `"undefined"` without throwing, and the `&&` short-circuits before the identifier is dereferenced.
