## Codebase Patterns
- Lando-owned verified downloads route through the core `Downloader` (`@lando/core/Downloader`), which issues bytes through the core-private `HttpClient` chokepoint and is HTTPS-ONLY with no memory-mode bytes (download to a file dest + `readFile`). Request-scoped proxy/CA trust rides the core-private ambient `NetworkTrust` tag (`core/src/http-client/network-trust.ts`), applied by `HttpClientBasicLive.stream` via `Effect.serviceOption` at call time (inert construction); a caller wraps its download effect with `Effect.provideService(NetworkTrust, ...)`. `fetchInitForNetwork`/`shouldBypassProxy` live ONLY there + the file-sync-mutagen exception; never re-add a bespoke per-site copy.
- `DownloaderLive` publishes `pre-download`/`download-progress`/`post-download` (schemas pre-exist in `sdk/src/events/download.ts`, already in the snapshot) via a `{redactText, publish}` seam baked into its Live closure — `EventService` resolved with `Effect.serviceOption` (R stays `Scope`), so EVERY construction site that wants events must `Layer.provide(EventServiceLive)` (the bootstrap generator + recipe-tarball + self-update do). Progress = `Stream.tap`+`Ref` over the response body (keep `verified-stream` pure); `post-download` = `Effect.onExit` (interruption still brackets). Events carry scheme+host `urlOrigin` ONLY; `callerId`/`failureDetail` go through a per-call redactor from `request.redactionTokens`, and `failureDetail` is a controlled tag-derived summary, never a raw URL/cause. The `Downloader` contract suite is `runDownloaderContract(harness)` in `@lando/sdk/test`; the in-memory `TestDownloader` (`@lando/core/testing`) reuses the real `makeDownloaderService` over an in-memory `HttpClient` (the network seam), exactly like `TestManagedFileStore`. Egress fence (every byte via `HttpClient`, one stream per miss) is a byte-provenance+call-count invariant in the optional `harness.egress` section; proxy/CA/`NO_PROXY` assertions stay in a CORE-only canary (never put `NetworkTrust`/`fetchInitForNetwork` in `@lando/sdk/test`).
- `spec/beta-1/prd.json` has INCONSISTENT unicode escaping across entries (some literal `§`/`—`, some `\uXXXX`). NEVER round-trip it through `json.dump`/`JSON.stringify` to flip a story's `passes`/`notes` — any `ensure_ascii` setting rewrites ~half the file and creates massive spurious churn. Edit the target story's `passes`/`notes` lines surgically with the editor (anchor on the unique `"priority": N,` line) and verify the diff is ~2 lines.
- Resolving a PRD priority/dependency inversion (a higher-priority story needs a tag/service a LATER story publishes): introduce a CORE-PRIVATE provisional `Context.Tag` carrying only the slice you need, keep it out of `@lando/sdk` and `core/src/services/index.ts`, give it the same string id the SDK tag will use, and let the later story promote it + swap the live layer. Wire it provided-only into the consumer's bootstrap layer (not as a standalone minimal service) and do NOT add it to `MinimalRuntimeServices`. Example: core-private `HttpClient` for `DownloaderLive` (US-286) ahead of the HttpClient SDK story.
- A `Layer.effect(Tag, gen yields Dep)` service whose published method signature has `R = Scope.Scope` (no `Dep`) closes over `Dep` at CONSTRUCTION; the method body then only needs `Scope`. Provide `Dep` via `.pipe(Layer.provide(DepLive))` at wiring so the public service R stays clean.
- Helper params that consume `Schema.optional(...)` request fields (which produce `T | undefined`) must type those params `?: T | undefined`, not `?: T` — under `exactOptionalPropertyTypes` you cannot pass a `T | undefined` value into a `?: T` slot.
- `Bun.spawn` does NOT propagate JS-mutated `process.env` to the child (verified: setting `process.env.X` then spawning leaves `$X` empty in the child). A subprocess test that needs env isolation (e.g. pointing `LANDO_USER_DATA_ROOT` at a temp dir so `lando init`'s managed-file ledger does not pollute the real home) MUST pass it explicitly via `Bun.spawn({ env: { ...process.env, X: value } })`. In-process helpers can still mutate `process.env` directly.
- Widening a public `Schema.Literal` enum (e.g. adding `FileFormat` members) is additive/backward-compatible but breaks every exhaustive `switch` over it that has no `default` (the codec `encode`/`decode` switches) at typecheck time — add the new cases in the same atomic change. Watch for DUPLICATED literals: `core/src/managed-file/service.ts` `LedgerEntrySchema.format` was a hand-copied `FileFormat` literal; reuse the imported SDK `FileFormat` schema value so the set stays in one place. Note it in `sdk/API_COMPATIBILITY.md` "Compatibility notes" (enum widening is not a new field), then `bun run codegen:schema-snapshot`; appending members at the END keeps the first value stable so public-schema contract fixtures do not churn.
- Publishing a new public registered SDK schema is a FIVE-place lockstep: `rawPublicSchemaRegistry` + `PUBLIC_SCHEMA_DESCRIPTIONS` + per-top-level-field `PUBLIC_FIELD_DESCRIPTION_EXEMPTIONS` in `sdk/src/schema/json-schema.ts`, plus a `PUBLIC_SCHEMA_CONTRACT_FIXTURES` entry in `sdk/test/schema/public-schema-contracts.ts` at the SAME ordinal position (its coverage test does an order-sensitive `toEqual(JSON_SCHEMA_NAMES)`). Every new RUNTIME export from `@lando/sdk/{schema,errors}` and every new `Context.Tag` in `services/index.ts` must also be listed in `sdk/API_COMPATIBILITY.md` (the backward-compat test does an exact `toEqual` on the additive lists; type-only interfaces are exempt). Then `bun run codegen:schema-snapshot` and `git add` the new `dist/schemas/*.json` + `docs/reference/schemas/*.mdx` (the drift gate asserts artifacts are git-tracked). A new SDK service tag follows the scratch pattern: real `Context.Tag` class in a per-service file + `export *` + an `export declare class` mirror in `services/index.ts`; pick a service-prefixed name for its options interface since plain `ApplyOptions` is already the provider's.
- To use a native/heavy library (e.g. `@opentui/core`) from a bundled plugin that core STATICALLY imports, hide it behind a constructed-specifier dynamic import (`"@opentui/" + "core"`) inside a sibling module the plugin entry only `await import()`s; the bundler cannot trace it, so `bun build --compile` stays green and the cold-start graph is untouched. Lock the rule with a grep canary and a fail-open loader so the compiled binary (no native assets in `$bunfs`) degrades to the non-native path.
- Default TTY renderer visual tests should drive `LandoTreePainter` directly for deterministic 80/100/120-column frames, then use tmux + visual-qa `tui-check` for the real terminal surface; keep plain/json/non-TTY regression assertions beside any TTY polish.
- Renderer frame capture has two surfaces: `LandoTreePainter.snapshot().frameLines` is the UNSTYLED logical frame; `painter.consume(event)` RETURNS the styled repaint bytes (last return over the full event list = final styled frame), fully synchronous/deterministic. The painter's redraw clear prefix is `cursorUp(n) + \r + eraseDown`; the bare `\r` is not an ANSI escape and survives `stripAnsi`, silently adding an invisible +1 column — strip carriage returns when extracting a frame from `consume()`. The task-tree painter is NOT wide-char aware (measure by code units, pad `<= cols`); `formatSummary` IS (`console-layout.displayWidth`, pads `=== cols`). Assert widths on the ANSI-stripped raw frame, never a tokenized string. The committed visual-QA golden corpus + readable 4-class diff lives at `core/test/cli/renderer/visual-qa-harness.ts`; regenerate with `LANDO_UPDATE_VISUAL_QA=1`.
- The default `lando` renderer is plugin-owned: `@lando/renderer-lando` exports a `rendererFactories` map (id `lando`) + the `renderer = Layer.empty` slot; core injects reusable rendering primitives via `coreLandoRendererPrimitives` (typed by the type-only `@lando/sdk/renderer` subpath) and materializes the factory in `core/src/cli/renderer/bundled-renderers.ts` (`bundledRendererRegistry` + the `landoRenderer` convenience contribution). `renderer-boundary.ts` resolves `lando` from that registry; `plain`/`json`/`verbose` stay core. A new bundled renderer plugin follows the provider-registry pattern (core directly imports the workspace plugin package), not the codegen bundled-plugin table.
