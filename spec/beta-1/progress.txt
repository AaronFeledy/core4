## Codebase Patterns
- Publishing a new public registered SDK schema is a FIVE-place lockstep: `rawPublicSchemaRegistry` + `PUBLIC_SCHEMA_DESCRIPTIONS` + per-top-level-field `PUBLIC_FIELD_DESCRIPTION_EXEMPTIONS` in `sdk/src/schema/json-schema.ts`, plus a `PUBLIC_SCHEMA_CONTRACT_FIXTURES` entry in `sdk/test/schema/public-schema-contracts.ts` at the SAME ordinal position (its coverage test does an order-sensitive `toEqual(JSON_SCHEMA_NAMES)`). Every new RUNTIME export from `@lando/sdk/{schema,errors}` and every new `Context.Tag` in `services/index.ts` must also be listed in `sdk/API_COMPATIBILITY.md` (the backward-compat test does an exact `toEqual` on the additive lists; type-only interfaces are exempt). Then `bun run codegen:schema-snapshot` and `git add` the new `dist/schemas/*.json` + `docs/reference/schemas/*.mdx` (the drift gate asserts artifacts are git-tracked). A new SDK service tag follows the scratch pattern: real `Context.Tag` class in a per-service file + `export *` + an `export declare class` mirror in `services/index.ts`; pick a service-prefixed name for its options interface since plain `ApplyOptions` is already the provider's.
- To use a native/heavy library (e.g. `@opentui/core`) from a bundled plugin that core STATICALLY imports, hide it behind a constructed-specifier dynamic import (`"@opentui/" + "core"`) inside a sibling module the plugin entry only `await import()`s; the bundler cannot trace it, so `bun build --compile` stays green and the cold-start graph is untouched. Lock the rule with a grep canary and a fail-open loader so the compiled binary (no native assets in `$bunfs`) degrades to the non-native path.
- Default TTY renderer visual tests should drive `LandoTreePainter` directly for deterministic 80/100/120-column frames, then use tmux + visual-qa `tui-check` for the real terminal surface; keep plain/json/non-TTY regression assertions beside any TTY polish.
- The default `lando` renderer is plugin-owned: `@lando/renderer-lando` exports a `rendererFactories` map (id `lando`) + the `renderer = Layer.empty` slot; core injects reusable rendering primitives via `coreLandoRendererPrimitives` (typed by the type-only `@lando/sdk/renderer` subpath) and materializes the factory in `core/src/cli/renderer/bundled-renderers.ts` (`bundledRendererRegistry` + the `landoRenderer` convenience contribution). `renderer-boundary.ts` resolves `lando` from that registry; `plain`/`json`/`verbose` stay core. Painter/keybindings/formatters stay in core (shared with the verbose TTY path) — only the lando assembly lives in the plugin. Tests needing the lando layer import `landoRenderer` and use `landoRenderer.makeEventConsumer(io)` / `landoRenderer.makeService(io)` (the core `makeLandoRenderer*` builders were removed). A new bundled renderer plugin follows the provider-registry pattern (core directly imports the workspace plugin package), not the codegen bundled-plugin table.
- Curl-pipe installer post-install setup is env-driven for tests/non-interactive use: `LANDO_INSTALL_RUN_SETUP=1` opts into `lando setup --yes`; `LANDO_INSTALL_SKIP_SETUP=1` or `LANDO_INSTALL_NONINTERACTIVE=1` skips without prompting. Both installers should print the canonical `lando shellenv` command plus snippet after verified install.
- §17.9 criteria 1-9 on linux-x64: `core/test/build/linux-acceptance-criteria-1-9.test.ts` is the runnable harness — it spawns nested `bun test -t "<filter>"` spot checks against `release.test.ts`, `update-manifest.test.ts`, `ci-workflow.test.ts`, and (on linux-x64) `compile.test.ts`; parent tests need ~180s timeouts.
- §17.9 criteria 10-14 on linux-x64: `core/test/build/linux-acceptance-criteria-10-14.test.ts` nests `bun test ./<path> -t "<filter>"` spot checks; escape `+` and `(` in `-t` regex filters; use `./` path prefix for bun test file args.
- Linux release signing is manifest-layer, not Stage 9: Stage 11 writes binary-target SHA256SUMS/SHA512SUMS and GPG-signs/verifies them; Stage 12 keyless-cosign signs/verifies SHA256SUMS with complete GitHub OIDC request env; library-only releases keep manifest files but must not require linux binaries.
- External installed plugin contribution modules must be resolved against the plugin package root and normalized to `file://` specifiers during registry discovery; later loaders import the manifest value directly, so validating without rewriting leaves relative modules resolved from core instead of the plugin package.
- Windows release signing in Stage 9 is Authenticode first, then keyless cosign (`.sig` + `.crt`) using GitHub OIDC env; keyed cosign env belongs to manifest/signature stages, not the Windows binary signing gate.
- Release compile targets and generated CI platform ids share `CI_PLATFORMS` from `scripts/ci-platforms.ts`; do not duplicate the platform id/target list in release or workflow code.
- Release rehearsal keeps the canonical 13-stage table intact: use `--through-stage` for ordered prefixes and credential gates for `LOCAL_REHEARSAL=1` warning-skips, not stage removal or reordering.
- The release deprecation gate is woven into `runRelease` AFTER the `1-codegen` stage and OUTSIDE its try/catch (so its failure is not re-wrapped as a `1-codegen` error) — it is NOT a `RELEASE_STAGES` entry and never routes through `runner.spawn/shell`, because `release.test.ts` asserts `RELEASE_STAGES` ids and the observed-stage order equal the exact 13-id list. It throws `ReleaseStageError("deprecation-gate", …)` on failure and runs in CI and local rehearsal identically (not credential-gated). The gate is an injectable `deprecationGate` option (default = `checkDeprecationReleaseGate({env})`); existing orchestration tests inject a passing stub so they stay fast and decoupled from real repo deprecation state.
- Compiled-`$bunfs` dispatch parity tests should be authored by first empirically diffing `bun core/bin/lando.ts <args>` vs `core/dist/lando <args>`, then asserting CROSS-PATH equality (`normalizeOutput`/`normalizeJsonEnvelope` from `core/test/cli/parity/normalize.ts`) rather than hardcoded strings. Two recurring gotchas: compiled `--help` prints only the description line + `USAGE` + `ALIASES` (no `ARGUMENTS`/`FLAGS` sections), so help parity must assert only the shared fragments; and compiled argv rejection is NOT automatic — each hand-rolled `runMetaPlugin*`/command handler must replicate OCLIF's parse errors (`rejectInvalidInvocation`, missing-required-arg, pre-`--` unknown-flag) explicitly or the compiled path silently over-accepts what source rejects.
- Plugin-root containment is provable end-to-end with `core/test/cli/_util/fs-tree.ts` (`listTree`/`treeCreatedSince`/`pathsOutsidePrefixes`): spawn the command with isolated `LANDO_USER_*` roots, then assert created paths under `LANDO_USER_DATA_ROOT` stay within `plugins/` (or the data root is untouched). In-process variants set/restore `LANDO_USER_DATA_ROOT` with `Reflect.deleteProperty` (biome forbids the `delete` operator).
- Generated e2e guide scenarios stay host-safe by default: author them with `defaultLayer: e2e` or `<Scenario layer="e2e">`, tag the selected per-PR subset with `"@smoke"`, include a cleanup step, and let the generated test self-skip unless `LANDO_GUIDE_E2E=1`, `LANDO_SCENARIO_E2E_BINARY`, and `LANDO_TEST_PODMAN_SOCKET` are present.
- Schema snapshot/codegen drift gates must diff every generated schema output together: `sdk/test/fixtures/schema-snapshot.json`, `dist/schemas`, and `docs/reference/schemas`; otherwise generated docs can drift while CI stays green.
- Public SDK JSON Schema artifacts are generated from `JSON_SCHEMA_NAMES` via `scripts/build-schema-snapshot.ts`; keep the registry, snapshot scope, and `dist/schemas/<kebab-name>.json` artifact loop in lockstep.
- Deprecation runtime usage should emit through `EventService` from `DeprecationService.use` after recording the use and before any severity error; telemetry consumes that same event from a scoped subscriber, not a second deprecation path.
- Deprecation registry population belongs in the plugins bootstrap layer via `DeprecationPluginRegistryLive`; keep the minimal `DeprecationServiceLive` empty and O(1), and call `register` only from registration/bootstrap walks, never from use hot paths.
- Registry plugin postinstall trust keys on the requested package identity (`plugin:add` spec after version stripping), never package-controlled manifest names; authoring-root trust is only for local/link flows and must not apply to managed registry cache paths.
- Plugin install trust is a two-phase path: package extraction/manifest validation is inert and must not run lifecycle scripts; only a trusted result may run a package postinstall through `bunSelfInstall` / BunSelfRunner so pre/post bun-self-exec events and the recursion guard are preserved.
- Schema deprecation metadata should use the shared `SchemaDeprecationAnnotationId` annotation and AST unwrapping helpers so JSON Schema, generated reference content, and optional/union-wrapped fields stay in sync.
- Adding an optional field to a public JSON-schema-registered SDK struct (e.g. `PluginManifest`): the §13.2 annotation gate (`assertPublicSchemaAnnotations` in `sdk/src/schema/json-schema.ts`) requires either a detectable field description OR the `<Schema>.<field>` path in `PUBLIC_FIELD_DESCRIPTION_EXEMPTIONS`. A description on a `Schema.optional(...)`-wrapped inner schema is NOT detected — add the exemption entry, then `bun run codegen:schema-snapshot`. `core/src/plugins/bundled.ts` carries plugin manifests by import, so manifest-field additions need no `codegen:bundled-plugins` diff.
- Library-mode executable guides: select with `<Run runtime="library">` carrying explicit `code` (executed inside the generated `Effect.gen` body with `context`/`Effect`/`expect`/`LandoCore`/`LandoTesting` in scope) and `displayCode` (reader-visible host TS). A scenario must be all-library or all-cli/shell; mixed runs are rejected at generation time. Per-frame runtime metadata is intentionally NOT on the PublicTranscript schema — runtime is derived per scenario.
- Public-transcript docs consumption: render the generated `dist/transcripts/public/guides/<guideId>/<scenarioId>[.<variant>].json` artifacts through `@lando/core/docs/render` (`core/src/docs/render`), not by hand-parsing JSON — it decodes with the `PublicTranscript` schema and produces deterministic, source-linked HTML. The emit side (`scripts/build-guide-scenarios.ts`) and the missing-artifact gate (`scripts/check-public-transcripts.ts`, run as `bun run check:public-transcripts` in the `guide-scenarios-linux-x64` CI job) MUST share the exported `publicTranscriptRelativePath` helper so artifact paths never drift; a guide counts as "requires a transcript" when it is a `Shipped` INDEX.md row with a renderable (`render !== false`) scenario.
- Effect scope finalizers run uninterruptibly, which masks `Effect.timeout` (the timer fires but cannot interrupt the timed-out effect). A bounded best-effort flush inside `Effect.addFinalizer` must wrap its body in `Effect.interruptible(...)` so the timeout can actually cancel a hanging effect; otherwise shutdown can hang forever on a stuck I/O effect.
- Telemetry redaction is enforced in the transport's `record` (`core/src/telemetry/service.ts`) via `redactTelemetryData` BEFORE `Queue.offer`, so no disabled/failing/plugin sink ever sees a raw payload. The layer both allowlists known events to their inventory fields (constraining update telemetry) and scrubs free-string values; reuse `scrubTelemetryValue`/`redactTelemetryData` from `core/src/telemetry/redaction.ts` for any new event rather than scrubbing ad hoc at the call site.
- Release supply-chain stages should generate artifact-specific SBOM manifest links first, then enforce that every manifest artifact has an SBOM; manifest artifact paths/SBOM links must stay POSIX-style `dist/...` regardless of host OS.
- Binary cosign verification commands should be generated from one release-stage helper and reused for both CI `cosign verify-blob` execution and `dist/release-notes.md`, so published instructions cannot drift from the command CI proved.
- POSIX self-update launch safety uses two probes: first the temp artifact before replacement, then the installed executable after replacement; the second probe is the one that can restore `.bak` and attach rollback-failure details to `UpdateLaunchProbeError` without changing the error tag.
- Windows self-update must never rename over the running `.exe` in-process; stage the verified binary beside the target, launch the delayed replacement helper with explicit backup/fallback paths, and let the current process exit.
- Installer trust roots split by platform: `install.sh` derives/verifies `SHA256SUMS.asc` beside the checksum manifest with its embedded GPG key, while `install.ps1` verifies the manifest's cosign `.sig`/`.crt` using an embedded identity/issuer trust-root object; the update manifest's `checksums.signature` stays `.sig` for self-update and Windows.
- Command RESULT rendering is mode-aware via a `RenderContext` ({mode, columns, isTTY}) built once in `runWithRendererHandling` (`core/src/cli/renderer-boundary.ts`) and passed as the 2nd arg to every `render` callback; `LandoCommandSpec.render` takes `(result, input?, ctx?)` and the compiled `runCompiledCommand`/`runScratchEffect` render params take `(value, ctx)`. To add decorated default-renderer output to a command, build a typed `SummaryDocument` (`core/src/cli/renderer/summary.ts`) and gate it with `isDecoratedContext(ctx)` (true only when `mode==="lando" && isTTY`); the non-decorated branch must reuse the existing render function byte-for-byte so `plain`/`json`/non-TTY output and dual-dispatch parity stay unchanged. Commands with their own `--format=json` keep the json branch ahead of the decoration gate.
- Decorated grouped summaries reuse CJK/wide-char-aware primitives in `core/src/cli/renderer/console-layout.ts` (`displayWidth`, `truncateToWidth`, `wrapToWidth`, `boxTop/boxBottom/boxSeparator/boxBody`, `toneChip`, `paintTone`) — NOT the task-tree painter's internal `visibleLength`/`wrapFrameLines` (those are byte-pinned by first-paint tests and are not wide-char aware). Chips are PLAIN text; color is applied to the whole line via `paintTone` so truncation never clips an escape sequence and status is never color-only.
