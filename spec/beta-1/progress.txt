## Conventions

- This log is a durable progress narrative, not a changed-file ledger. Git records diffs; this file records outcomes, decisive verification, blockers, and durable learnings.
- Story entries should explain what became true, what materially proved it, and what remains blocked or deferred. Do not add review choreography, PR mechanics, generated-path inventories, exact test-count trivia, or formatting/comment-only changes.
- If a story relaxes a shipped contract, record the decision and the synchronized spec/PRD/snapshot updates. Otherwise keep the focus on Beta 1 closure state.

## Durable learnings

- Business logic stays in pure Effect. Filesystem, process, network, terminal, provider, and renderer effects belong behind services or seams; command bodies should compose services and return structured results.
- Public contracts come from Effect Schema with tagged errors and generated schema snapshots. Additive schema exports need field annotations, compatibility bookkeeping where required, and snapshot/docs refresh; machine error envelopes only serialize `_tag`, `message`, and `remediation`, so machine-required context must appear in the message.
- Generated-file discipline is non-negotiable. Change the generator, run the matching codegen, and include intentional emitted outputs; do not hand-edit generated workflows, manifests, bootstrap layers, compiled OCLIF manifest, plugin tables, or schema artifacts.
- Source and compiled CLI paths must stay behaviorally paired. OCLIF flags, compiled `run.ts` parsing, command result schemas, streaming modes, and parity fixtures move together whenever command routing changes.
- Renderer and redaction boundaries own output. Commands and provider seams should not write directly to stdout/stderr; user-visible text, StreamFrames, events, doctor contexts, and provider error details must pass through the shared renderer/redactor seams.
- Durable host state and platform paths use shared services. Versioned/locked state belongs in `StateStore`; Lando roots and derived paths come from `makeLandoPaths`/`PathsService`; host/provider probing uses `@lando/sdk/probe` rather than ad hoc retry loops.
- Runtime bundles for managed Linux setup must contain a local-engine Podman. A checksum-valid manifest can still pin an operationally wrong asset; the managed service requires `podman --root ... --runroot ... system service`, which a `podman-remote-static-*` client cannot provide.
- Runtime-bundle verification must exercise network create/remove and a published-port container through the managed Podman root/runroot/config shape. Version checks and a ready API socket do not prove that bundled networking helpers are selected or complete.
- Runtime-bundle host keys and release artifact ids are different domains. Runtime bundles publish only `linux-x64`, `linux-arm64`, `darwin-arm64`, and `win32-x64`; CLI/release artifacts can still include Intel Mac or differently named Windows targets.
- Log-follow lifecycle semantics live in the provider-neutral SDK engine. It owns handle close via explicit close plus `Stream.ensuring`, byte-level UTF-8-safe framing, tail/since/copytruncate/rotation diagnostics, and planner-owned capability degradation. Do not reintroduce scoped file handles that close before streams run.
- Log-source availability is a planning truth, not a mid-stream surprise. Required follow sources fail planning when `serviceLogSources` is unavailable; optional follow sources are recorded as unavailable; redirect sources ride console and are never followed.
- Truth hierarchy is spec > PRD > progress. Progress can summarize and explain, but it must not preserve stale unreleased behavior over the spec or mark pending stories complete.

## Foundation summary — US-372..US-400 (2026-07-02 through 2026-07-06)

The first Beta 1 wave converted many Alpha 4 compatibility gaps into enforceable contracts. The durable foundation was less about isolated features and more about making the repo harder to drift: schema snapshots, machine envelopes, boundary gates, generated metadata, and dispatch parity became the standard closure proof for CLI/runtime work.

Material outcomes:

- Provider setup and runtime readiness moved toward typed verdicts instead of stringly failures.
- CLI commands increasingly returned result schemas and structured envelopes rather than bespoke text/JSON.
- Deferred command graduation became a known multi-surface operation, not just adding an OCLIF file.
- Config write surfaces converged on explicit verb rejection, canonical serialization, and validation before write.
- Include/offline paths gained cache-boundary proofs so `--no-network` cannot quietly hit network seams.
- Doctor reporting became an aggregation of typed sections with NDJSON/text parity and redacted context.
- Public schema additions were paired with annotations, fixtures, compatibility records, and snapshot drift checks.
- Boundary gates became part of the ordinary acceptance path, not late cleanup.

Durability and probing work landed across provider setup, managed runtime readiness, health/readiness checks, app lifecycle, cache behavior, and doctor reporting. Repeated review fixes moved host/provider retry and timeout behavior onto typed seams and probe-style classifiers, with redaction at the final event/error boundary rather than inside pure parsers.

Managed-file and renderer discipline hardened during this wave. Runtime-owned files gained ownership rules and drift checks; CLI output moved toward central result encoding and renderer-owned frames; direct writes and ad hoc JSON became review findings rather than acceptable shortcuts.

Source/compiled CLI parity became a recurring acceptance criterion. Graduated commands had to update OCLIF wrappers, compiled dispatch, generated manifests, machine-output schemas, and parity fixtures together. This prevented source-only fixes from silently leaving the compiled `$bunfs` path behind.

Configuration and command surfaces were made more agent-drivable. App/global config write paths were hardened for typo rejection, canonical serialization, validation errors with remediation, structured output, and safer cache/offline behavior. Config translator and include-update surfaces moved toward deterministic preview/write semantics and network-boundary proofs.

MCP and agent-environment groundwork landed. MCP became a projection of command specs with allowlist generation, structured result envelopes, cancellation/input handling, redacted diagnostics, and doctor visibility. Agent-context env forwarding started with fixed host markers, then gained config policy, opt-outs, audit reporting, and precedence fixes.

Important constraints that survived this wave:

- A command-level `--format` flag is declarative; central command-base machinery owns the actual output mode.
- Global/bootstrap commands must not import app-only services just to reuse app logic; plan-based helpers keep layers small.
- Config command seams can host validation that would otherwise widen unrelated service error unions.
- Generated allowlists should be literal-data modules with freshness tests, not hand-maintained parallel registries.
- Source-mode `CORE_VERSION` is not a release version; compiled artifacts are stamped separately.

Version-constraint work became a cache and doctor concern, not just a parse check. `lando:` constraints were enforced during app loading, represented in cache freshness, honored by skip mode without mutating stale caches, and surfaced through doctor output.

Material verification for this foundation included root typecheck/lint, schema-snapshot drift checks, generated-manifest/codegen idempotency, renderer/redaction/machine-output/state/probe/path boundary gates, focused unit and contract suites, parity dispatch suites, guide lint/coverage where guides owned the surface, and manual CLI smoke on source and compiled paths where meaningful.

## Agent-native milestone — US-401..US-409 (2026-07-06 through 2026-07-07)

Configurable agent env forwarding became a first-class host policy. `agentEnv` gained global config, exact-name allow/deny, a Landofile opt-out, `LANDO_AGENT_ENV=0`, and `info --deep` audit output. The important implementation constraint is that policy resolves fresh per invocation; it is not cached into app plans because config and per-invocation switches must take effect immediately.

The env-forwarding policy preserved precedence: forwarded host markers sit below service-declared environment and explicit exec/task env. The audit surface reports names only, never values.

`app open` and the logical container-to-host `runLando` round trip established the host-proxy command pattern. `open` resolves URLs from the plan, opens through a ShellRunner-backed platform helper, degrades safely on headless Linux, and publishes redacted lifecycle events. The in-container path proved allowlist, cwd remap, env filtering, retained-runtime dispatch, event redaction, envelope parity, and exit-code parity.

Host opening remained plan-only. Targets are resolved from app plan routes/endpoints; container-provided URLs are not trusted as authority.

The production host-proxy transport remained incomplete. The physical Unix socket, compiled shim binary, container mount, token auth, concurrency cap, and recursion guard were explicitly deferred; US-403 proved the logical contract only.

Version constraints and hot caches were made operational. Landofile `lando:` constraints enforce against `CORE_VERSION`, surface tagged errors/remediation, support skip mode, and participate in app-plan/tooling cache freshness. Doctor reports learned to explain constraint status.

Cache freshness includes the sources that can change command/version semantics: Landofile bytes, lockfile provenance, local include content, and normalized `lando:` constraint state.

Recipe/toolbox/scratch work provided a disposable tooling path. The `toolbox` recipe, raw `type: lando` service, recipe list/describe/validate commands, `apps:scratch:run`, streaming JSON frames, env forwarding into scratch exec, cleanup/keep semantics, and reserved alias rules landed with both source and compiled dispatch coverage.

Scratch execution treats a non-zero tool exit as a successful command result carrying that exit code. Ownership cleanup remains scope-bound unless `--keep` successfully hands the scratch to the registry.

The real buildKey cache remained incomplete. Scratch warm-repeat verification proved fresh scratch acquisition, no warm pool, and stable content-addressed image-build inputs; it did not implement or prove a runtime build-results cache hit because that remained spec-only in Beta 1.

Verification centered on typecheck/lint, schema/codegen idempotency, boundary gates, MCP/host-proxy allowlist freshness, command parity, scratch lifecycle tests, guide scenarios for owned docs, and real Podman sandbox smoke where the managed runtime was available.

## Runtime-bundle milestone — US-410..US-412 (2026-07-07)

Runtime-bundle publishing and manifest pinning were implemented. The publish workflow assembles immutable `runtime-v<version>` assets for the four runtime host keys, uses pinned upstream source digests, packs reproducible archives, guards release immutability, and regenerates the committed manifest pin. The committed manifest was repinned to real `runtime-v0.1.0` release assets with HTTPS URLs, authoritative SHA-256 digests, real byte sizes, and no `darwin-x64` runtime entry.

What is complete:

- Release-mode URL derivation points at this repository's `runtime-v*` assets by default.
- Assembly verifies upstream bytes fail-closed before packaging.
- Archive packing is deterministic, so byte drift is reviewable.
- Manifest resolution still supports explicit override for local testing.
- The committed manifest schema accepts the real pins and rejects placeholder classes through explicit gates.

Manifest invariants became release and CI gates. Offline checks reject placeholder-like checksums, zero sizes, off-repo URLs, missing required host keys, and runtime-version mismatches. Live URL/length verification was scheduled for nightly/release contexts rather than every PR.

Subsequent operational validation exposed a separate Linux managed-setup gap. The published Linux asset is checksum-valid and installs, and the bundled binary reports Podman 6.0.0, but it is a remote-only client: `podman --root` rejects with `unknown flag: --root`, and it cannot launch `system service` with the managed `--root`/`--runroot` arguments. A zero-override `lando setup` for provider-lando on Linux therefore does not pass.

This is not a local manifest bug and not a URL swap within the same upstream Podman release. The asset must be republished with a local-engine-capable Linux Podman and the committed manifest must be repinned. US-430 tracks that corrected publish-and-pin loop without reopening US-410 or US-411.

Verification proved checksum/manifests/gates, deterministic assembly, schema-valid pins, override precedence, extraction layout, and published URL byte/sha matching. It did not prove managed zero-override Linux setup because the published asset cannot satisfy the local-engine requirement.

## Podman 6 milestone — US-413..US-424 (2026-07-07 through 2026-07-08)

Podman 6 version gates landed before provider action. Setup rejects old or unparsable Podman versions with typed provider-unavailable errors and remediation, using the selected CLI/API source as appropriate. The sandbox caveat is important: the default apt Podman in `lando4-dev` was 5.4.2 and correctly failed the floor; Homebrew Podman 6 on PATH was needed for live provider=podman verification.

Provider=podman setup did complete when Homebrew Podman 6 was selected. Provider-lando managed setup did not, because it depends on the runtime bundle asset discussed above.

CI provisioning moved Linux Podman-backed jobs to Homebrew Podman 6 with helper binaries and a fail-fast host-contract assertion. This replaced the runner's older distro Podman path for provider integration and guide e2e contexts.

The CI assertion compares semantic version tuples rather than packed integers and fails closed when the host binary is absent or unparsable.

The Podman HTTP API prefix migrated to `/v6.0.0`, and a static gate prevents new production provider code from constructing Podman 5 libpod prefixes. Version-floor enforcement remains separate from API-prefix correctness.

Managed-machine and host-target seams were updated for Podman 6. Machine start avoids hijacking the user's default connection, OS upgrade uses the Podman 6 command shape, WSL unsupported paths fail with typed remediation, Intel Mac is gated as unsupported for runtime bundles, and managed machine CA import/Windows Hyper-V prep remained advisory or platform-specific.

The Intel Mac decision applies to runtime bundles, not to every release artifact. Runtime support and release packaging intentionally retain separate target vocabularies.

Several Podman 6 seams landed as typed future-implementation units rather than live wiring: image-pull progress parsing/events, OOMKilled died-event doctor surfacing, libpod volume-prune selection/parsing, health status inspection/waiting, and managed `containers.conf` loopback port binding. These seams were unit/contract verified with pure parsers, fake transports, redaction boundaries, and generated schema updates where surfaced.

Future-implementation seams followed a consistent shape:

- Pure parser/classifier first, with no redaction inside pure parsing.
- Boundary builder second, where identifiers and provider payloads are redacted.
- Fake transport or captured spawn tests for exact provider/API behavior.
- No live provider action unless the story explicitly required it and the environment could support it.

Live integration was environment-limited. Provider=podman setup against Homebrew Podman 6 succeeded where documented, and compiled smoke commands passed, but default provider-lando managed setup remained blocked by the remote-only Linux runtime bundle. macOS/Windows machine lifecycle behavior could not be live-exercised from the Linux sandbox.

## Service-log contract milestone — US-425..US-427 (2026-07-08 through 2026-07-09)

The declared-log-source SDK contract landed first. `LogSource`, `LogSourceInput`, `LogSourceId`, log stream/strategy literals, reserved `console` semantics, `LogChunk.source`, `LogOptions.sources/source`, and required `ProviderCapabilities.serviceLogSources` became public or provider contract surface as appropriate.

The SDK additions were intentionally layered before runtime behavior. US-425 changed the contract and schema surface; US-426/US-427 made that contract visible to planning and redirect reification; US-428/US-429 made follow/runtime/CLI behavior observable.

Review tightened contract truthfulness: provider capability keys derive from schema fields, lossy Landofile encoding is rejected, public JSON schema registration was added, and compatibility/export tests cover the new surface.

Plan-time merge and cache semantics followed. User and service-type log sources merge by id, `extends` preserves inherited declarations with child precedence, invalid redirect on unsupported bases is rejected, and the app-plan cache/fingerprint includes resolved source declarations so service catalog changes cannot reuse stale plans.

Absolute path/id hardening beyond the existing schema primitives was explicitly deferred to the follow/redirect wire-up rather than overloading the initial SDK-contract story.

Redirect reification shipped as a build-plan foundation. `redirect` sources on Lando-built services emit deterministic symlink build steps to stdout/stderr, while runtime follow selection filters to `follow` sources only. A review fix bumped the cache schema so warm caches without redirect build steps do not survive the new semantics.

At this point the contract, capability bit, planner merge, cache invalidation, and redirect-to-console path existed. Real provider-owned file following and CLI/source filtering were still the next stories.

## US-428 — provider-owned follow log sources (2026-07-09)

US-428 shipped the provider-neutral `follow` log-source engine and provider integration surface. The engine implements finite snapshots, follow backfill, appended writes, missing-file pending/unavailable diagnostics, logrotate rename/create, copytruncate, byte-level UTF-8-safe framing, CRLF cleanup, final partial-line flushing at EOF, per-source tail, bounded line truncation, and `since` handling only for timestamped sources.

The engine emits diagnostics as `LogFollowEvent`s and projects only line events into `LogChunk`s. This keeps provider chunks stable while preserving diagnostic information for later surfacing.

Handle lifecycle was the critical implementation constraint. The engine keeps `RuntimeProviderShape.logs` as `Stream<LogChunk, ProviderError>` with no extra Scope requirement; file handles are closed by explicit `handle.close` plus `Stream.ensuring` on normal completion, interruption, and dropped streams. Construction-scoped file acquisition was avoided because it can close real handles too early while in-memory tests mask the issue.

The polling loop deliberately avoids repo-banned retry/schedule helpers for this class of host/provider probing. It uses clock-driven pagination/sleep and a bounded readiness loop for missing files.

Real providers merge the live console stream with file-follow streams only when a `logFileAccess` seam is injected. If the seam is undefined, providers stream console only and do not fail. The actual container file-read adapter remains a later live wave.

Planner-owned capability degradation was made truthful. Required follow sources fail during planning when `serviceLogSources` is unavailable and name the redirect alternative; optional follow sources become unavailable metadata on the service plan. Providers now advertise `serviceLogSources` only when the follower seam exists, so they do not claim a capability they will silently omit.

`app:logs` forwards each service's resolved sources to providers. The SDK fake provider and contract suites proved console plus tagged source projection and finite termination; the dedicated engine suite proved the file-follow failure modes under deterministic clock control.

Meaningful verification included typecheck/lint, focused SDK log-follow engine tests, provider follow tests, app-planner capability tests, provider capability tests, and review fixes for capability truthfulness. The initial-backfill final partial-line behavior stayed as implemented because the contract test explicitly asserted it.

## US-429 — CLI/source visibility and filtering (2026-07-09)

US-429 completed the declared-log-source user surface. `lando logs` accepts `--source <id>` for a declared source or the implicit `console`; unknown ids fail up front with known-source context; the flag composes with service, follow, tail, and since options.

The command normalizes `since` before provider dispatch, preserving the US-393 contract that bad time expressions fail before provider selection.

Filtering semantics became explicit. A file-source request filters resolved sources and tells providers to suppress the container console stream so output is restricted to that source. `--source console` maps to console-only behavior because `LogSourceId` deliberately rejects the reserved `console` id. Redirected sources keep their console-backed stream rather than being suppressed to empty output, and unavailable follow sources fail up front when provider capability is missing.

This preserves the distinction between redirected sources and followed file sources: redirects are already console by construction, while follow sources need provider file access.

Every log line now carries source visibility. TTY rendering labels declared-source lines while preserving the existing console appearance; JSON mode uses the central StreamFrame contract with an additive optional `source` field on stdout/stderr frames. The schema snapshot, SDK compatibility record, and spec StreamFrame block were refreshed for that additive field.

Redaction was fixed for both output modes. Providers still emit raw `LogChunk`s, but the follow StreamFrameSink now redacts file content for TTY and JSON paths before rendering. A boundary test locks that registered secrets in followed files are masked in rendered and JSON output while the on-disk file remains raw.

`lando info` now reports each service's resolved log sources with id, path, strategy, and availability: available, redirected-to-console, or unavailable with reason. Review fixes made unavailable reasons visible in human output as well as the structured result.

Capability gating from the earlier logs work stayed intact. Providers without `serviceLogs` still fail up front, and providers without `serviceLogSources` cannot silently pretend followed sources are available.

Verification included typecheck/lint, redaction and renderer boundary gates, schema snapshot regeneration, focused logs/info scenarios, result-encode and machine-output checks, dual-dispatch parity, provider log-follow/capability/contract suites, app-planner/log-source coverage, and compiled CLI help QA for the new flag.

Durable lessons: most source plumbing already existed from US-425..428; US-429 mainly surfaced it through CLI, renderer, info, schema, and provider suppression behavior. Any future stream of user/file content must redact TTY output as well as JSON output.
