## Codebase Patterns
- Migrating a bespoke durable file (lock + quarantine + version-envelope) onto `StateStore` while keeping a zero-arg sync factory and unchanged public shape (the scratch registry, `core/src/scratch-app/registry.ts`): build the bucket from the dependency-free `makeStateStore()` factory (the same value `StateStoreLive = Layer.succeed(StateStore, makeStateStore())` wraps), NOT `yield* StateStore` ŌĆö tests call `makeScratchRegistry()` directly and run its effects with no `StateStore` service in context. Store `A = entries-array` (NOT the full `{version,entries}` envelope) so `update` operates on the array and the json codec frames it as `{version, data:[...]}` without a redundant second version field; reconstruct the legacy envelope shape (`{version:1, entries}`) only at the service `read()` boundary. `onCorrupt:"quarantine"` + `default:[]` makes a malformed `registry.bin` (not even a valid `{version,data}` envelope) rename to `registry.bin.corrupt-<ts>` and fall back to `[]` (the StateStore `get` quarantines BOTH frame-decode and schema-decode failures). The bucket's own advisory lock (`registry.bin.lock`) now serializes mutations; the old explicit lock helper (`acquireScratchRegistryLock` on `registry.lock`) is retained as a thin delegation to a new exact-lock-path helper `acquireAdvisoryLockAt(lockPath, op)` exported from `core/src/state/lock.ts` (`withAdvisoryLock(file,...)` always derives `${file}.lock`, so it cannot lock an exact `registry.lock` path). Map `StateStoreError -> ScratchAppError` (carry the state error as `cause`, pass through `remediation`) so callers depending on `_tag`/`operation`/`message` are unchanged; add `Effect.asVoid` after `bucket.update` (it returns the new value) to keep `upsert`/`remove` returning `Effect<void>`. `ScratchRegistryLive` export is unchanged so the generated `scratch` bootstrap layer needs no regen.
- Durable on-disk state with cross-process serialization (`StateStoreLive`, `core/src/state/`) needs TWO lock layers, both load-bearing: an `O_CREAT|O_EXCL` token lockfile (`lock.ts`) for OTHER processes, AND a per-absolute-path in-process `Effect.unsafeMakeSemaphore(1)` (module-level `Map`) for same-process fibers. The file lock alone does not reliably serialize fibers under Effect's scheduler ŌĆö two `acquire` retry loops interleave in the window between a holder's release-unlink and the next `O_EXCL` create, dropping a write. Never take over a lock just because its record reads back `null` (a holder mid-flush reads as null); gate the null-record takeover on the lockfile's own `mtime`. Resolve named roots inside the layer via the pure `resolveLandoRoots()`/`makeLandoPaths()` from `@lando/core/paths` (the `ManagedFileServiceLive` precedent), NOT `PathsService` ŌĆö it is a sibling in the `minimal` `Layer.mergeAll`, so `Effect.serviceOption(PathsService)` cannot see it. SDK custom codecs (`encode/decode`, no version slot) own their whole file byte-for-byte and skip version framing; only built-in `json`/`binary` are versioned envelopes.
- The canonical Landofile serializer pair (`@lando/sdk/landofile` `emit.ts`/`parser.ts`) is an ASYMMETRIC round-trip contract: the emitter and parser must agree on the map-key shape. The emitter's `KEY_SHAPE` (`^[A-Za-z0-9_.-]+, all three write paths via the `entriesOf` helper), the parser's `parseMap` key match, and the parser's `parseList` list-item-map `mapMatch` regex must stay in lockstep ŌĆö widening the emitter's accepted key without widening the parser's list-item matcher makes an array-of-maps key emit valid YAML that parses back as a scalar. `parseScalar` ordering is load-bearing: quoted-literal branches (`"ŌĆ”"`/`'ŌĆ”'`) MUST run before the `${` expression rejection, or quoted `${secret:ŌĆ”}` is wrongly rejected (the ┬¦7.8.1 secret round-trip). Strict "encoded wire form" object detection is `Object.getPrototypeOf(v) === Object.prototype || === null` (NOT `typeof object && !Array.isArray`, which silently accepts/corrupts `Date`/`RegExp`/`Map`/class instances). `emitLandofileYaml(value, { sortKeys })` opts into recursive lexicographic key ordering without reordering arrays; default is insertion order (byte-identical). Adding the option/hardening leaves the ┬¦13.2 schema snapshot unchanged (no schema registered).
- The app-plan cache key (`deriveAppPlanCacheKey`, `core/src/cache/app-plan.ts`) folds composition inputs via its `serviceInputs` slot. `AppPlanner.planApp` is two-phase for this: a cheap resolve-only pre-pass collects each service's resolved `base` + ordered FeatureRef list (base defaults first, then `resolution.features`) and the contributed app-feature DEFINITION inputs (id/priority/activatedBy/selectors/requires/pluginId/config); the cache key is derived and read from those BEFORE any plan production, so a warm hit still skips `composeService`/`legacyServicePlan`/`composeAppFeatures`. On a miss the SAME resolution objects are reused ŌĆö `ServiceType.resolve()` runs exactly once per service. Hashing app-feature definition inputs is equivalent to hashing the post-activation result because activation is a pure function of the resolved drafts (which derive from featureRefs+config already in the key). A `lando`-base service type pulls its whole `LANDO_BASE_DEFAULT_FEATURE_IDS` stack through `PluginRegistry.loadServiceFeature` (deduped against resolution features, resolution wins); a planner test using `base: "lando"` must register every base-default id (no-op the unused) or use `base: "l337"` when the base stack is incidental. Compare `ServicePlan.metadata.resolvedAt` via `String(...)`, never `===` ŌĆö `DateTime.Utc` objects for the same instant are reference-unequal.
- The ┬¦7.2 overlay primitive is the exported `mergeValues` in `core/src/landofile/merge.ts` (deep-merge maps, replace scalar arrays, merge object arrays by recognized identity key name/id/hostname/service, right-wins scalars). Reuse it for ANY "child overlays parent per ┬¦7.2" merge (e.g. service-type `extends:` inheritance) instead of hand-rolling a deep merge ŌĆö a naive `{...a,...b}` or record-only deep merge silently drops the object-array identity-key merge the spec mandates. Service-type `type:` reference parsing must be exact-id-first (colon-bearing ids like `php:8.2`/`elasticsearch:8`/`node:22` resolve whole), only splitting at the RIGHTMOST colon into `<name>:<version>` when no exact id is registered; never first-colon-split and never fall back to version-splitting when the exact lookup fails with a cycle/depth `ServiceTypeCollisionError`. `extends:` cycle/depth (Ōēż4 hops) detection is load-time in `composeExtendedServiceType` (the chain is static on the ServiceType); the synthesized type spreads the leaf so private bridge keys (`__legacyToServicePlan`) survive and only `resolve`/`artifacts`/`versions` are replaced. Version pinning lands via `service.image` overlay (the legacy bridge + Phase B read the authored `service`), so overlay onto the stored service, not just `normalizedConfig.image`.
- Shared SDK contract suites that prove seam usage should expose typed structural observation records (request URLs, DownloadRequest shapes, DataEndpoint tags, operation names, finalizer/probe metadata), not opaque counters; keep core-only implementation types inside `@lando/core/testing` and let `@lando/sdk/test` assert portable record shapes.
- A boundary lint gate (`scripts/check-<x>-boundary.ts`) follows one shape: exported pure `check<X>({ root? }) => Promise<{ ok, offenders }>`, recursive `readdir` over `SCANNED_ROOTS = ["core/src","plugins"]` collecting `.ts` non-`.test.ts`, a `CARVE_OUTS` Set (prefer EMPTY ŌĆö migrate offenders, do not carve), and an `import.meta.main` block that writes pass to stdout / failure list to stderr + `process.exitCode = 1`. Test it with a `mkdtemp` fake `root` proving exact offenders AND the legit-pass cases. Wire it by adding `"check:<x>-boundary"` to `package.json` AND a step in `scripts/build-ci-workflow.ts` `renderStaticChecks()` (after the other boundary gates), then `bun run codegen` and assert `git diff --exit-code .github/workflows/ci.yml` is only the new step; update `core/test/build/ci-workflow.test.ts` (step assertion), `docs/ci-runbook.md` + `core/test/build/ci-runbook.test.ts` (the runbook enumerates the gate list), and confirm `core/test/scripts/codegen-ci.test.ts` setup-bun count is UNCHANGED (a step in the existing job adds no new job). Authored gate/test files are NOT auto-biome-formatted ŌĆö run `bun run biome check --write <files>` before lint or the `biome check .` lint gate fails on import-order/formatting.
- If an Effect `Schema.filter` encodes a public-contract constraint, add equivalent `jsonSchema` metadata (pattern/format/bounds) and assert the generated artifact; Effect-only validation is insufficient for JSON Schema consumers.
- The redaction-boundary gate (`scripts/check-redaction-boundary.ts`) flags via TypeScript AST: (1) `ts.isStringLiteral`/`ts.isNoSubstitutionTemplateLiteral` whose `.text` EXACTLY equals `[redacted]` or `[REDACTED]` (whole-string, not substring, not comments), and (2) `RegularExpressionLiteral` nodes whose source is an ad-hoc secret pattern via a CONSERVATIVE multi-signal heuristic (bearer/authorization; URL-userinfo `@`+`[^@`/`:[^`; signed-query `[?&]ŌĆ”(token|api_key|ŌĆ”)=`; or >=2 of password/secret/token/credential/bearer/apikey/api_key keywords) ŌĆö a single `token` mention in an ordinary URL regex is NOT flagged. The one redaction implementation is `@lando/sdk/secrets` `createRedactor(profile)`; the gate forced consolidation of the last hand-rolled redactors. Writing the gate surfaced a FOURTH ad-hoc redactor (`plugins/provider-podman/src/index.ts` inline `/^([^:]+:\/\/)[^@/]+@/u` masking to `<redacted>`) that exploration missed ŌĆö the gate is the source of truth, not a grep. Migrate, never carve out: each provider now has a `src/redact.ts` mirroring `provider-lando/src/redact.ts` (`createRedactor("secrets")` ŌåÆ `redactString`/`redactDetails`). NOTE the canonical `secrets` `urlUserinfo` class masks the WHOLE `user:pass` userinfo to `[redacted]@` (not just the password), so a migrated redactor changes `user:[REDACTED]@` ŌåÆ `[redacted]@`; update assertions to the real primitive output. "Audited services cannot weaken redaction" is proved at each service's REAL redaction point, not by fabricating a seam: ShellRunner/BunSelfRunner pull from `RedactionService.forProfile("secrets")`; Downloader composes `createRedactor("secrets", { values })` in its event seam (switched from value-only `createSecretRedactor` to gain pattern coverage); HostProxy (disabled) emits no secret-bearing field and its diagnostics route through `core/src/cli/redact.ts`; FileSync's SDK errors are redacted at the renderer/`redactDetails` boundary, not in the engine.
- The recipe YAML subset parser (`parseRecipeYaml`) does NOT support block scalars (`key: |` / `key: >`); recipe manifests and test fixtures must use inline string values (e.g. `default: seed line`) or strict decode fails. `collectPrompts` is plain-async (no Effect context), so any child-process need follows the `choicesRunner` precedent: an injectable runner with a `Bun.spawn`-based default plus a `spawner` seam for unit tests (see `editor-command.ts`/`choices-command.ts`). The `editor` prompt type intercepts at the top of `runInteractivePrompt` and returns `undefined` to fall through to text-semantics when no `$VISUAL`/`$EDITOR` is configured; recipe prompt-type deferral is gated by the (now empty) `REJECTED_PROMPT_TYPES` set in `manifest/service.ts`, mirroring `REJECTED_BUN_VERBS`.
- Effect-managed "handle" objects (an App handle, etc.) that own start-state resources should: capture the ambient `Scope.Scope` via `yield* Effect.scope`, fork a per-handle `handleScope = Scope.fork(parent, ExecutionStrategy.sequential)`, and track at most one live child scope in a `Ref<Scope.CloseableScope | undefined>` (effect-v3 type is `Scope.CloseableScope`, NOT `Scope.Closeable`). Serialize lifecycle mutations with `Effect.makeSemaphore(1)`, run every ref/scope swap under `Effect.uninterruptible`, and clean up a freshly-forked scope on failed/interrupted work via `Effect.onError`. Manually closing a child scope before its parent closes is a safe no-op (Effect scopes finalize exactly once), which yields "close exactly once" + "tear down on parent close" for free. A scope-acquired resource (`FileSyncEngine.createSession`) should own its own terminate finalizer; route teardown by providing the managed scope, never by adding a second explicit finalizer at the call site (double-terminate).
- Lando-owned verified downloads route through the core `Downloader` (`@lando/core/Downloader`), which issues bytes through the core-private `HttpClient` chokepoint and is HTTPS-ONLY with no memory-mode bytes (download to a file dest + `readFile`). Request-scoped proxy/CA trust rides the core-private ambient `NetworkTrust` tag (`core/src/http-client/network-trust.ts`), applied by `HttpClientBasicLive.stream` via `Effect.serviceOption` at call time (inert construction); a caller wraps its download effect with `Effect.provideService(NetworkTrust, ...)`. `fetchInitForNetwork`/`shouldBypassProxy` live ONLY there + the file-sync-mutagen exception; never re-add a bespoke per-site copy.
- `DownloaderLive` publishes `pre-download`/`download-progress`/`post-download` (schemas pre-exist in `sdk/src/events/download.ts`, already in the snapshot) via a `{redactText, publish}` seam baked into its Live closure ŌĆö `EventService` resolved with `Effect.serviceOption` (R stays `Scope`), so EVERY construction site that wants events must `Layer.provide(EventServiceLive)` (the bootstrap generator + recipe-tarball + self-update do). Progress = `Stream.tap`+`Ref` over the response body (keep `verified-stream` pure); `post-download` = `Effect.onExit` (interruption still brackets). Events carry scheme+host `urlOrigin` ONLY; `callerId`/`failureDetail` go through a per-call redactor from `request.redactionTokens`, and `failureDetail` is a controlled tag-derived summary, never a raw URL/cause. The `Downloader` contract suite is `runDownloaderContract(harness)` in `@lando/sdk/test`; the in-memory `TestDownloader` (`@lando/core/testing`) reuses the real `makeDownloaderService` over an in-memory `HttpClient` (the network seam), exactly like `TestManagedFileStore`. Egress fence (every byte via `HttpClient`, one stream per miss) is a byte-provenance+call-count invariant in the optional `harness.egress` section; proxy/CA/`NO_PROXY` assertions stay in a CORE-only canary (never put `NetworkTrust`/`fetchInitForNetwork` in `@lando/sdk/test`).
- `spec/beta-1/prd.json` has INCONSISTENT unicode escaping across entries (some literal `┬¦`/`ŌĆö`, some `\uXXXX`). NEVER round-trip it through `json.dump`/`JSON.stringify` to flip a story's `passes`/`notes` ŌĆö any `ensure_ascii` setting rewrites ~half the file and creates massive spurious churn. Edit the target story's `passes`/`notes` lines surgically with the editor (anchor on the unique `"priority": N,` line) and verify the diff is ~2 lines.
- Resolving a PRD priority/dependency inversion (a higher-priority story needs a tag/service a LATER story publishes): introduce a CORE-PRIVATE provisional `Context.Tag` carrying only the slice you need, keep it out of `@lando/sdk` and `core/src/services/index.ts`, give it the same string id the SDK tag will use, and let the later story promote it + swap the live layer. Wire it provided-only into the consumer's bootstrap layer (not as a standalone minimal service) and do NOT add it to `MinimalRuntimeServices`. Example: core-private `HttpClient` for `DownloaderLive` (US-286) ahead of the HttpClient SDK story.
- A "gut-and-replace, delete X, no adapter" SDK contract story whose CONSUMING engine + catalog migration land in LATER stories: scope the deletion to the SDK PUBLIC boundary only (remove the old type/member from `@lando/sdk/*`), and keep the old behavior alive behind a CORE/PLUGIN-PRIVATE structural bridge ŌĆö a `__privateKey` property on the migrated objects + a fail-loud reader (e.g. `legacyServicePlan()` reading `__legacyToServicePlan`, commented "delete in US-NNN"). This honors "no adapter in the SDK" (the bridge is never an SDK export) while keeping the build green. SDK-test helpers that drive the private bridge get RELOCATED into the consumer package's private test dir, not kept in `@lando/sdk/test`. The whole change is ONE atomic commit because `bun run typecheck` is red repo-wide until the last consumer migrates ŌĆö verify incrementally with per-file `bun test <path>` (Bun tolerates unrelated type errors at runtime) and defer the full typecheck to the end. Example: US-355 `ServiceTypeShape.toServicePlan` ŌåÆ `ServiceType.resolve` with the bridge carrying plan production until the ┬¦6.11 engine (US-359) lands.
- A `Layer.effect(Tag, gen yields Dep)` service whose published method signature has `R = Scope.Scope` (no `Dep`) closes over `Dep` at CONSTRUCTION; the method body then only needs `Scope`. Provide `Dep` via `.pipe(Layer.provide(DepLive))` at wiring so the public service R stays clean.
- Helper params that consume `Schema.optional(...)` request fields (which produce `T | undefined`) must type those params `?: T | undefined`, not `?: T` ŌĆö under `exactOptionalPropertyTypes` you cannot pass a `T | undefined` value into a `?: T` slot.
- `Bun.spawn` does NOT propagate JS-mutated `process.env` to the child (verified: setting `process.env.X` then spawning leaves `$X` empty in the child). A subprocess test that needs env isolation (e.g. pointing `LANDO_USER_DATA_ROOT` at a temp dir so `lando init`'s managed-file ledger does not pollute the real home) MUST pass it explicitly via `Bun.spawn({ env: { ...process.env, X: value } })`. In-process helpers can still mutate `process.env` directly.
- Widening a public `Schema.Literal` enum (e.g. adding `FileFormat` members) is additive/backward-compatible but breaks every exhaustive `switch` over it that has no `default` (the codec `encode`/`decode` switches) at typecheck time ŌĆö add the new cases in the same atomic change. Watch for DUPLICATED literals: `core/src/managed-file/service.ts` `LedgerEntrySchema.format` was a hand-copied `FileFormat` literal; reuse the imported SDK `FileFormat` schema value so the set stays in one place. Note it in `sdk/API_COMPATIBILITY.md` "Compatibility notes" (enum widening is not a new field), then `bun run codegen:schema-snapshot`; appending members at the END keeps the first value stable so public-schema contract fixtures do not churn.
- Publishing a new public registered SDK schema is a FIVE-place lockstep: `rawPublicSchemaRegistry` + `PUBLIC_SCHEMA_DESCRIPTIONS` + per-top-level-field `PUBLIC_FIELD_DESCRIPTION_EXEMPTIONS` in `sdk/src/schema/json-schema.ts`, plus a `PUBLIC_SCHEMA_CONTRACT_FIXTURES` entry in `sdk/test/schema/public-schema-contracts.ts` at the SAME ordinal position (its coverage test does an order-sensitive `toEqual(JSON_SCHEMA_NAMES)`). Every new RUNTIME export from `@lando/sdk/{schema,errors}` and every new `Context.Tag` in `services/index.ts` must also be listed in `sdk/API_COMPATIBILITY.md` (the backward-compat test does an exact `toEqual` on the additive lists; type-only interfaces are exempt). Then `bun run codegen:schema-snapshot` and `git add` the new `dist/schemas/*.json` + `docs/reference/schemas/*.mdx` (the drift gate asserts artifacts are git-tracked). A new SDK service tag follows the scratch pattern: real `Context.Tag` class in a per-service file + `export *` + an `export declare class` mirror in `services/index.ts`; pick a service-prefixed name for its options interface since plain `ApplyOptions` is already the provider's.
