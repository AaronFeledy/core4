## Codebase Patterns
- Plugin contribution surfaces that must become part of a materialized artifact (e.g. `globalServices:` → global `.lando.dist.yml`) follow this seam: a pure resolver in `core/src/services/*` collects across `PluginRegistry.list` manifests (deterministic, id-sorted), detects cross-plugin id collisions as a tagged error, then a materializer turns each accepted contribution into the artifact shape via an INJECTABLE loader (default = dynamic `import()` of the contribution `module`; tests inject a fake or point at a temp module). The low-level writer service (e.g. `GlobalAppService.regenerateDist`) stays frozen and dumb; resolution lives in core helpers the command composes. For a `bootstrap: "global"` command to `yield* PluginRegistry`, `PluginRegistryLive` must be in the TOP-LEVEL `Layer.mergeAll` of `makeProviderRuntimeLive` (not only `Layer.provide`'d into `RuntimeProviderRegistryLive`), and the compiled-dispatcher cast in `core/src/cli/run.ts` must widen to the command's full required-service set.
- User-app CLI commands resolve the current project via `loadUserLandofile(landofileService)` (`core/src/cli/app-resolution.ts`), never `landofileService.discover` directly — a canary test (`core/test/cli/reserved-app-id-canary.test.ts`) forbids `.discover` in `core/src/cli/commands/**`. The helper rejects the reserved `global` app id with `AppIdReservedError` before any plan is built. Reason: `LandofileService.discover` / `AppPlanner.plan` error unions are frozen by the SDK backward-compat fixture, so new resolution errors must ride a new seam, and each command's hand-maintained error union (plus composing commands like `rebuild`/`restart`) gains the new error.
- `lando doctor` subsystem recovery is classified per-subsystem in `core/src/cli/commands/doctor-subsystems.ts` via a `SubsystemSpec.recovery` of `automatic` (non-privileged, exposes a re-runnable `setup()` — proxy, ssh) or `manual` (privileged setup like CA trust-store / host DNS, or no `setup()` at all — certs, host-proxy, healthcheck, scanner). `--fix` only re-runs `setup()` for `automatic` degraded subsystems; `manual` ones report `fixOutcome: skipped-manual`. Doctor never auto-elevates. Fix outcomes ride in the check `context` (`fixOutcome`/`fixCommand`/`fixExitCode`/`fixError`) so both the plain-text and NDJSON renderers surface them with no renderer changes.
- New doctor CLI flags must be wired in BOTH dispatch paths to keep source/compiled parity: the OCLIF `static flags` + `inputDoctorOptions` in `core/src/cli/oclif/commands/meta/doctor.ts`, and a hand-rolled `parse*Flag` in `core/src/cli/run.ts` `runDoctor`. Both funnel into the same `doctorReport(options)`.
- Running `@lando/core/cli` in-process (source mode → OCLIF `execute()`) calls `process.exit` on any erroring command, which silently kills the surrounding test runner. Scenario/test harnesses that invoke the CLI in-process must only drive commands that succeed; faked/canned runners exist precisely to avoid this for `start`/`destroy`/`curl`-style flows. Spawn the compiled binary (`core/dist/lando`, overridable via `LANDO_SCENARIO_E2E_BINARY`) when you need to run an erroring command without taking down the runner.
- `machineFailure()` in `plugins/provider-lando/src/setup.ts` is the single factory for all Podman machine errors; extend its regex conditions to detect new OS-specific virtualization prerequisites before adding new error subclasses.
- `buildSetupSteps()` drives both the event-tree children list and which setup actions run; adding a platform requires updating that function + the corresponding `if (platform === ...) ...` block in `setupProviderLando`.
- Windows Podman machine lifecycle is structurally identical to macOS (`ensureWindowsPodmanMachine` mirrors `ensureMacOSPodmanMachine`); the separation exists for future divergence (e.g. WSL-specific machine flags).
- Host-mutating live tests should gate on both the expected platform and an exact opt-in env value (`process.platform !== "win32" || process.env.GATE_VAR !== "1"`) so accidental truthy values do not run provider setup on the wrong host.
- All three providers (docker/lando/podman) realize networking through the shared `@lando/sdk/schema` helpers (`landoNetworkNames`, `landoServiceNetworkAliases`, `landoAppNetworkName`, `landoSharedNetworkName`), not from `plan.networks` directly. To make every provider consume a new per-app networking field, widen those helpers to read it (with a slug-derived fallback for legacy/hand-built plans) instead of editing each provider — provider behavior stays identical when the plan reproduces the slug derivation.
- Adding any new export to `@lando/sdk/schema` (schemas AND helper functions) requires adding it to the `Additive Alpha schema exports` list in `sdk/API_COMPATIBILITY.md`; the `sdk-backward-compatibility` test diffs live exports vs that list. Adding a field to a snapshotted schema (e.g. `AppPlan`) also requires `bun run codegen:schema-snapshot`.
- Adding a new executable-guide component touches five registration points: SDK `sdk/src/docs/components/props.ts` (prop schema + `decode*Either` helper + the `BETA_COMPONENTS` allowlist consulted by `assertAlpha2Component`), `scripts/build-guide-scenarios.ts` (the `GuideStepComponent` union + `parseStepComponent` + `renderStepComponent`), the `scripts/lint-guides.ts` component allowlist, the SDK schema registry in `sdk/src/schema/index.ts` (export/import/`JSON_SCHEMA_REGISTRY`/`getJsonSchema`), and `SDK_SCHEMA_NAMES` in `scripts/build-schema-snapshot.ts`; finish with `bun run codegen:schema-snapshot` and add the new export name to `sdk/API_COMPATIBILITY.md`. (A component can be schema-registered long before it is code-emitting — e.g. `<Hidden>` only needed the decode helper, allowlist flip, codegen, lint, and runtime seam.)
- Any new scenario transcript frame kind must be added to the `TranscriptFrame` union in `sdk/src/docs/transcript.ts`; `persistTranscript` runs `Schema.encodeSync(Transcript)` at scenario teardown and throws on an unknown frame kind.
- Executable-guide scenarios fan out into one generated test file per axis variant. `tabs:` frontmatter declares the single implicit `default` axis's ordered values; `variantsOf()` in `scripts/build-guide-scenarios.ts` turns those into per-scenario variant files named `<scenario-id>.<value>.test.ts` with a `// @variant: default=<value>` header. A scenario body is an ordered `Step | TabsBlock`; `<Tabs>`/`<Tab name>` live inside `<Scenario>` and each tab carries its own `<Step>`s. `resolveVariantSteps()` computes the block step-name union in source order (first occurrence wins) and emits `test.skip(<step>, …)` for union steps a variant's tab omits. A guide with `tabs:` but no `<Tabs>` block still fans every scenario into N identical variants. Multi-axis `axes:` (a record of axis-name → ordered values) fans scenarios into the Cartesian product of all axes in declaration order; a variant is now an ordered list of `{axis,value}` pairs, the filename suffix is the dot-joined values, and the header is space-joined `// @variant: axis=value` pairs. `<Tabs axis="A">` resolves only against that variant's value for axis A (axis prop required when a guide declares >1 axis, inferred for a single axis). Per-cell `variants:` overrides are keyed by the dot-joined cell key and refine `skip` (guide-wide `skip:` ?? per-cell → emits a whole-variant `test.skip`), `tags` (guide ∪ cell → `// @tags:`), and `platforms` (cell wins → `// @platforms:`); only per-cell overrides emit those header lines so guides without `variants:` stay byte-identical. `tabs:`/`axes:` are mutually exclusive and `variants:` keys must resolve to real cells — both enforced on decode by `GuideFrontmatterChecked` (a decode-only refinement; the exported `GuideFrontmatter` stays a plain struct so its JSON snapshot keeps a single named definition). `findDeferredGuideFrontmatter` now only defers `defaultLayer: e2e` (§19.11). NOTE: guide frontmatter is parsed by the Landofile YAML subset (`core/src/landofile/parser.ts`), which only supports block-style maps (no inline `{ }` flow maps) and whose map-key charset now includes `.` so dot-joined `variants:` keys parse.
- Transcript suppression has a single runtime seam: `ScenarioContext.hidden(effect)` (depth-counted, restored on failure) funnels all frame writes through an `appendFrame` choke point in `makeScenarioContext`. Any "emit-but-don't-render" guide construct (`<Hidden>`) should reuse `context.hidden` rather than re-plumbing the frame array.
- `scripts/lint-guides.ts` separates pure AST rules from filesystem-aware fixture rules. `lintGuideContent(sourcePath, content, options?)` stays pure and deterministic — fixture rules read an injected `options.fixtures` inventory (`GuideFixtureInventoryEntry[]` = `{ name, sourcePath, scope: "local"|"shared", kind: "directory"|"symlink"|"other", symlinkPaths? }`), never the disk — so they are unit-testable by passing synthetic inventories (no fixture dirs on disk). Only `lintGuides(root)` walks the disk: it derives each guide's `id` from frontmatter (NOT the MDX path — fixtures live at `docs/guides/<id>/fixtures/<name>` even when the MDX is nested like `docs/guides/services/mongodb.mdx`), inventories local + shared (`docs/guides/fixtures/`) fixture roots, and flags symlinks at the fixture root or any descendant. Add new guide-content lint rules as a pure `lint*` helper called from `lintGuideContent`; add new fixture/disk rules behind the injected inventory. Component-prop validation reuses the SDK `decode*PropsEither` helpers (e.g. `decodeVerifyPropsEither` for `<Verify>` matcher shape) so lint mirrors codegen exactly and needs no new SDK exports.
- Lifecycle event scope: events in `sdk/src/events/index.ts` carry a `scope` discriminator — `scope: "app"` on the canonical per-app analogs (`pre-start`/`post-start`/`pre-stop`/`post-stop`, which are SCHEMA-ONLY, never emitted) and `scope: "global"` on the `pre-/post-global-start|stop` family. The actually-emitted per-app path uses a SEPARATE family (`pre-app-start`/`pre-app-stop`/`pre-service-*`, keyed by `eventName`+`appRef`+`providerId`) that carries no `scope` — leave it alone when adding scope-tagged work. Event payload schemas are NOT frozen (not in `sdk-mvp-surface.json`, not snapshotted, not in `API_COMPATIBILITY.md`); adding a new event schema or a field only needs a presence entry in `sdk/test/library/exports.test.ts`. A command that publishes events must add `EventService` to its required-service union, `EventError` to its failure union, and (for `bootstrap: "global"` commands) the `run.ts` `globalRuntimeLayer` cast must be widened to include `EventService` (the global runtime tier already provides `EventServiceLive`).

**Recurring CI / environment themes (not regressions).**
- `provider-integration-linux-x64` live Docker/Podman "logs emits at least one chunk" was repeatedly flaky: root causes were the SDK contract's tight `logs` timeout (bumped 5s→15s) and provider-docker treating framed Docker log bytes as single-chunk (fixed by buffering framed bytes across socket chunks); the live fixture now emits a periodic log line. **Cross-cutting learnings retained.**
- The SDK backward-compat fixture (`sdk/test/fixtures/sdk-mvp-surface.json`) freezes existing service-tag method signatures by exact match: you cannot widen a frozen tag's error union — surface new errors through a new non-frozen seam. Additive *new* tags only need a `sdk/API_COMPATIBILITY.md` entry; new tagged *errors* are tracked by `sdk/test/library/exports.test.ts` presence only (not API_COMPATIBILITY, not the schema snapshot).

- Bundled `globalServices:` resolve from a static plugin export captured in `core/src/plugins/bundled.ts`, never the manifest `module:` path (relative imports can't resolve and `import()` is impossible in a `bun build --compile` binary); the `module:` field is docs + the non-bundled fallback only.

## [2026-05-30 CDT] - US-112 (YAML round-trip hardening)
- Closed the remaining global-app Landofile emitter/parser soundness gap that blocked PR #218: an empty record nested in an array (`[{}]`) was emitted as a bare `-`, which the Landofile YAML subset parser rejected as "Malformed YAML" — an unparseable round-trip. The emitter now writes empty records as the flow-empty map `- {}`, and `parseScalar` round-trips the flow-empty `{}` back to `{}` (block list items, nested inline arrays, and bare map values all converge through that one parser branch) while populated inline objects stay rejected. Added parser unit coverage and three complementary materialize round-trip tests (block, single-element, nested-inline).
- Verification: typecheck, lint, full `bun test` (1948 pass / 23 skip / 0 fail) passed; `meta:global:install` create/idempotent-rerun QA in an isolated `LANDO_USER_DATA_ROOT` confirmed well-formed output.
- **Learnings for future iterations:**
  - An empty mapping has NO block-style representation as a YAML sequence item — `- ` alone parses as null/throws. The only portable form is the flow-empty map `- {}`. Any emitter that serializes arbitrary nested structures (here `emitArray`) must use `{}` for empty records, and the matched parser must accept flow-empty `{}` (the single allowed inline object) even though it rejects populated inline objects. This is the natural extension point US-114/115 will hit when emitting real `globalServices:` arrays.
  - When hardening the global-app YAML emitter, drive both halves of the matched pair from ONE parser branch (`parseScalar`'s `{}` check) rather than special-casing each call site (`parseList`, `parseInlineArray`) — duplicate special-cases are exactly what review bots flag, and the block-list / inline-array paths already funnel array-item scalars through `parseScalar`.
