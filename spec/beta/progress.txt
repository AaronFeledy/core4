## Codebase Patterns
- `machineFailure()` in `plugins/provider-lando/src/setup.ts` is the single factory for all Podman machine errors; extend its regex conditions to detect new OS-specific virtualization prerequisites before adding new error subclasses.
- `buildSetupSteps()` drives both the event-tree children list and which setup actions run; adding a platform requires updating that function plus the corresponding `if (platform === ...) ...` block in `setupProviderLando`.
- Windows Podman machine lifecycle is structurally identical to macOS (`ensureWindowsPodmanMachine` mirrors `ensureMacOSPodmanMachine`); the separation exists for future divergence (e.g. WSL-specific machine flags).
- Host-mutating live tests should gate on both the expected platform and an exact opt-in env value (`process.platform !== "win32" || process.env.GATE_VAR !== "1"`) so accidental truthy values do not run provider setup on the wrong host.
- `runProviderContract` is the single shared assertion path for provider contracts; extending it lifts coverage for every matrix caller automatically.
- `RuntimeProviderRegistry` does not dispatch through plugin Layers automatically — each new provider id needs a dedicated branch in `core/src/providers/registry.ts` `providerFor`.
- `BUNDLED_PLUGINS`-shape assertions appear in three places (`core/test/plugins/bundled.test.ts`, `core/test/services/plugin-registry.test.ts`, `core/test/services/runtime-provider-registry.test.ts`); update all three together when the bundled list changes.
- Adding a service type updates five test surfaces in `plugins/service-lando/test/` plus schema-snapshot; edit CI pre-pulls via `scripts/build-ci-workflow.ts`, not the workflow directly.
- `GuideFrontmatter` rejects unknown keys — allowed keys are exactly `id`, `defaultLayer`, `provider`, `timeout`, `platforms`, `tags`, `skip`, `deprecated` (no `title`/`description`).
- Per spec §20.8.1, global-service contribution failures should return a `{accepted, rejected}` envelope rather than failing the whole plan.

## [2026-05-28 01:08 UTC] - US-083 Live Go HTTP Scenario
- Added the live/minimal Go HTTP scenario acceptance that the prior US-083 review left open: brings up a Go service against a real Podman socket, fetches the HTTP endpoint, and runs `lando go version` through `ProviderExecToolingEngineLive` against the live provider.
- The test is gated by `test.skipIf(!process.env.LANDO_TEST_PODMAN_SOCKET)` and skipped by default; CI's provider-integration job already exports that env var.
- **Learnings for future iterations:**
  - Live HTTP-server scenarios for new service types follow the `bring-up.integration.test.ts` pattern (hand-built `ServicePlan`/`AppPlan`, `bringUp(plan, { podmanApi })` via real `makePodmanApiClient`, `fetch("http://127.0.0.1:<port>")`, `bringDown` in `finally`) rather than the `AppPlanner` planning path used in fake-provider scenario tests. The two test files complement each other: scenario test exercises planner → tooling, integration test exercises provider lifecycle → live HTTP.
  - Live integration tests that route tooling through `ProviderExecToolingEngineLive` need to provide a `RuntimeProviderRegistry.select` that returns the **same** live `RuntimeProvider` instance the bring-up used. Building the provider once via `RuntimeProvider.pipe(Effect.provide(makeProviderLayer({ podmanApi: api })))` and capturing it for both the bring-up call and the tooling layer keeps the container-name/network state consistent.
  - For `go run` to work inside `golang:1.22` against a bind-mounted source dir, write both `main.go` and a minimal `go.mod` (stdlib-only). `GOFLAGS=-mod=mod` is defensive against the image picking up a stray GOPATH-style layout.
  - `golang:1.22`'s default Cmd is not a long-lived process, so the service plan must supply `command: ["go", "run", "/app/main.go"]` (or similar) to keep the container alive while the HTTP server serves. The default `tail -f /dev/null` keep-alive from `go.ts` is for the planner-driven path where the tooling alias drives one-shot exec; a live HTTP scenario explicitly overrides it.
  - Port publishing in `provider-lando` bringUp maps `endpoint.port` to `HostIp: 127.0.0.1, HostPort: <port>`, so the host port equals the container port — picking a non-default port (31082 vs the type's default 8080) avoids collisions with whatever else is bound to 8080 on a CI runner.

Pin the test expectation to the unversioned form, not the alias target's `id`.
  - `CommandSpec` accepts BOTH `string` and `readonly string[]`. Any helper that flattens argv (e.g. for argv-redaction checks) must handle both. Tests that assert credential plaintext absence from `plan.command`/`plan.entrypoint` need this.

  - Default-credential redaction is structurally enforceable today even without a plan-level redaction layer: assert (a) credentials live in keyed env vars, (b) credential plaintext does NOT appear in `plan.command`/`plan.entrypoint` argv. The full §6.8/§6.10/§6.12.4 redaction (logs, telemetry, LANDO_INFO, rendered config) needs a publishing-layer redact-token set that is not yet wired and is tracked separately.
