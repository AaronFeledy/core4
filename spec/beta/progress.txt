## Codebase Patterns
- `machineFailure()` in `plugins/provider-lando/src/setup.ts` is the single factory for all Podman machine errors; extend its regex conditions to detect new OS-specific virtualization prerequisites before adding new error subclasses.
- `buildSetupSteps()` drives both the event-tree children list and which setup actions run; adding a platform requires updating that function + the corresponding `if (platform === ...) ...` block in `setupProviderLando`.
- Windows Podman machine lifecycle is structurally identical to macOS (`ensureWindowsPodmanMachine` mirrors `ensureMacOSPodmanMachine`); the separation exists for future divergence (e.g. WSL-specific machine flags).
- Host-mutating live tests should gate on both the expected platform and an exact opt-in env value (`process.platform !== "win32" || process.env.GATE_VAR !== "1"`) so accidental truthy values do not run provider setup on the wrong host.

- **Learnings for future iterations:**
  - Memcached has NO persistent storage (in-memory cache); `storage: []` on the ServicePlan is the correct shape, unlike redis/mongodb/postgres which all declare a data volume.
  - The TCP-port-probe healthcheck pattern (`bash -c "exec 3<>/dev/tcp/127.0.0.1/<port>"`) used by go/python/ruby works equally well for protocol-light TCP services like memcached — no need to ship the `memcached` binary or a custom probe.
  - Tooling aliases for non-canonical commands (like `memcached-tool` which is not present in the default `memcached:1.6` image) can be implemented as `sh -c` wrappers using BusyBox-style `nc`; the scenario test exercises the wrapper through the fake provider exec without requiring the binary to exist at planning time.
  - Running the full `bun test` suite triggers `core/test/scripts/codegen-ci.test.ts`, which re-runs the workflow codegen and writes `.github/workflows/ci.yml`. If `ci.yml` contains hand-edits not reflected in `scripts/build-ci-workflow.ts` (e.g. ad-hoc `podman pull` lines or extra test invocations), the full-suite run will mutate `ci.yml` and the codegen output will diverge from the committed file. Story builds should `git checkout .github/workflows/ci.yml` after running the full suite to keep the commit scoped to the story.
