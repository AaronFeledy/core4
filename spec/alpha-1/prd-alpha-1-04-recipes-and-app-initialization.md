# PRD: ALPHA1-04 — Recipes and app initialization

## Introduction

This PRD covers Phase 2 Alpha 1 work for **Recipes and app initialization**. It translates the Alpha 1 section of [`spec/ROADMAP.md`](../ROADMAP.md) into implementation-sized stories while preserving the MVP rule that the detailed spec parts remain source of truth.

Depends on: **PRD-02, PRD-03**.


## Source References

- [`spec/08-cli-and-tooling.md`](../08-cli-and-tooling.md) — §8.8 recipe model and `apps:init`.
- [`spec/06-services.md`](../06-services.md) — service catalog targets generated by canonical recipes.
- [`spec/17-executable-tutorials.md`](../17-executable-tutorials.md) — later executable-guides/scenarios direction; Alpha 1 recipes should not conflict with it.

## Goals

- Replace the MVP hardcoded recipe with a real Alpha 1 recipe parser.
- Ship interactive init prompts and canonical common-stack recipes.
- Keep remote/dynamic recipe features deferred and visibly unsupported.

## User Stories

### US-025: Parse full Alpha 1 `recipe.yml` schema

**Description:** As a recipe author, I can write `recipe.yml` using the Alpha 1-supported schema.

**Acceptance Criteria:**
- [ ] Parser tests cover metadata, prompts, files, Landofile output, and postInit sections from §8.8
- [ ] Schema rejects Alpha-3-only remote/dynamic fields with remediation
- [ ] MVP hardcoded recipe path migrates to the recipe parser without breaking MVP scenario
- [ ] Tests pass
- [ ] Typecheck passes
- [ ] Lint passes

### US-026: Support built-in prompt types

**Description:** As a user running `lando init`, I can answer text/select/multiselect/confirm/number/secret/path prompts.

**Acceptance Criteria:**
- [ ] CLI fixture tests cover each prompt type with deterministic stdin/stdout
- [ ] Secret prompts do not echo values in renderer transcripts
- [ ] Prompt validation errors are recoverable in interactive mode and typed in non-interactive mode
- [ ] Tests pass
- [ ] Typecheck passes
- [ ] Lint passes

### US-027: Ship cwd-only recipe source

**Description:** As an Alpha 1 user, I can initialize from recipes in the current checkout/binary bundle only.

**Acceptance Criteria:**
- [ ] Recipe source tests cover built-in/cwd discovery and reject git, tarball, npm, registry sources with Alpha 3 remediation
- [ ] Recipe resolution is deterministic and does not perform network access
- [ ] Bundled recipe files are embedded in the compiled binary via `scripts/build-bundled-recipes.ts` → `core/src/recipes/bundled.ts` with no disk read at runtime; codegen drift is caught by the bundled-codegen gate in PRD-07 US-049
- [ ] Tests pass
- [ ] Typecheck passes
- [ ] Lint passes

### US-028: Ship canonical common-stack recipes

**Description:** As an alpha tester, I can scaffold the canonical v4.0 recipe set from §8.8.10, including WordPress, Laravel, node-api, Django, FastAPI, and Rails apps.

**Acceptance Criteria:**
- [ ] Each §8.8.10 canonical recipe has a scenario test that runs init, parses the generated Landofile, and verifies the recipe snapshot names the generated service ids, `type:` entries, and tooling command ids
- [ ] The binary ships the full §8.8.10 canonical recipe set in Alpha 1; Drupal and v3-style recipe compatibility shims remain out of scope per §8.8.10
- [ ] Recipe READMEs document alpha limitations and required host prerequisites
- [ ] Tests pass
- [ ] Typecheck passes
- [ ] Lint passes

### US-029: Support programmatic `landofile.ts` output

**Description:** As an advanced recipe author, I can generate a Landofile programmatically during init.

**Acceptance Criteria:**
- [ ] Tests cover a recipe that emits Landofile data via `landofile.ts` and validates the result through LandofileService
- [ ] Programmatic output is sandboxed to documented inputs; no remote code fetch is allowed
- [ ] Generated Landofile remains compatible with app planning
- [ ] Tests pass
- [ ] Typecheck passes
- [ ] Lint passes

### US-030: Implement Alpha 1 `postInit` bun install action

**Description:** As a recipe user, post-init can run Bun install where the recipe declares it.

**Acceptance Criteria:**
- [ ] Scenario test covers `postInit: bun: { verb: install }` using a fake process runner or fixture package
- [ ] `bun install` runs only inside the generated destination/app directory, never global Lando/plugin/cache directories
- [ ] Tests reject outside-destination `cwd` or path traversal attempts for postInit actions
- [ ] Docs/remediation explicitly state network access and package lifecycle script behavior for `bun install`
- [ ] Unsupported postInit actions (`script`, `add`, `create`, `run`, `x`) return Alpha 3 remediation
- [ ] Post-init failures leave clear instructions, redact env/registry token values in output, and do not delete generated files
- [ ] Tests pass
- [ ] Typecheck passes
- [ ] Lint passes

### US-031: Make `apps:init` interactive full prompt flow

**Description:** As a new user, `lando init` guides me through recipe and prompt selection.

**Acceptance Criteria:**
- [ ] Interactive scenario test covers choosing a recipe and answering prompts through stdin
- [ ] Non-interactive flags can supply prompt values for CI scenarios
- [ ] Init progress renderer-mode coverage is owned by PRD-05 US-039
- [ ] Tests pass
- [ ] Typecheck passes
- [ ] Lint passes

### US-032: Validate recipe artifacts in CI

**Description:** As a maintainer, every shipped recipe remains executable as the service catalog evolves.

**Acceptance Criteria:**
- [ ] Recipe test layer runs all canonical recipe parse/plan tests in CI
- [ ] At least one Linux provider integration scenario covers a generated common-stack app when env prerequisites exist
- [ ] Failures identify recipe name and generated file path
- [ ] Tests pass
- [ ] Typecheck passes
- [ ] Lint passes

## Functional Requirements

- Implement only the Phase 2 Alpha 1 surface assigned to this PRD.
- Preserve all accepted MVP behavior and regression coverage.
- Match existing Bun workspace conventions: `bun run typecheck`, `bun run lint`, `bun test`, and generated-file updates through `bun run codegen` where applicable.
- Source CLI behavior and compiled binary behavior must stay aligned for user-visible commands touched by this PRD.

## Non-Goals

- Do not implement features listed in the Alpha 1 index cross-cutting non-goals.
- Do not stabilize non-SDK library APIs beyond the `unstable`/dev-channel promise.
- Do not add new external dependencies unless the relevant spec part already requires them or a separate architecture decision approves them.

## Technical Considerations

- Use the spec part referenced by each story as the source of truth when details conflict with this PRD.
- Prefer fake-client/unit coverage for provider and CLI behavior; live runtime tests must be env-gated.
- Default runtime provider for tests in this PRD is `TestRuntimeProvider` from `@lando/sdk/test`; live `provider-lando`/`provider-docker` cases must be gated on `LANDO_TEST_PODMAN_SOCKET` / `LANDO_TEST_DOCKER_SOCKET` (or `DOCKER_HOST`).
- Keep tagged errors and remediation text consistent across source OCLIF and compiled `$bunfs` paths.
- Avoid broad refactors while implementing a story; each story should be reviewable independently.

## Success Metrics

- Every user story in this PRD is accepted with its tests merged.
- `bun run typecheck`, `bun run lint`, and `bun test` pass for the whole workspace.
- Alpha 1 roadmap exit criteria remain achievable without adding unplanned Alpha 3/Beta 1 scope.

## Open Questions
