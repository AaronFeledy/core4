# PRD: ALPHA3-05 — Global app

## Introduction

The global app (§20 / `spec/18-global-app.md`) is the home for cross-app shared services — by default Traefik (proxy) and Mailpit (mail capture). Alpha 3 ships the full `GlobalAppService`, the `globalServices:` plugin contribution surface, the reserved `global` app id, the `meta:global:*` command namespace, `Global` lifecycle event scope, and auto-start via `AppFeature.requires.globalServices`.

Depends on: **ALPHA3-02** (service catalog includes Mailpit's base service type — see PRD-02 also handles `static`/passthrough), **ALPHA3-04** (subsystems supply the shared cross-app network and proxy client).

## Goals

- Make the global app a first-class concept on disk and at runtime.
- Ship Traefik (proxy) and Mailpit (mail capture) as bundled global services.
- Let plugins contribute additional `globalServices:` without modifying core.
- Auto-start the global app when any per-app `AppFeature.requires.globalServices` declares a dependency.

## User Stories

### US-111: `GlobalAppService` core service + reserved id `global`

**Description:** As the runtime, the `global` app id is reserved; any user attempt to declare it in a Landofile is rejected with `AppIdReservedError`.

**Acceptance Criteria:**
- [ ] `GlobalAppService` Effect Service tag, contract, and Live Layer in `@lando/core`.
- [ ] `AppIdReservedError` tagged error published in `@lando/sdk`; emitted when a Landofile declares `name: global` or any id resolving to `global`.
- [ ] Schema round-trips through the §13.2 snapshot gate.
- [ ] Tests pass; typecheck passes; lint passes.

### US-112: `<userDataRoot>/global/.lando.yml` materialization

**Description:** As the runtime, the global app has a materialized Landofile under the per-user data root; `meta:global:install` writes it idempotently.

**Acceptance Criteria:**
- [ ] `<userDataRoot>/global/.lando.yml` is generated by `meta:global:install`; reruns are idempotent.
- [ ] The materialized file references bundled global services + any plugin-contributed `globalServices:` entries.
- [ ] User edits to the global Landofile are preserved across `meta:global:install` reruns; conflicts produce a tagged remediation.
- [ ] Tests pass; typecheck passes; lint passes.

### US-113: `globalServices:` plugin contribution surface

**Description:** As a plugin author, I can contribute `globalServices:` from my plugin's `Layer` and have those services appear in the global app's Landofile.

**Acceptance Criteria:**
- [ ] `globalServices:` plugin contribution shape published in `@lando/sdk`; resolver merges contributions deterministically (last-wins per spec).
- [ ] Conflict between two plugin contributions on the same service id produces a tagged error at bootstrap.
- [ ] Contract suite verifies a fake plugin contributing a global service appears in the materialized global Landofile.
- [ ] Tests pass; typecheck passes; lint passes.

### US-114: bundled Traefik global service

**Description:** As a user, Traefik runs inside the global app and provides the default proxy referenced by per-app services.

**Acceptance Criteria:**
- [ ] `traefik` global service shipped from `@lando/service-lando` (or a dedicated `@lando/service-traefik` plugin — implementer's choice; spec §20.10 is silent on packaging).
- [ ] Default config exposes the Traefik dashboard on `traefik.lndo.site` (or configured base domain).
- [ ] Scenario test (with global app started) confirms a per-app service is reachable through Traefik.
- [ ] Tests pass; typecheck passes; lint passes.

### US-115: bundled Mailpit global service

**Description:** As a user, Mailpit runs inside the global app and per-app services have `MAIL_HOST`/`MAIL_PORT` env vars pointing at it.

**Acceptance Criteria:**
- [ ] `mailpit` global service shipped; SMTP and web UI exposed (`mailpit.lndo.site` by default).
- [ ] Per-app `LANDO_*` env contract extends to include `LANDO_MAIL_HOST` + `LANDO_MAIL_PORT` (resolved against the shared cross-app network).
- [ ] Scenario test sends mail from a per-app service and asserts it appears in Mailpit's UI/API.
- [ ] Tests pass; typecheck passes; lint passes.

### US-116: `meta:global:*` CLI namespace

**Description:** As an operator, I can run `meta:global:install`, `meta:global:start`, `meta:global:stop`, `meta:global:status`, `meta:global:uninstall`, `meta:global:config`, and `meta:global:destroy`.

**Acceptance Criteria:**
- [ ] All seven commands implemented; aliases per §8.2 (e.g. `lando global:install`).
- [ ] `meta:global:destroy` is gated behind `--yes` confirmation per the cross-cutting destructive-confirmation rule.
- [ ] Compiled `$bunfs` parity for every command (or unified dispatch from PRD-09 covers it).
- [ ] Tests pass; typecheck passes; lint passes.

### US-117: `Global` lifecycle event scope

**Description:** As an event subscriber, I receive `pre-global-start`, `post-global-start`, `pre-global-stop`, `post-global-stop` events distinct from per-app lifecycle events.

**Acceptance Criteria:**
- [ ] Event payloads published in `@lando/sdk` with schema; `scope: "global"` in the event payload distinguishes them from per-app `scope: "app"`.
- [ ] Plugin subscribing to `post-global-start` receives the event before any per-app service that declared `AppFeature.requires.globalServices` boots.
- [ ] Tests pass; typecheck passes; lint passes.

### US-118: auto-start via `AppFeature.requires.globalServices`

**Description:** As a user, running `lando start` on an app that needs the proxy will auto-start the global app if it is not running — no manual `meta:global:start`.

**Acceptance Criteria:**
- [ ] Per-app `AppFeature.requires.globalServices: ["traefik"]` (or analogous) wires the dependency.
- [ ] Auto-start respects the abort signal; if global app start fails, per-app start fails with a chained tagged error pointing at the global failure.
- [ ] `lando stop` does NOT auto-stop the global app — global stop is explicit only.
- [ ] Tests pass; typecheck passes; lint passes.

### US-119: global-app integration with `lando doctor`

**Description:** As an operator, `lando doctor` reports global app status (installed, started, contributing-plugin manifest, proxy / mailpit reachability).

**Acceptance Criteria:**
- [ ] `meta:doctor` includes a `global-app` section per §10.9.
- [ ] Output covers materialized Landofile path, last `meta:global:install` timestamp, per-service status, and plugin contribution list.
- [ ] Tests pass; typecheck passes; lint passes.

## Functional Requirements

- FR-1: `global` app id is reserved; user attempts fail closed.
- FR-2: `<userDataRoot>/global/.lando.yml` is the materialized source-of-truth; `meta:global:install` writes it idempotently.
- FR-3: `globalServices:` is the only contribution surface for cross-app services; conflicts fail at bootstrap.
- FR-4: Traefik and Mailpit ship as bundled global services and back the default `ProxyService` and mail-capture experience.
- FR-5: `meta:global:*` namespace covers the full lifecycle (install / start / stop / status / config / uninstall / destroy).
- FR-6: `Global` lifecycle event scope is distinct from per-app scope.
- FR-7: `AppFeature.requires.globalServices` triggers auto-start; auto-stop is never triggered.

## Non-Goals

- `dependsOn: ["global:<service>"]` declarations in per-app Landofiles (post-4.0 per §14.2; the `AppFeature` mechanism is the supported path).
- Multi-tenant global apps (one global app per user data root in Alpha 3).
- Persistent storage migrations across `meta:global:install` reruns (Beta 1).
- Web UI for global-app management (post-GA).
- Mailpit / Traefik configuration UIs beyond Landofile-driven config.

## Technical Considerations

- The global app's bootstrap level is `app` (not a new level); reused infrastructure from per-app start.
- Reserved-id enforcement happens at the `LandofileService` parse stage so the error surfaces before any plan is built.
- `globalServices:` contributions merge through the same plugin-registry resolver used elsewhere; conflict resolution is "last contributor wins, with bootstrap-time error if explicit override was not requested".
- The materialized global Landofile is a real file the user can hand-edit between `meta:global:install` runs — preserving user edits requires a 3-way merge (default, user, contribution); design lives in §20.
- Auto-start triggers must keep the abort signal threaded so Ctrl-C during a per-app start stops global app boot too.

## Success Metrics

- A fresh `lando setup && lando init … && lando start` produces a working global app on first run with no intermediate command.
- Mailpit captures mail from at least three different framework presets (Drupal / Laravel / Rails) without per-recipe code.
- `meta:doctor --json` lists every global service's status.

## Open Questions

- Should Traefik's dashboard be exposed by default, or gated behind a `--with-dashboard` setup flag for security clarity? Default: exposed (matches existing v3 behavior).
- Should `meta:global:uninstall` delete user-data-root global state, or just unregister the global app? Default: unregister + warn; `--purge` deletes data.
- How is the user-edit / contribution conflict surfaced — interactive merge, conservative no-op, or fail with remediation? Default: fail with remediation pointing at the conflicting key; user resolves manually.
