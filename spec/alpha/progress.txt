## Codebase Patterns

- Planner capability gates live in `core/src/services/planner.ts`; service-plan fields that require provider support should fail before apply with `CapabilityError` carrying service, feature, capability, provider id, and remediation.
- `meta:setup` runs at provider bootstrap through `RuntimeProviderRegistry.select()`; compiled `$bunfs` commands that bypass OCLIF need matching handlers in `core/src/cli/run.ts`.
- `provider-lando` setup state is persisted under `<userDataRoot>/providers/provider-lando/setup-state.json` when a state directory is configured.
- provider-lando live macOS machine lifecycle checks stay gated behind `LANDO_TEST_PROVIDER_LANDO_MACOS`; fake `PodmanMachineRunner` tests cover default behavior.
- Provider capability tests should force the target platform through provider factory options (for example `makeProviderLayer({ platform, ... })`) instead of relying on `process.platform` when asserting Linux vs macOS matrices.
- Provider lifecycle state should persist an encoded `AppPlan` under the provider state directory and treat stale/corrupt state as a cache miss, not a hard failure.
- App stop must preserve applied plan state for later fresh-process info/destroy commands; app destroy should remove that state after provider cleanup.
- Provider partial-apply rollback removes (force DELETE) every touched container and the app network; volumes are NEVER removed on rollback. Volume removal is `destroy(--volumes)`-only.
- Provider error `details` must be passed through `redactDetails` (see `plugins/provider-lando/src/redact.ts`) before attachment; it masks credential-named keys plus env-style `NAME=VALUE` substrings in string bodies.
- Provider errors should carry an operation-scoped name (e.g. `bringUp.network`, `bringUp.create`, `bringUp.start`) so manual cleanup can identify the exact failing API call, plus `remediation` and `cause` per spec Â§5.7.
- Shared `ServiceConfig` fields used by only a subset of service types (e.g. `framework`) belong on the schema as optional; per-type validation/rejection lives in `ServiceType.toServicePlan` so unrelated types ignore the field and supported types consume it. `LandofileShape` strict decoding stays intact this way.
- `ServiceTypePlanInput.appName` is the canonical seam for service types that need accurate `LANDO_APP_NAME`/`LANDO_PROJECT`; it is threaded by `AppPlanner.plan` from `landofile.name` and falls back to `basename(appRoot)` when unset.
- Beta-deferred Landofile surfaces (top-level `includes:`/`secrets:`/`env_file:`, and Landofile-level configuration expressions `${...}`/`{{ ... }}`) are rejected by `LandofileService` with `NotImplementedError` + Beta remediation BEFORE schema validation, so consumers see a tagged "defer to Beta" error instead of a generic "excess property" message. Widening this rejection set means widening the `LandofileService.discover` and (transitively) `AppPlanner.plan` error unions plus every CLI command alias (`StartAppError`, `StopAppError`, `DestroyAppError`, `InfoAppError`) that depends on them.
- Authored `service.appMount.excludes/includes/readOnly` are applied by `AppPlanner.plan` on top of the service-type's emitted `AppMountPlan`; service types may keep emitting `excludes: []`/`includes: []` defaults without per-type plumbing. The planner also expands each effective exclude into a service-scope shadow `DataStorePlan` named `<appName>-<service>-<kebab(target/excludePath)>` with a matching `DataStoreMountPlan` on the service; entries starting with `!` and entries listed in `includes` are skipped.
- Plug-in service Layers that need the active runtime provider must NOT depend on the `RuntimeProvider` Tag â€” that tag is bound to the stub in every runtime layer (`makeProviderRuntimeLive`/`makeAppRuntimeLive`). Real providers come from `RuntimeProviderRegistry.select(plan)`, so command paths resolve the provider at the call site and thread it as an explicit argument (see `ToolingEngine.run(invocation, plan, provider)` in `core/src/services/tooling-engine.ts`, mirroring the `startApp`/`stopApp`/`destroyApp` pattern).
- `LandoCommandBase.runEffect` threads OCLIF's parsed `flags` and `args` into `spec.run({ argv, signal, flags, args })`; source-mode command specs can read flags/args from `input` directly. Compiled `$bunfs` commands still parse `argv` themselves (no OCLIF at runtime), so commands that need flag support in BOTH paths should ship an `extractInput` helper that reads either shape and a small argv parser inside `run.ts`.
- Compiled `$bunfs` dispatcher in `core/src/cli/run.ts` intercepts global `--help`/`--version`. Pass-through commands (currently `meta:bun` and `meta:x`) must be added to the `isBunOrX` short-circuit at the top of `runCompiledCli` so the embedded Bun receives those flags. `MVP_COMMAND_IDS` in `core/src/cli/oclif/command-base.ts` doubles as the alpha-implemented-commands set; adding a canonical id there is enough to unblock `runEffect` from emitting `NotImplementedError`.
- Host-wide Lando-app discovery in Alpha walks `<userDataRoot>/providers/provider-lando/apps/*.json`. `listServices` decodes a permissive subset (`plan.id`, `plan.name`, `plan.root`, `plan.services` keys) rather than the full `AppPlan` so fixtures don't need every required schema field; the full `AppPlan` schema remains the persistence contract. `apps:poweroff` reuses the same discovery and supports `--keep-global` / `--keep-scratch` / `--yes`.
- Plugin install in `core/src/cli/commands/plugin-add.ts` runs in this order: (1) Bun-spawn-based npm install into `<userDataRoot>/plugins/`, (2) decode `landoPlugin` from `package.json` against the `PluginManifest` Effect schema, (3) verify `manifest.entry` resolves inside the package dir (including symlink walk via `realpath`), (4) consult the in-memory `globalTrustStore: Set<string>` and prompt the user (or honor `--trust` / `--yes` / `--non-interactive`) BEFORE registering the plugin. Non-interactive runs without `--trust` fail closed with a tagged `NotImplementedError` pointing at `spec/10-plugins.md`. Persistent trust storage stays deferred to Beta.
- Recipe manifest parsing follows the LandofileService shape: YAML parse â†’ pre-decode `NotImplementedError` rejection for Beta-deferred sections (`runs:`, `fetchAllowlist:`, `editor` prompt type, `choicesFrom:`, non-`install` `bun` verbs) â†’ strict `Schema.decodeUnknownEither(RecipeManifest, { onExcessProperty: "error" })` â†’ tagged-error preservation. Live Layer is `RecipeManifestServiceLive` in `core/src/recipes/manifest/service.ts`; the standalone `parseRecipe(source, content)` Effect is exported for callers that want to skip the Tag wiring (used by `initApp`).
- Bundled built-in recipes live next to their existing emit modules under `core/src/recipes/builtin/<id>/`. Until the Â§17.2 bundled-recipes codegen lands, ship `recipe.yml` as a TypeScript string constant (`<id>/manifest.ts`) so the compiled `$bunfs` binary has no runtime FS dependency. The `initApp` MVP scaffold parses its bundled manifest before writing files; new built-in recipes follow the same pattern.
- Built-in recipe rendering uses the `BUILTIN_RECIPE_RENDERERS` registry in `core/src/recipes/builtin/registry.ts`. Each recipe folder ships `manifest.ts` (embedded recipe.yml + id constants), `render.ts` (a `RecipeRenderer` returning a `dest â†’ content` map keyed by the manifest's `files: dest` entries), and a `README.md` documenting alpha limitations and host prerequisites. `initApp` looks up the renderer by `manifest.id` only when `resolved.root === undefined` (bundled ref), so local-recipe id collisions cannot poach a bundled renderer.
- Recipe codegen scripts that write into `core/src/**` must end with `biome check --write <output>` (not just `biome format --write`) because Biome's organize-imports rule reorders multi-import generated blocks; using `format` alone passes formatting but trips the full lint gate when the import set grows.
- Recipe post-init execution lives in `core/src/recipes/post-init/runtime.ts#runPostInit`. Inject `BunSelfSpawner` + `PostInitIO` through `InitAppOptions.postInitSpawner` / `postInitIO` for tests so the live `Bun.spawn` path stays untouched; defaults write to `process.stdout` for `message` actions. `bun.install` resolves the action `cwd:` against the destination, rejects `..` segments before realpath, then realpath-resolves BOTH the destination and the cwd to detect symlink escape (sibling tmp realpaths like `/tmp` vs `/private/tmp` are why both sides need realpath). Non-zero exit raises `RecipePostInitError(kind: "exit")` with a redacted remediation that names the destination, mentions network access, dependency lifecycle scripts, and `rm -rf node_modules && lando bun install` for retry, and explicitly states generated files were preserved. `gitInit` / `command` / any action with `when:` defer to Beta via `NotImplementedError`; unsupported `bun` verbs (`script/add/create/run/x`) are rejected before schema decode in `core/src/recipes/manifest/service.ts`.
- Recipe prompt collection runs through `collectPrompts({ prompts, answers, yes, nonInteractive, io })` in `core/src/recipes/prompts/runtime.ts`. The I/O surface is injectable via `PromptIO` â€” `createStdioPromptIO()` for production and `createBufferedPromptIO({ inputs })` for fixture tests with scripted stdin/stdout. Per-type unit tests should use the buffered IO and assert on `io.stdout()` / `io.stderr()` directly; CLI integration tests use `Bun.spawn({ stdin: "pipe" })`. `secret` answers MUST never appear in the prompt transcript or any error message; runtime validation messages are sanitized to never echo the rejected input. `--full` is no longer a synonym for non-interactive defaults: use `--yes` to accept defaults and `--no-interactive` to fail fast on missing answers.
- Global CLI flags that need to behave identically in source OCLIF and the compiled `$bunfs` paths should live as a shared helper under `core/src/cli/<flag>.ts` and be invoked at BOTH entry points: (a) `LandoCommandBase.runEffect` BEFORE the MVP guard and `this.parse(this.ctor)` â€” mutate `this.argv.length = 0; this.argv.push(...stripped)` so OCLIF parse never sees the unknown flag; (b) the top of `runCompiledCli` in `core/src/cli/run.ts` BEFORE the `--help` / `--version` / per-command short-circuits, with a passthrough escape hatch for `bun` / `meta:bun` / `x` / `meta:x` so embedded Bun receives those flags. New tagged errors used in either path must be taught to BOTH `commandErrorMessage` implementations (one in `core/src/cli/oclif/command-base.ts`, one in `core/src/cli/run.ts`) including the `details.unshift(tag)` header convention.
- `LandofileService` discovery walks for `.lando.yml` AND `.lando.ts` in lock-step per directory; both present in the same directory raises `LandofileParseError` with a "remove one form" message. The TS path goes through `core/src/landofile/ts-loader.ts#loadLandofileTs`, which (1) `Bun.Transpiler.scan({ loader: "ts" })`s the source to enumerate static + dynamic imports, (2) defensively regex-scans for `require(...)` calls (ESM-mode TS hides those from `scan`), (3) rejects any forbidden node builtin / URL-scheme import / relative import that resolves outside the app root with `LandofileSandboxError` carrying a spec Â§7.1.1 remediation, then (4) wraps `import(filePath + "?t=â€¦")` + default-export resolution in `Effect.timeoutFail(LandofileTimeoutError)`. Default timeout is 5000ms, overridable via `LANDO_LANDOFILE_TS_TIMEOUT_MS`. Default export may be a `Landofile` value or `(ctx) => Landofile | Promise<Landofile>`; the resolved value flows through the same `LandofileShape` decode as the YAML path so validation errors stay `LandofileValidationError`.
- `EventService` exposes two subscriber surfaces. `subscribe(name)` returns a `Stream` that subscribes to the underlying `PubSub` lazily (on first pull) and is fine for late-attached consumers. `subscribeQueue` (no args) is the cold-start, no-replay surface: it returns a scoped Effect that resolves to a `Queue.Dequeue<LandoEvent>` once the `PubSub.subscribe` handshake has completed, so any `publish` issued after the Effect succeeds is guaranteed to enqueue. Renderers and any other "must not miss the first event" consumers should call `subscribeQueue` inside `Layer.scopedDiscard` and fork a `Queue.take` consumer; add a finalizer that drains `Queue.takeAll` before interrupting the fiber so trailing events published in the same tick as scope close are still rendered (this is what `core/src/cli/renderer/runtime.ts#makeRendererLive` does for the plain/json renderers). Extending `EventService` always breaks every mock â€” sweep `Layer.succeed(EventService, { â€¦ })` literals across `core/test/cli/*.scenario.test.ts` and `sdk/test/services/tags.test.ts` in the same change or `tsc -b` will fail.
- Renderer-side line formatting is pure and lives in `core/src/cli/renderer/format.ts` (`renderPlainLine` / `renderJsonLine` return `string | null`; `null` means "not a task tree event, skip"). The Layer in `runtime.ts` only does plumbing: it calls the formatter and writes to `RendererIO.writeStdout` / `writeStderr` based on the renderer (`plain` â†’ stdout, `json` â†’ stderr; `lando` aliases plain until rich-TTY behavior lands). Tests inject `createBufferedRendererIO()` and assert on `stdout()` / `stderr()` / `stdoutLines()` â€” no `Bun.spawn` round trip needed for renderer unit coverage. The named snapshot fixture lives at `core/test/cli/fixtures/renderer.task-tree.concurrent.ndjson` and is loaded raw (one JSON object per line) so the renderer is exercised against a deterministic event ordering across runs.

- Command-index cache helpers live in `core/src/cache/command-index.ts` and `core/src/cache/command-index-writer.ts`; app-command freshness includes Landofile stat metadata plus a `toolingFingerprint`, plugin-command freshness includes a manifest fingerprint, and callers that must surface cache build failures should use `writePluginCommandCacheStrict` instead of the best-effort `writePluginCommandCache`.
- `cwd-app-map` cache helpers live in `core/src/cache/cwd-app-map.ts`; use a fixed magic/version/hash header (4 B `LCWM` + 8 B BE schema version + 32 B sha256-of-payload = 44 B) plus `node:v8` serialize/deserialize (not `Bun.serialize`, which is unavailable in this repo's Bun) and let `apps:list` merge cache-only entries with provider state. Three invariants the helper MUST keep or US-041 regresses: (1) `readCwdAppMap` compares stored `landoVersion` against `CORE_VERSION` and treats a mismatch as a silent cache miss (Â§12.1 whole-cache invalidation; corruption still surfaces a `CacheError` with remediation because the AC explicitly demands it); (2) `normalizeEntries` MUST NOT use `new Map([newEntry, ...existing].map(...))` â€” `Map` is last-write-wins, so a stale existing entry would overwrite the freshly upserted one. The dedup is a first-wins loop with a regression test in `core/test/services/cache.test.ts`; (3) `listServices` in `core/src/cli/commands/list.ts` MUST default `userCacheRoot` to `resolveUserCacheRoot()` â€” otherwise the OCLIF `apps:list` / compiled `apps:list` / `poweroff` paths skip the cache silently because they call `listServices()` with no options. Atomic-via-rename writes are deferred to US-042; today's `writeFile` is acceptable because the sha256 header turns torn writes into a clean `CacheError` (which `apps:list` swallows to `[]`).
- Binary cache encoding fixtures live in `core/test/cache/fixtures/binary-cache/*.bin` and are compared byte-for-byte by `core/test/cache/binary-encoding.test.ts`; intentional format/header/schema-version changes should update fixtures and exported cache header constants together.
- Generated CI drift gates live in `scripts/build-ci-workflow.ts` and should run focused `bun run codegen:*` commands followed by `git diff --exit-code -- <generated files>`; generated JSON fixtures (for example the schema snapshot) should be Biome-clean from the generator, not fixed by hand.
- The dev prerelease GitHub Actions workflow is generated by `scripts/build-release-workflow.ts` through the `release-workflow` entry in `scripts/codegen.ts`; do not hand-edit `.github/workflows/release.yml`.
- The npm dev-package release job prepares publishable package metadata with `scripts/prepare-npm-dev-packages.ts`: it publishes `@lando/sdk` as a same-version `dev` dependency, rewrites `@lando/core`'s temporary `workspace:*` SDK dependency to that exact alpha version, then dry-runs and publishes `@lando/core` with `--tag dev` while asserting `latest` is unchanged.
- `makeLandoRuntime` embedding tests can pass host-provided service overrides through `plugins.layers`; the runtime merges those Layers after the base bootstrap Layer, so duplicate service tags (for example `RuntimeProvider` / `RuntimeProviderRegistry`) are resolved in favor of the host layer.
- `@lando/core/cli/operations` is the unstable Effect-bearing command API for embedding hosts; use `invokeOperation(operation, { render, renderError })` when a host needs result/error/output as data instead of throwing or writing to stdio. Keep it out of `core/src/cli/index.ts` so the CLI fast path does not statically import Effect/OCLIF.
