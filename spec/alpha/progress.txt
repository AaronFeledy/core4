## Codebase Patterns

- Planner capability gates live in `core/src/services/planner.ts`; service-plan fields that require provider support should fail before apply with `CapabilityError` carrying service, feature, capability, provider id, and remediation.
- `meta:setup` runs at provider bootstrap through `RuntimeProviderRegistry.select()`; compiled `$bunfs` commands that bypass OCLIF need matching handlers in `core/src/cli/run.ts`.
- `provider-lando` setup state is persisted under `<userDataRoot>/providers/provider-lando/setup-state.json` when a state directory is configured.
- provider-lando live macOS machine lifecycle checks stay gated behind `LANDO_TEST_PROVIDER_LANDO_MACOS`; fake `PodmanMachineRunner` tests cover default behavior.
- Provider capability tests should force the target platform through provider factory options (for example `makeProviderLayer({ platform, ... })`) instead of relying on `process.platform` when asserting Linux vs macOS matrices.
- Provider lifecycle state should persist an encoded `AppPlan` under the provider state directory and treat stale/corrupt state as a cache miss, not a hard failure.
- App stop must preserve applied plan state for later fresh-process info/destroy commands; app destroy should remove that state after provider cleanup.
- Provider partial-apply rollback removes (force DELETE) every touched container and the app network; volumes are NEVER removed on rollback. Volume removal is `destroy(--volumes)`-only.
- Provider error `details` must be passed through `redactDetails` (see `plugins/provider-lando/src/redact.ts`) before attachment; it masks credential-named keys plus env-style `NAME=VALUE` substrings in string bodies.
- Provider errors should carry an operation-scoped name (e.g. `bringUp.network`, `bringUp.create`, `bringUp.start`) so manual cleanup can identify the exact failing API call, plus `remediation` and `cause` per spec §5.7.
- Shared `ServiceConfig` fields used by only a subset of service types (e.g. `framework`) belong on the schema as optional; per-type validation/rejection lives in `ServiceType.toServicePlan` so unrelated types ignore the field and supported types consume it. `LandofileShape` strict decoding stays intact this way.
- `ServiceTypePlanInput.appName` is the canonical seam for service types that need accurate `LANDO_APP_NAME`/`LANDO_PROJECT`; it is threaded by `AppPlanner.plan` from `landofile.name` and falls back to `basename(appRoot)` when unset.
