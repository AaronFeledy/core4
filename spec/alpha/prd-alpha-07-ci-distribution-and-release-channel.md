# PRD: ALPHA-07 — CI, distribution, and release channel

## Introduction

This PRD covers Phase 2 Alpha work for **CI, distribution, and release channel**. It translates the Alpha section of [`spec/ROADMAP.md`](../ROADMAP.md) into implementation-sized stories while preserving the MVP rule that the detailed spec parts remain source of truth.

Depends on: **PRD-01 through PRD-06**.


## Source References

- [`spec/13-testing-and-distribution.md`](../13-testing-and-distribution.md) — CI/test/distribution layers.
- [`spec/15-binary-build-and-release.md`](../15-binary-build-and-release.md) — binary build/release engineering.
- [`spec/16-deprecation-and-surface-evolution.md`](../16-deprecation-and-surface-evolution.md) — unstable surface/deprecation boundaries.

## Goals

- Extend CI from MVP artifact checks to Alpha schema/library/recipe gates.
- Publish Linux x64 dev prereleases and npm dev-tag packages.
- Document Alpha install, verification, and bug-report paths.

## User Stories

### US-049: Add schema and bundled-codegen snapshot gates to CI

**Description:** As maintainers, schema changes and bundled-plugin/recipe set changes are intentional and visible in PR review.

**Acceptance Criteria:**
- [ ] CI includes a schema-snapshot diff job generated by `scripts/build-ci-workflow.ts`
- [ ] Schema-snapshot scope is `@lando/sdk` schema exports plus the bundled plugins' manifest schemas; out-of-tree plugin manifests are not part of the gate
- [ ] CI re-runs `scripts/build-bundled-plugins.ts` and `scripts/build-bundled-recipes.ts` and fails via `git diff --exit-code` when `core/src/plugins/bundled.ts` or `core/src/recipes/bundled.ts` is out of date
- [ ] Failing test covers generator output for both the schema gate and the bundled-codegen gate
- [ ] Schema-snapshot and bundled-codegen update commands are documented (`bun run codegen`)
- [ ] Tests pass
- [ ] Typecheck passes
- [ ] Lint passes

### US-050: Add library API and recipe test layers to CI

**Description:** As maintainers, Alpha-only surfaces are continuously verified.

**Acceptance Criteria:**
- [ ] Generated CI workflow runs library API tests and recipe test layer separately or as named steps
- [ ] Workflow tests assert the job/step names so branch protection can target them
- [ ] Env-gated live provider tests remain skipped unless secrets/host config exist
- [ ] Tests pass
- [ ] Typecheck passes
- [ ] Lint passes

### US-051: Promote Linux x64 workflow artifact to dev prerelease

**Description:** As an alpha tester, I can download a Linux x64 binary from the dev channel.

**Acceptance Criteria:**
- [ ] Release workflow promotes successful build artifact to a GitHub prerelease tagged for dev/alpha
- [ ] Dev prerelease publishes a single `SHA256SUMS` text file alongside the binaries (one line per artifact, `<sha256>  <filename>` format); no GPG/cosign signing required in Alpha
- [ ] No signing, SBOM, installer scripts, or self-update are required in Alpha, but `SHA256SUMS` publication is required
- [ ] Promotion is generated from scripts/codegen, not hand-edited YAML
- [ ] Sandboxed first-launch test runs the distributed artifact in a network namespace with egress blocked and asserts exit 0 plus no non-loopback `connect()` calls under `strace -e trace=network` (no telemetry beacon) — enforces the index non-goal
- [ ] Tests pass
- [ ] Typecheck passes
- [ ] Lint passes

### US-052: Publish `@lando/core` alpha package to npm dev tag

**Description:** As an embedding tester, I can install `@lando/core@4.0.0-alpha.N` from npm dev tag.

**Acceptance Criteria:**
- [ ] Package publish workflow for `@lando/core` uses workspace build artifacts and dry-run test coverage
- [ ] The packaging plan documents whether workspace dependencies are bundled, published as dev-tag dependencies, or otherwise resolved without changing the roadmap's named `@lando/core` Alpha distribution scope
- [ ] Publishing uses protected CI credentials or trusted publishing; no local developer token path is accepted
- [ ] Publish command/config uses `--tag dev` and has a test/assertion that `latest` is not changed
- [ ] Tests pass
- [ ] Typecheck passes
- [ ] Lint passes

### US-053: Keep Bun version source pinned to `.bun-version`

**Description:** As CI maintainers, lockfile compatibility does not drift from package engine ranges.

**Acceptance Criteria:**
- [ ] Workflow generator tests assert all setup-bun uses `bun-version-file: .bun-version`
- [ ] No generated workflow reads `package.json#engines.bun` as the CI Bun version source
- [ ] AGENTS/CI runbook note remains accurate
- [ ] Tests pass
- [ ] Typecheck passes
- [ ] Lint passes

### US-054: Document alpha install and bug-report path

**Description:** As an alpha tester, I know how to install, verify, and report useful failures.

**Acceptance Criteria:**
- [ ] Docs cover dev prerelease binary download, `sha256sum -c SHA256SUMS` verification, npm dev install (`npm install @lando/core@dev`), `lando doctor`, and required logs/artifacts
- [ ] Docs state Linux x64 binary + npm `dev` install (Linux/macOS) support and Windows deferral clearly; macOS users use the npm path until macOS binary promotion lands in Beta
- [ ] Docs are linked from release notes or README
- [ ] Tests pass
- [ ] Typecheck passes
- [ ] Lint passes

### US-055: Assert branch protection for Alpha CI jobs

**Description:** As maintainers, the expanded CI matrix is required before merging Alpha work.

**Acceptance Criteria:**
- [ ] Branch-protection assertion includes static checks, build-linux-x64, provider integration, schema gate, library API, and recipe layers
- [ ] Test can run without network by reading documented required-checks list or a gh API mock
- [ ] Docs list exact required job names
- [ ] Tests pass
- [ ] Typecheck passes
- [ ] Lint passes

### US-056: Keep multi-platform matrix deferred to Beta except explicit Alpha macOS provider checks

**Description:** As maintainers, Alpha CI grows deliberately without pretending full platform support.

**Acceptance Criteria:**
- [ ] Workflow tests assert no broad Windows or linux-arm64 release matrix is generated in Alpha
- [ ] Any macOS provider validation is either documented manual QA or explicit opt-in job, not required default CI unless approved
- [ ] Roadmap deferrals remain mirrored in PRD non-goals
- [ ] Tests pass
- [ ] Typecheck passes
- [ ] Lint passes

## Functional Requirements

- Implement only the Phase 2 Alpha surface assigned to this PRD.
- Preserve all accepted MVP behavior and regression coverage.
- Match existing Bun workspace conventions: `bun run typecheck`, `bun run lint`, `bun test`, and generated-file updates through `bun run codegen` where applicable.
- Source CLI behavior and compiled binary behavior must stay aligned for user-visible commands touched by this PRD.

## Non-Goals

- Do not implement features listed in the Alpha index cross-cutting non-goals.
- Do not stabilize non-SDK library APIs beyond the `unstable`/dev-channel promise.
- Do not add new external dependencies unless the relevant spec part already requires them or a separate architecture decision approves them.

## Technical Considerations

- Use the spec part referenced by each story as the source of truth when details conflict with this PRD.
- Prefer fake-client/unit coverage for provider and CLI behavior; live runtime tests must be env-gated.
- Keep tagged errors and remediation text consistent across source OCLIF and compiled `$bunfs` paths.
- Avoid broad refactors while implementing a story; each story should be reviewable independently.

## Success Metrics

- Every user story in this PRD is accepted with its tests merged.
- `bun run typecheck`, `bun run lint`, and `bun test` pass for the whole workspace.
- Alpha roadmap exit criteria remain achievable without adding unplanned Beta/RC scope.

## Open Questions

- None blocking; resolve story-level ambiguities by updating this PRD and the authoritative spec part together.
